# ADR-001: Multi-Database Infrastructure Strategy with Docker Compose Development Environment

> **Date**: 2026-04-05
> **Decision Date**: Pending
> **Review By**: 2026-07-05 — revisit after Phase 2 completion to evaluate whether dedicated databases justify operational cost vs. consolidation
> **Decision Owner**: Tech Lead
> **Status**: Proposed
> **Supersedes**: None
> **Deciders**: Tech Lead, Backend Lead, Infrastructure Engineer
> **Phase**: 0 | **PR**: N/A (foundational decision documenting existing architecture)

## Context

The Legal AI Platform requires a data layer that supports five distinct workloads:

1. **Transactional data** — contracts, users, tenants, audit logs with ACID guarantees and row-level security (multi-tenant isolation)
2. **Graph relationships** — contract-to-entity mappings, clause hierarchies, obligation networks for GraphRAG traversal
3. **Vector similarity search** — document embeddings for semantic retrieval (RAG pipeline), hybrid search with sparse+dense vectors
4. **Caching and real-time messaging** — session management, rate limiting, Celery task brokering, pub/sub for WebSocket events
5. **Document/object storage** — legal document files (PDF, DOCX), versioned artifacts, backup blobs

The platform is in Phase 0 (Stabilization), with ~75K lines of existing code already integrated against PostgreSQL 15, Neo4j 5, Qdrant, Redis 7, and MinIO via Docker Compose. The Phase 0 BRD establishes the prerequisite that the existing infrastructure must be stabilized (green tests, CI, dependency audit) before Phase 1 infrastructure upgrades (PG 17, pgvector, Qdrant 1.12, Valkey).

The decision at hand is: **should we continue with the purpose-built multi-database architecture and Docker Compose local environment, or pivot to a simpler topology before investing in Phase 1+ upgrades?** This is the right time to evaluate — Phase 0 stabilization is the last low-cost pivot point before Phase 1 locks in infrastructure upgrade investments across 16+ PRs.

### Constraints

- **Existing codebase**: ~75K LoC already integrates with all five databases via repository pattern abstractions
- **Team size**: Small team (2-3 engineers), no dedicated DBA or SRE
- **Budget**: Zero additional infrastructure budget for Phase 0; cloud spend deferred to production planning
- **Timeline**: Phase 1 starts Week 3; infrastructure decisions must be locked by end of Week 2
- **Compliance**: Legal industry requires tenant data isolation, audit trails, and data residency capability
- **AI workloads**: RAG pipeline requires sub-second vector similarity search across 100K+ document embeddings per tenant

## Assumptions

> If any assumption changes, re-evaluate this decision. The Decision Owner (Tech Lead) monitors these.

| # | Assumption | Falsification Criteria | Impact if Wrong |
|---|-----------|----------------------|-----------------|
| A1 | Developer machines have ≥16GB RAM and can run all 6 Docker containers simultaneously | Docker Compose `up` fails or OOM-kills containers on team machines | Must drop to subset profiles or move to remote dev environments |
| A2 | PostgreSQL + pgvector provides sufficient vector search performance (p95 < 200ms for top-50 cosine similarity over 100K embeddings per tenant) | Benchmark in Phase 1 PR 1.2.5 shows p95 > 200ms | Qdrant remains the primary vector store; pgvector is supplementary only |
| A3 | Neo4j Community Edition is sufficient for GraphRAG workloads (≤10M nodes, ≤50M relationships per tenant) | Graph traversal p95 > 2s for 3-hop queries or CE feature gaps block development | Evaluate Neo4j Enterprise or alternative (e.g., Apache AGE as PG extension) |
| A4 | The team will have at least one engineer with Neo4j/Cypher proficiency by Phase 3 (AI/ML Overhaul) | No team member can write production Cypher by Week 8 | GraphRAG scope must simplify or defer; consider embedding graph context in PG |
| A5 | Qdrant's hybrid search (sparse + dense vectors, added in 1.7+) eliminates the need for a separate Elasticsearch instance | Hybrid search recall < 85% on legal corpus benchmarks in Phase 3 | Add Elasticsearch/OpenSearch to the stack (increases operational complexity) |
| A6 | Docker Compose is sufficient for local development through Phase 4; Kubernetes is only needed for staging/production | Developer experience degrades (startup > 3 min, frequent OOM, port conflicts) | Introduce Docker Compose profiles or remote dev clusters earlier |
| A7 | MinIO's S3-compatible API means production migration to AWS S3/GCS requires no application code changes | MinIO-specific API behavior causes bugs when switching to cloud S3 | Abstract storage client needs integration tests against real cloud provider |

## Scope

This decision affects:

- **Infrastructure**: `docker-compose.yml`, `docker-compose.dev.yml`, all container definitions and healthchecks
- **Backend data layer**: `backend/app/repositories/` — all repository implementations
- **Backend configuration**: `backend/app/core/` — database connection factories, settings
- **CI pipeline**: `.gitea/workflows/ci.yml` — service containers for integration tests
- **Phase 1 upgrade scope**: PRs 1.1.x (PG 17), 1.2.x (pgvector), 1.3.x (Redis/Valkey), 1.4.x (Qdrant 1.12)
- **Operational documentation**: Runbooks, monitoring, backup procedures

## Decision Drivers

1. **Workload-specific performance requirements**: Vector similarity search (p95 < 200ms), graph traversal (p95 < 2s for 3-hop), transactional consistency (ACID for contract mutations), and sub-10ms cache reads are fundamentally different access patterns that no single database optimizes for simultaneously
2. **Existing codebase integration**: ~75K LoC already uses repository-pattern abstractions against all five databases; migrating to a different topology has a cost floor of 4-6 person-weeks of refactoring + regression risk, which the Phase 0 BRD explicitly identifies as the kind of risk to avoid
3. **Multi-tenant data isolation**: Legal industry compliance requires per-tenant isolation at the data layer. PostgreSQL RLS, Neo4j label-based filtering, Qdrant collection-per-tenant, and MinIO bucket-per-tenant each leverage native isolation primitives of their respective stores
4. **Phase 1 upgrade path already planned**: The Modernization Roadmap defines 16 PRs for infrastructure upgrades (PG→17, pgvector, Qdrant→1.12, Redis→Valkey). This decision validates or rejects that plan
5. **Developer experience**: Local development must start in < 2 minutes and provide production-representative data behavior without cloud dependencies

## Options Considered

### Option A: Continue Multi-Database Architecture (PostgreSQL + Neo4j + Qdrant + Redis + MinIO) with Docker Compose

Maintain the current purpose-built database topology. Each database handles the workload it was designed for. Docker Compose orchestrates local development; production will use managed services or Kubernetes.

**Architecture:**
```
┌─────────────────────────────────────────────────────────────────┐
│                     Docker Compose (dev)                         │
│                                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌───────┐  ┌─────┐ │
│  │PostgreSQL│  │  Neo4j   │  │  Qdrant  │  │ Redis │  │MinIO│ │
│  │  15→17   │  │    5     │  │ 1.7→1.12 │  │7→Valk.│  │     │ │
│  │+pgvector │  │+APOC+GDS │  │ +hybrid  │  │       │  │ S3  │ │
│  │          │  │          │  │  search   │  │       │  │ API │ │
│  │Txn + RLS │  │Graph RAG │  │Vector sim│  │Cache  │  │Docs │ │
│  └──────────┘  └──────────┘  └──────────┘  └───────┘  └─────┘ │
│       │              │             │            │          │     │
│       └──────────────┴─────────────┴────────────┴──────────┘     │
│                    Repository Pattern Abstraction                 │
└─────────────────────────────────────────────────────────────────┘
```

**Pros:**
- Zero migration cost — validates existing ~75K LoC as-is
- Each database operates at optimal performance for its workload type
- Repository pattern already abstracts database access; swapping implementations is isolated
- Independent scaling in production — can scale vector search without scaling graph DB
- Phase 1 upgrade path (16 PRs) proceeds as planned; no roadmap disruption
- Native multi-tenant isolation primitives per database (RLS, collections, labels, buckets)

**Cons:**
- Operational complexity: 5+ databases to monitor, back up, upgrade, and troubleshoot
- Local development requires ~6-8GB RAM for all containers simultaneously
- Cross-database transactions are impossible; must use saga pattern for consistency across stores
- Team must maintain proficiency across PostgreSQL, Cypher, Qdrant API, and Redis commands
- No single query language or tooling for cross-store analytics

**Effort**: S (no change required; validate existing setup in Phase 0)
**Risk**: Medium (operational complexity is manageable with small team now, but grows with production traffic)

### Option B: Consolidate to PostgreSQL-Only with Extensions (pgvector, Apache AGE, pg_cron)

Replace Neo4j with Apache AGE (graph extension for PostgreSQL), replace Qdrant with pgvector, use PostgreSQL LISTEN/NOTIFY instead of Redis pub/sub, and keep MinIO for object storage.

**Pros:**
- Single database to operate, back up, and monitor (plus MinIO for blobs)
- Single query language (SQL + Cypher-like for AGE)
- True cross-store transactions — graph, vector, and relational queries in one transaction
- Simpler CI — one database service container instead of four
- Reduced RAM footprint for local development (~2GB vs ~6-8GB)

**Cons:**
- **Apache AGE is immature**: v1.5 (latest stable) lacks Neo4j's APOC procedures, GDS algorithms, and has limited Cypher compliance — no variable-length path patterns, no MERGE with complex patterns. GraphRAG queries would need significant rewriting
- **pgvector performance ceiling**: Benchmarks show pgvector is 2-5x slower than Qdrant for >100K vectors with HNSW indexing. For 100K embeddings per tenant × multiple tenants, this may not meet p95 < 200ms target
- **No native pub/sub replacement**: PostgreSQL LISTEN/NOTIFY doesn't support pattern subscriptions, message persistence, or the distributed queue semantics Celery requires. Would still need Redis for task brokering
- **Migration cost**: Estimated 4-6 person-weeks to rewrite Neo4j repositories (Cypher → AGE SQL), Qdrant repositories (gRPC → pgvector SQL), and Redis-backed caching/session/queue logic
- **Loss of independent scalability**: Graph-heavy and vector-heavy workloads compete for the same PostgreSQL resources; can't scale vector search independently
- **Existing test suite invalidated**: ~75K LoC integration tests assume separate database connections; all would need rewriting

**Effort**: XL (4-6 person-weeks of migration + 2-3 weeks of regression testing)
**Risk**: High (Apache AGE maturity risk, pgvector performance risk, migration risk during early project phase)

### Option C: Managed Cloud Services from Day One (AWS RDS + Neptune + OpenSearch + ElastiCache + S3)

Skip Docker Compose local development; all engineers develop against shared cloud services from the start.

**Pros:**
- Zero local infrastructure management
- Production-grade scaling, backups, and monitoring from day one
- Managed patching and upgrades
- High availability built-in (multi-AZ)

**Cons:**
- **Cost**: Minimum viable managed stack (RDS db.t3.medium + Neptune db.t3.medium + OpenSearch t3.small + ElastiCache t3.micro + S3) = ~$800-1,200/month for dev/staging alone, before production. The Phase 0 BRD specifies $0 infrastructure budget
- **Developer experience**: Requires internet connectivity for all development; local-first development impossible. Latency to cloud adds 20-50ms to every database call during development
- **Vendor lock-in**: Neptune uses Gremlin (not Cypher), OpenSearch replaces Qdrant's vector API, ElastiCache is Redis-compatible but has feature gaps. Switching providers requires significant refactoring
- **Service parity gap**: AWS Neptune is Gremlin-based, not Cypher-compatible. Would require rewriting all Neo4j/Cypher queries to Gremlin — estimated 3-4 person-weeks
- **Existing codebase**: All repository implementations assume local connection strings and specific client libraries; cloud migration requires SDK changes throughout

**Effort**: L (2-3 person-weeks of cloud setup + 3-4 person-weeks of code migration)
**Risk**: High (budget constraint violation, vendor lock-in, Neptune/Gremlin migration)

### Option D: Kubernetes (minikube/k3s) for Local Development

Replace Docker Compose with a local Kubernetes cluster for development parity with production.

**Pros:**
- Local environment mirrors production topology exactly
- Helm charts reusable across dev/staging/production
- Service mesh, network policies, and resource limits testable locally
- Native support for horizontal scaling simulation

**Cons:**
- **Resource requirements**: minikube/k3s + 5 databases requires ≥16GB RAM dedicated to the cluster, plus host OS. Most developer machines can't support this alongside IDE and browser
- **Complexity disproportionate to phase**: Phase 0-2 are a small team (2-3 engineers). Kubernetes operational overhead (RBAC, secrets, PVCs, ingress, cert-manager) is not justified until production deployment planning
- **Slower iteration**: `docker compose up` takes 30-60s; `minikube start` + `helm install` takes 3-5 minutes. Container image builds for Kubernetes add 30-60s per change
- **Debugging difficulty**: Kubernetes abstractions (pods, services, ingress) add layers between developer and application. `docker compose logs` is simpler than `kubectl logs -f deployment/backend -c backend`
- **Team learning curve**: No team member has production Kubernetes experience; adding this learning curve to Phase 0-1 delays stabilization

**Effort**: L (2-3 person-weeks of Helm chart creation, cluster configuration, CI integration)
**Risk**: Medium (resource constraints, team learning curve, slower iteration)

## Decision

**Chosen option**: Option A — Continue Multi-Database Architecture with Docker Compose, because:

1. It directly addresses **Decision Driver #1** (workload-specific performance) by using each database for its designed strength rather than forcing a single database to handle all patterns suboptimally.
2. It satisfies **Decision Driver #2** (existing codebase) with zero migration cost, preserving the ~75K LoC investment and allowing Phase 0 to focus on stabilization rather than infrastructure migration.
3. It supports **Decision Driver #3** (multi-tenant isolation) through native isolation primitives in each database.
4. It validates **Decision Driver #4** (Phase 1 upgrade path) by confirming the 16-PR infrastructure upgrade plan proceeds as designed.
5. It meets **Decision Driver #5** (developer experience) — Docker Compose starts all services in < 60s with healthchecks.

The rejected alternatives each fail on at least one critical driver: Option B fails on performance risk (pgvector) and maturity risk (Apache AGE) while introducing 4-6 weeks of migration. Option C violates the $0 budget constraint and introduces vendor lock-in. Option D adds operational complexity disproportionate to team size and project phase.

## Consequences

### Positive

- **Zero roadmap disruption**: Phase 0 stabilization and Phase 1 infrastructure upgrades proceed as planned per the Modernization Roadmap
- **Proven architecture validated**: The existing repository-pattern abstractions are confirmed as the correct abstraction layer, giving the team confidence to invest in stabilization
- **Best-in-class per workload**: Each database can be independently tuned and upgraded (e.g., Qdrant 1.12 for hybrid search) without affecting other stores
- **Production flexibility preserved**: Can deploy to managed services (RDS, Memorystore, etc.) or self-hosted Kubernetes without changing application code, because the repository pattern abstracts connection details
- **Developer autonomy**: No cloud dependency for local development; works offline, on airplane, on any machine with Docker

### Negative

- **Operational surface area**: 5 databases × (monitoring + backups + upgrades + security patches) = significant ops burden as the platform moves to production. This must be addressed in Phase 2.5 (Monitoring Stack) and production planning
- **Cross-database consistency**: No distributed transactions across PostgreSQL, Neo4j, Qdrant. The saga pattern (already in `systemPatterns.md`) adds complexity to multi-store writes (e.g., contract creation that writes to PG + Neo4j + Qdrant)
- **Knowledge breadth required**: Team must be proficient in SQL, Cypher, Qdrant REST/gRPC API, and Redis commands. This is a hiring and onboarding constraint

### Trade-offs & Accepted Risks

- **Accepting higher operational complexity** in exchange for workload-optimized performance and zero migration risk. Mitigation: Docker Compose healthchecks, init scripts, and Phase 2.5 monitoring stack
- **Accepting saga pattern complexity** for cross-database consistency in exchange for independent scalability. Mitigation: repository pattern limits blast radius of consistency bugs to the service layer
- **Accepting local RAM requirements (~6-8GB for containers)** in exchange for production-representative local development. Mitigation: `docker-compose.dev.yml` already exists for running databases only; application runs natively. Docker Compose profiles can disable unused services
- **Deferring production topology decision** (managed vs. self-hosted, single-region vs. multi-region) to Phase 4-5. This is acceptable because the repository pattern abstracts connection details, making the pivot cost low
