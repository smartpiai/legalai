# Business Requirements Document

> **Phase**: Phase 0 — Stabilization
> **Author**: Product Owner (Legal AI Platform)
> **Date**: 2026-04-05
> **Status**: Draft

---

## 1. Executive Summary

1. **What is proposed**: A 2-week stabilization sprint covering 14 PRs across three workstreams — fix all failing tests (5 PRs), audit and lock dependencies with security scanning (4 PRs), and establish CI pipelines with recorded baseline metrics (5 PRs). This is the prerequisite gate for the entire Modernization Roadmap.
2. **Why it matters**: The platform has ~75K lines of code with an unknown test pass rate, no coverage reporting, no linting enforcement, no dependency lock files, and no security scanning. Building Phase 1–5 (infrastructure, backend, AI/ML, frontend, data) on this foundation carries unquantified regression risk, blocks CI-based quality gates, and exposes the project to undetected CVEs in 80+ transitive dependencies.
3. **The ask**: ~2 person-weeks of engineering effort, zero infrastructure budget (uses existing Gitea CI runner), 2-week timeline (Weeks 1–2). No additional headcount required.

## 2. Business Problem

### Current State
- **Failing tests with unknown scope**: The `WorkflowDesignerPage` test suite passes at ~59% and `AdminDashboardPage` at ~83%. The full backend `pytest` and frontend `vitest` suite pass rates are unknown — they are not run in CI with coverage reporting. Stale `_green` test variants exist alongside canonical tests, creating confusion about what "passing" means.
- **No dependency lock files**: Backend uses a bare `requirements.txt` with no pinned transitive dependencies. No `pyproject.toml` with project metadata exists. `package-lock.json` may not be committed. This means builds are non-reproducible — different developers and CI get different dependency trees.
- **No security scanning**: Neither `pip-audit` nor `npm audit` runs in any automated pipeline. With 80+ Python packages and 100+ npm packages, the current CVE exposure is unknown.
- **Minimal CI pipeline**: The existing `.gitea/workflows/ci.yml` runs `pytest` (no coverage flag), `npm test` (no coverage flag), and `docker-compose build`. No linting (`ruff`, `mypy`, `eslint`, `tsc --noEmit`). No coverage thresholds. No baseline recording. No artifact storage.
- **No baseline metrics exist**: Test count, code coverage %, build time, and frontend bundle size have never been recorded. Without baselines, there is no way to detect regressions introduced by Phase 1–5 changes.

### Desired State
- All tests pass (100% pass rate) or are explicitly skipped with documented `reason=` justifications.
- Stale `_green` test variants audited — duplicates deleted, valid tests merged into canonical suites.
- `pyproject.toml` with `[project]` metadata is the Python packaging source of truth.
- All transitive dependencies pinned via `pip-compile` (`requirements.lock`) and `package-lock.json` committed.
- `pip-audit` and `npm audit` have been run; all CVEs documented with severity and remediation plan.
- CI runs on every PR: backend `pytest` + coverage report, frontend `vitest` + coverage report, `ruff check` + `mypy` on backend, `eslint` + `tsc --noEmit` on frontend.
- Baseline metrics recorded and stored: test count, coverage %, build time (seconds), frontend bundle size (KB).

## 3. Impact of Inaction

| Risk | Category | Estimated Impact |
|------|----------|-----------------|
| Unknown CVEs in 80+ Python and 100+ npm transitive dependencies — no audit has ever been run | Security | Potential data breach of sensitive legal documents; compliance audit failure; estimated remediation cost of an incident: $500K–$2M (legal, notification, reputation) |
| Failing tests mask regressions introduced by Phase 1 infrastructure upgrades (PG 17, Valkey, Qdrant 1.12) | Quality / Timeline | Phase 1 has 16 PRs that change infrastructure underneath the application. Without a green test baseline, regressions are undetectable. Estimated debugging cost: 1–2 additional person-weeks per undetected regression |
| Non-reproducible builds — no dependency lock files | Operational | "Works on my machine" failures. CI builds may pass while local builds fail (or vice versa). Estimated wasted developer time: 2–4 hours/week across the team |
| No coverage reporting — cannot enforce quality gates on Phase 1–5 PRs | Quality | Phase 3 alone introduces 58 PRs touching AI agent architecture. Without coverage gates, untested code paths accumulate. Industry data: each 10% drop in coverage correlates with 40% increase in production defects |
| No linting or type checking in CI — code quality drift | Operational | Without `ruff`, `mypy`, `eslint`, and `tsc` in CI, every PR risks introducing type errors, unused imports, and style violations that compound over 158 total roadmap PRs |
| Phases 1–5 are blocked — cannot safely begin modernization without known-good baseline | Timeline | The Modernization Roadmap specifies Phase 0 as prerequisite. Every week of Phase 0 delay pushes the entire 12–15 week roadmap by the same amount. At current burn rate (~$40K/week loaded eng cost), each week of delay = $40K in extended timeline |

## 4. Proposed Solution

Execute the Modernization Roadmap Phase 0 ("Stabilization") as defined in [MODERNIZATION_ROADMAP.md](../../MODERNIZATION_ROADMAP.md). This phase establishes the safety net (green tests + CI quality gates + known baselines) upon which all subsequent phases build.

### In Scope
- **0.1 — Fix Failing Tests** (5 PRs): Fix `WorkflowDesignerPage` tests (59% → 100%), fix `AdminDashboardPage` tests (83% → 100%), audit all `_green` test variants, fix or skip (with documented reason) all broken backend and frontend tests.
- **0.2 — Dependency Audit & Security Scan** (4 PRs): Add `pyproject.toml` with project metadata, run `pip-audit` / `npm audit` and document all CVEs, pin all transitive deps with `pip-compile`, commit `package-lock.json`.
- **0.3 — CI & Baseline Metrics** (5 PRs): Add backend `pytest` + coverage to CI, add frontend `vitest` + coverage to CI, add `ruff check` + `black --check` + `mypy` to CI, add `eslint` + `tsc --noEmit` to CI, record baseline metrics (test count, coverage %, build time, bundle size).

### Out of Scope
- **Infrastructure upgrades** (PostgreSQL, Redis, Qdrant, Neo4j, pgvector) — Phase 1 scope; already has its own [BRD](../phase-1/1.0_brd_infrastructure-upgrades.md).
- **Monitoring stack** (Prometheus, Grafana, OpenTelemetry, Sentry) — Phase 2.5 scope per the roadmap.
- **Docker Compose changes** — the existing `docker-compose.yml` and `docker-compose.dev.yml` are functional; Phase 0 does not modify them.
- **Application code changes** — Phase 0 fixes tests and adds tooling; it does not change business logic, API endpoints, or frontend components.
- **CVE remediation** — Phase 0 documents CVEs and creates a remediation plan. Actual dependency upgrades to resolve CVEs happen in Phase 1 (infrastructure) and Phase 2 (backend dependencies).

| `pyproject.toml` migration does not break existing `pip install -r requirements.txt` workflow | Low: temporary dual-support (requirements.txt + pyproject.toml) can bridge the transition | PR 0.2.1 — verify both install paths work |
| Team has capacity for ~2 person-weeks in Weeks 1–2 with no competing priorities | Medium: if Phase 0 slips to Week 3, it overlaps with Phase 1 start, creating resource contention | Sprint planning confirmation before kickoff |
| Existing test infrastructure (pytest fixtures, vitest config, docker-compose test services) is functional | Medium: if test infra is broken, additional setup work is needed before tests can be fixed | First day of Phase 0 — attempt to run full suites locally |

## 6. Alternatives Considered

| Alternative | Strategic Rationale | Why Not Chosen |
|-------------|--------------------| ---------------|
| Do nothing — start Phase 1 directly on current codebase | Save 2 weeks; begin infrastructure upgrades immediately | Unacceptable: Phase 1 changes PG version, cache backend, and vector store underneath ~75K lines of code. Without green tests, regressions are invisible. The Modernization Roadmap explicitly states "Everything else builds on a green CI and known-good baseline. Do this first." |
| Fix tests only (skip dependency audit and CI improvements) | Faster — ~1 week instead of 2 | Insufficient: green tests without CI enforcement means they'll break again with the first Phase 1 PR. No security scan leaves CVE exposure unknown. No baselines means Phase 1–5 regressions are undetectable |
| Run a minimal stabilization (CI only, skip test fixes) | Even faster — ~0.5 weeks | Counterproductive: CI with failing tests generates noise, not signal. Engineers will ignore CI failures, defeating the purpose |
| Outsource dependency audit to a security vendor | Offload security scanning to specialists | Unnecessary for Phase 0: `pip-audit` and `npm audit` are automated tools. Vendor engagement is appropriate for a full security review (Phase 2, Sec Review) but overkill for an initial CVE inventory |
| Defer to a "stabilization week" after Phase 1 | Get infra upgrades done first, stabilize after | Backward: upgrading infrastructure without a test safety net is the highest-risk approach. Debugging a PG 17 migration issue is 3–5x harder without known-good test baselines |

## 8. Expected Return

| Benefit | Quantification | Timeframe |
|---------|---------------|-----------|
| Regression detection for Phase 1–5 (158 PRs) | Prevents estimated 1–2 person-weeks of undetected regression debugging per phase (5 phases × 1.5 weeks avg = 7.5 person-weeks saved) | Over Weeks 3–15 (Phases 1–5) |
| CI feedback loop on every PR | Saves ~30 min/PR of manual test running × 158 PRs = 79 hours saved | Over the full roadmap |
| CVE inventory and remediation plan | Avoids unknown security exposure; enables informed prioritization of dependency upgrades in Phase 1–2 | Immediate upon PR 0.2.2 |
| Reproducible builds via lock files | Eliminates "works on my machine" issues; estimated 2–4 hours/week saved × 15 weeks = 30–60 hours saved | Weeks 1–15 |
| Baseline metrics for regression tracking | Enables data-driven quality gates: "this PR dropped coverage from 78% to 72% — fix before merge" | Continuous from Week 2 onward |
| Unblocks Phase 1–5 execution | Prerequisite gate: $0 of the $600K+ modernization investment (estimated 15 weeks × $40K/week) can proceed safely without Phase 0 | Immediate upon Phase 0 completion |

**Estimated payback period**: Phase 0 costs ~2 person-weeks (~$20K loaded). The regression prevention value alone (7.5 person-weeks = ~$75K) yields a ~3.75x return. Payback is achieved within the first 2–3 PRs of Phase 1 where regressions would otherwise go undetected.

## 11. Dependencies & Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Failing tests reveal broken business logic (not just stale fixtures) | Medium | High — scope expands, timeline slips | Time-box root cause analysis to 2 hours per failing test. If a fix requires code changes beyond test files, create a follow-up ticket and skip with `reason="blocked on PROJ-NNN"` |
| `pip-audit` surfaces a critical CVE in a core dependency (e.g., `fastapi`, `sqlalchemy`) | Medium | Medium — may require emergency upgrade before Phase 1 | If CVSS ≥ 9.0, escalate immediately; create a hotfix PR outside Phase 0 scope. For CVSS < 9.0, document and schedule for Phase 2 |
| `pyproject.toml` migration breaks Dockerfile or CI `pip install` | Low | Low — quick fix | PR 0.2.1 is small and independently testable; keep `requirements.txt` as fallback until `requirements.lock` is proven |
| Gitea CI runner insufficient for expanded pipeline | Low | Low — slower feedback, not blocked | Monitor CI queue times after PRs 0.3.1–0.3.4; if > 15 min, parallelize jobs or add a second runner |
| Team pushback — "we should build features, not fix tests" | Medium | Medium — morale, velocity | Frame Phase 0 as the foundation that makes Phase 1–5 possible. Share the payback math: 2 weeks now saves 7.5+ weeks of regression debugging later |
| Baseline metrics show unexpectedly low coverage (< 50%) | Medium | Low — doesn't block Phase 0 completion | Record the baseline as-is. Low coverage is information, not a blocker. Coverage improvement targets belong in Phase 1–5 Test Specs |
