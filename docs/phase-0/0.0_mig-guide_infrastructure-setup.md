> **Phase**: 0 | **PR(s)**: 0.1.1–0.1.5 (test fixes), 0.2.1–0.2.4 (deps), 0.3.1–0.3.5 (CI pipeline)

## 1. Overview

This migration takes the Legal AI Platform from a broken-CI, unpinned-dependency, no-metrics state to a fully green 7-job CI pipeline with reproducible builds and baseline quality metrics. No application code is modified. No database schemas change. No Docker Compose files are altered.

| Aspect | Before | After |
|--------|--------|-------|
| CI pipeline | 3 jobs (test-backend, test-frontend, build), failing | 7+ jobs (tests ×2, lint ×2, security-audit, build, baseline-record), all green |
| Backend tests | Unknown pass rate, no coverage | 100% pass (or explicitly skipped w/ `reason=`), coverage XML/HTML artifacts |
| Frontend tests | Fails on `npm test` (vitest), no coverage | 100% pass, coverage report artifact |
| Python deps | `requirements.txt` only (no transitive pinning) | `requirements.txt` + `requirements.lock` (pip-compile w/ hashes) |
| npm deps | `package-lock.json` exists but `node_modules` not installed | `npm ci` enforced in CI from committed `package-lock.json` |
| Python project metadata | No `pyproject.toml` | `pyproject.toml` with pytest/ruff/mypy config |
| Linting (backend) | None | `ruff check .` + `mypy app/ --ignore-missing-imports` |
| Linting (frontend) | `eslint` script exists, not in CI | `eslint .` + `tsc --noEmit` in CI |
| Security scanning | None | `pip-audit` + `npm audit` (informational, non-blocking) |
| Baseline metrics | None | `baseline.json` artifact on every `main` merge (test count, coverage %, build time, bundle size) |
| Docker Compose dev env | Exists (`docker-compose.dev.yml`) but env vars reference undefined `.env` vars | Documented `.env.example` verified, startup validated < 60s |

### Step 15: CI Job — Baseline Metrics Recording + Build Gate (PR 0.3.5)

| Duration Estimate | Executor |
|-------------------|----------|
| ~1 hour | Backend Engineer |

Create the baseline recording script:

```bash
cat > scripts/record-baseline.sh << 'SCRIPT'
#!/bin/bash
set -euo pipefail

BASELINE=$(cat <<EOF
{
  "timestamp": "$(date -u +%Y-%m-%dT%H:%M:%SZ)",
  "commit": "$(git rev-parse HEAD)",
  "backend_test_count": $(grep -oP '\d+ passed' backend/pytest-output.txt | grep -oP '^\d+' || echo 0),
  "backend_coverage_pct": $(python3 -c "import xml.etree.ElementTree as ET; print(round(float(ET.parse('backend/coverage.xml').getroot().attrib['line-rate'])*100, 1))" 2>/dev/null || echo 0),
  "frontend_test_count": $(grep -oP 'Tests\s+\K\d+' frontend/vitest-output.txt 2>/dev/null || echo 0),
  "frontend_coverage_pct": $(grep -oP 'All files\s+\|\s+\K[\d.]+' frontend/coverage/coverage-summary.txt 2>/dev/null || echo 0),
  "build_time_s": ${BUILD_DURATION:-0},
  "bundle_size_kb": $(du -sk frontend/dist/ 2>/dev/null | cut -f1 || echo 0)
}
EOF
)

echo "$BASELINE" > baseline.json
echo "$BASELINE" | python3 -m json.tool
SCRIPT
chmod +x scripts/record-baseline.sh
```

Update the `build` job and add `baseline-record` job in `.gitea/workflows/ci.yml`:

```yaml
  build:
    needs: [test-backend, test-frontend, lint-backend, lint-frontend, security-audit]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
    - uses: actions/checkout@v3
    - name: Build Docker images
      run: |
        START=$(date +%s)
        docker compose build
        echo "BUILD_DURATION=$(( $(date +%s) - START ))" >> $GITHUB_ENV

  baseline-record:
    needs: [build, test-backend, test-frontend]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
    - uses: actions/checkout@v3
    - name: Download test artifacts
      uses: actions/download-artifact@v3
      with:
        path: ./artifacts
    - name: Record baseline
      run: |
        cp artifacts/backend-coverage/pytest-output.txt backend/pytest-output.txt 2>/dev/null || true
        cp artifacts/backend-coverage/coverage.xml backend/coverage.xml 2>/dev/null || true
        cp artifacts/frontend-coverage/vitest-output.txt frontend/vitest-output.txt 2>/dev/null || true
        mkdir -p frontend/coverage
        cp artifacts/frontend-coverage/coverage-summary.txt frontend/coverage/ 2>/dev/null || true
        bash scripts/record-baseline.sh
    - name: Upload baseline
      uses: actions/upload-artifact@v3
      with:
        name: baseline-metrics
        path: baseline.json
        retention-days: 90
```

**Expected output**: After a merge to `main`, `baseline.json` is produced with all 6 metrics populated.

**Verification**:
```bash
# After merging to main, download baseline.json artifact and verify:
cat baseline.json | python3 -m json.tool
# Expected: valid JSON with all fields (backend_test_count > 0, backend_coverage_pct > 0, etc.)

# Verify all 7+ jobs ran:
# Check Gitea CI dashboard — should show: test-backend, test-frontend, lint-backend, lint-frontend, security-audit, build, baseline-record
```

### Step 16: Validate Docker Compose Dev Environment

| Duration Estimate | Executor |
|-------------------|----------|
| ~30 min | Any Engineer |

This step is **validation only** — no changes to Docker Compose files (per Tech Spec NG1, ADR-001).

```bash
# Ensure .env is populated
cp .env.example .env  # If not already done — review and set passwords

# Start dev databases
time docker compose -f docker-compose.dev.yml up -d

# Wait for healthchecks
echo "Waiting for services to become healthy..."
for i in {1..60}; do
  HEALTHY=$(docker compose -f docker-compose.dev.yml ps --format json 2>/dev/null | python3 -c "
import sys, json
services = [json.loads(line) for line in sys.stdin if line.strip()]
healthy = sum(1 for s in services if 'healthy' in s.get('Health','').lower() or s.get('State','') == 'running')
print(healthy)
" 2>/dev/null || echo 0)
  echo "  Healthy services: $HEALTHY / 6 (attempt $i)"
  [ "$HEALTHY" -ge 5 ] && break  # 5 of 6 OK (ClamAV takes long)
  sleep 5
done

# Verify individual services
docker compose -f docker-compose.dev.yml ps
```

**Expected output**: All 6 services running (postgres, redis, neo4j, qdrant, minio, clamav). At minimum 5 healthy within 60s (ClamAV may take up to 5 min).

**Verification**:
```bash

]
for name, ok in checks:
    status = '✅' if ok else '❌'
    print(f'{status} {name}: {b.get(name, \"MISSING\")}')
all_ok = all(ok for _, ok in checks)
print(f'\nOverall: {\"PASS\" if all_ok else \"FAIL\"}')
sys.exit(0 if all_ok else 1)
"
```

> Open these dashboards/terminals before starting. Phase 0 has no Grafana — monitoring is manual.

| Dashboard / Query | What to Watch | Abort If |

| Gitea CI dashboard (`/repo/actions`) | Each CI job status, duration | Any job takes > 15 min (runner may be stuck) |
| `docker compose -f docker-compose.dev.yml ps` | Container health status | Any core service (PG, Redis) stays "unhealthy" for > 2 min |
| `docker stats --no-stream` | Memory usage per container | Total container memory > 8GB (OOM risk on 16GB machines) |
| Terminal: `tail -f /tmp/*-test-output.txt` | Test output during fix PRs | > 20 tests failing with same root cause (systemic issue — stop, diagnose) |
| Gitea runner status page | Runner online/offline | Runner goes offline during CI run — restart runner, re-trigger |

**Abort conditions for the overall migration**:
- If > 50% of backend tests fail due to **import errors** (not test logic) → stop, investigate `requirements.txt` compatibility
- If frontend `npm ci` fails → `package-lock.json` is corrupt or out of sync; delete and regenerate
- If Gitea runner does not support parallel jobs → revise CI to combine jobs (see Tech Spec §9 mitigation)

## 7. Rollback Procedure

### Decision Authority

| Role | Authority |
|------|-----------|
| **Can call rollback** | Any engineer on the team, migration executor |
| **Must be notified** | Tech Lead, Engineering Manager |
| **Must approve re-attempt** | Tech Lead |

### Trigger Conditions

Rollback if any of:
- A merged PR breaks `main` branch CI and cannot be fixed within 1 hour
- Docker Compose dev environment stops working after a change
- `requirements.lock` causes import failures not present with `requirements.txt`
- Frontend `npm ci` fails after `package-lock.json` changes
- More than 3 engineers report local dev environment broken

### Rollback Steps

Every step in this migration is a separate PR. Rollback = revert the specific PR.

#### Rollback a single PR:

```bash
# Identify the merge commit
git log --oneline main | head -10

# Revert it
git revert <merge-commit-sha> --no-edit
git push origin main
```

#### Rollback CI pipeline to original 3-job state:

```bash
# Nuclear option — restore original ci.yml
git checkout main
git show <last-known-good-commit>:.gitea/workflows/ci.yml > .gitea/workflows/ci.yml
git add .gitea/workflows/ci.yml
git commit -m "rollback: restore original 3-job CI pipeline"
git push origin main
```

#### Rollback `requirements.lock` (if it causes install failures):

```bash
cd backend
# CI falls back to requirements.txt
git rm requirements.lock
git commit -m "rollback: remove requirements.lock — install from requirements.txt"
git push origin main

# Update ci.yml to use requirements.txt instead of requirements.lock
sed -i 's/requirements.lock/requirements.txt/g' .gitea/workflows/ci.yml
git add .gitea/workflows/ci.yml
git commit -m "rollback: CI uses requirements.txt"
git push origin main
```

### Rollback Verification

```bash
# After any rollback, verify CI is green:
# 1. Push a no-op commit to trigger CI
git commit --allow-empty -m "ci: verify rollback"
git push origin main

# 2. Watch Gitea CI dashboard — all jobs must pass
# 3. Verify locally:
cd backend && pytest tests/ -v 2>&1 | tail -5  # should pass
cd ../frontend && npx vitest run 2>&1 | tail -5  # should pass
```
