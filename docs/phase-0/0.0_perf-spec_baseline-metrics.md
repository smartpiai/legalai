## 1. Overview

This performance specification defines the latency budgets, resource targets, and regression thresholds for the Phase 0 stabilization infrastructure: the CI pipeline, Docker Compose dev environment startup, test suite execution, and the baseline metrics collection system.

Phase 0 does not deploy to production. The "users" are engineers waiting for CI feedback and running `docker-compose up` locally. Performance here directly impacts developer velocity — every minute added to CI is multiplied by ~158 PRs across the Modernization Roadmap.

**User-facing impact**:
- CI too slow → engineers context-switch → cognitive overhead → more bugs
- Docker Compose startup too slow → engineers skip local testing → broken PRs → CI churn
- Baseline collection unreliable → no regression detection → silent quality degradation in Phases 1–5

### SLO vs. SLA Definitions

- **SLO (Service Level Objective)**: Internal targets the team commits to. Missing an SLO triggers investigation and remediation.
- **SLA (Service Level Agreement)**: Contractual commitments to customers. Missing an SLA has legal/financial consequences.

All targets below are labeled as SLO or SLA. Phase 0 is entirely internal tooling — **no SLAs exist**. All targets are SLOs representing engineering standards. SLAs will apply starting Phase 3+ when customer-facing features are deployed.

## 2. Load Profile

Phase 0 "load" is developer-driven CI pipeline invocations and local Docker Compose usage. There is no production traffic.

| Parameter | Value |
|-----------|-------|
| Typical request rate | 8–12 CI pipeline runs/day (2 engineers × 2–3 pushes each × 2 workstreams) |
| Peak multiplier | 3x during PR review bursts (end of Week 1, end of Week 2 as PRs stack up) |
| Geographic distribution | Single location — all engineers co-located or on same Gitea runner |
| Tenant size distribution | N/A — single team, no multi-tenancy in CI |
| Concurrent users per tenant | 1–4 engineers pushing simultaneously; Gitea runner processes 1 pipeline at a time (pending open question #1 re: parallel job support) |
| Local dev environment | 2–4 engineers running `docker-compose.dev.yml` simultaneously on individual machines |

## 3. Performance Budgets

### Latency

| Operation | p50 Target | p95 Target | p99 Target | Hard Limit | Type |
|-----------|-----------|-----------|-----------|------------|------|
| CI pipeline total (all jobs, push → green/red) | 6 min | 8 min | 10 min | 12 min timeout | SLO |
| CI feedback latency (push → first status visible) | 30s | 60s | 90s | 120s | SLO |
| CI push-to-final-status (push → all checks reported) | 7 min | 10 min | 12 min | 15 min timeout | SLO |
| `test-backend` job (pytest + coverage) | 2 min | 3 min | 4 min | 5 min timeout | SLO |
| `test-frontend` job (vitest + coverage) | 2 min | 3 min | 4 min | 5 min timeout | SLO |
| `lint-backend` job (ruff + mypy) | 45s | 90s | 2 min | 3 min timeout | SLO |
| `lint-frontend` job (eslint + tsc) | 45s | 90s | 2 min | 3 min timeout | SLO |
| `security-audit` job (pip-audit + npm audit) | 60s | 2 min | 3 min | 4 min timeout | SLO |
| `build` job (docker-compose build) | 2 min | 3 min | 4 min | 5 min timeout | SLO |
| `baseline-record` job (metrics collection) | 15s | 30s | 45s | 60s timeout | SLO |
| Docker Compose dev startup (all 6 services healthy) | 45s | 90s | 2 min | 3 min | SLO |
| Docker Compose full stack startup (10 services healthy) | 90s | 3 min | 4 min | 5 min (excl. ClamAV) | SLO |

**Critical path budget** (parallel execution assumed):

```
Critical Path = max(test-backend, test-frontend, lint-backend, lint-frontend, security-audit)
              + build + baseline-record
              = max(3, 3, 1.5, 1.5, 2) + 3 + 0.5
              = 3 + 3 + 0.5 = 6.5 min (p50)
              = max(4, 4, 2, 2, 3) + 3 + 1 = 8 min (p95, under 10 min NFR-01)
```

**Sequential fallback** (if Gitea runner does not support parallel jobs):

```
Sequential = 3 + 3 + 1.5 + 1.5 + 2 + 3 + 0.5 = 14.5 min (p50, EXCEEDS NFR-01)
Mitigation: Combine lint-backend + lint-frontend → single lint job (saves ~1.5 min)
            Combine security-audit into lint job (saves ~2 min)
            Revised sequential = 3 + 3 + 3 + 3 + 0.5 = 12.5 min (still exceeds)
            Further: Single runner with concurrent job support or add second runner
```

### Cold Start vs. Warm Performance

| Operation | Cold (first run) | Warm (cached) | Notes |
|-----------|-----------------|---------------|-------|
| CI `test-backend` job | +60s for `pip install -r requirements.lock` | Within budget (pip cache hit) | Gitea runner caches `~/.cache/pip` between runs |
| CI `test-frontend` job | +45s for `npm ci` | Within budget (node_modules cache) | Cache `~/.npm` between runs |
| CI `build` job (docker-compose build) | +3–5 min for full image build (no layer cache) | Within budget (Docker layer cache hit) | First run after Dockerfile change is cold |
| Docker Compose dev (fresh pull) | +2–5 min for image pulls (PG, Redis, Neo4j, Qdrant, MinIO, ClamAV) | Within budget (images cached locally) | Only on first run or after `docker-compose pull` |
| Docker Compose dev (volumes exist) | Within budget | Within budget (data persisted) | PG, Neo4j skip initialization if volumes have data |
| Neo4j startup (cold, no data) | 30–40s (start_period in healthcheck) | 15–20s (data already loaded) | Neo4j plugin loading (APOC, GDS) adds startup time |
| ClamAV startup (signature download) | 3–5 min (start_period: 300s) | 60–90s (signatures cached in volume) | ClamAV excluded from startup time SLO; it's optional for dev |

## 4. Cost Projections

Phase 0 has no cloud infrastructure costs — all execution is on an existing Gitea CI runner and developer laptops. Cost is measured in **engineer-time**.

| Operation | Cost / Unit | Current Monthly Volume | Projected Volume (6 mo) | Current Monthly Cost | Projected Monthly Cost |
|-----------|------------|----------------------|------------------------|---------------------|----------------------|
| CI pipeline run (engineer wait time) | 8 min avg × $100/hr eng rate = $13.33/run | 40 runs (current 3-job pipeline, ~3 min) | 240 runs/mo (Phases 1–5, full 7-job pipeline) | $222 (40 × $5.00 @ 3 min) | $3,200 (240 × $13.33) |
| CI pipeline run (compute cost) | $0 (self-hosted Gitea runner) | 40 runs | 240 runs | $0 | $0 |
| Docker image pulls (bandwidth) | $0 (Docker Hub free tier) | ~20 pulls/mo | ~50 pulls/mo | $0 | $0 |
| CI artifact storage (90 day retention) | $0 (Gitea local storage) | 40 × 50 MB = 2 GB | 240 × 50 MB = 12 GB | $0 | $0 — verify Gitea runner disk capacity |
| Developer local Docker (electricity/resources) | Negligible | — | — | $0 | $0 |
| **Total** | | | | **$222** | **$3,200** (engineer wait time only) |

**Cost optimization opportunity**: Reducing CI pipeline p50 from 8 min to 5 min saves ~$960/month in engineer wait time at projected Phase 1–5 volume. Primary lever: Docker layer caching and pip/npm caching.

### Baseline Collection Procedure

Run the following on Day 1 of Phase 0 (2026-04-07) and record results:

```bash
# 1. CI pipeline baseline — trigger 3 runs on current pipeline, record times
#    Read from Gitea CI dashboard: Settings > Actions > Recent Runs

# 2. Docker Compose dev startup (cold — remove volumes first)
docker-compose -f docker-compose.dev.yml down -v
time docker-compose -f docker-compose.dev.yml up -d --wait
# Record: real time from `time` output
# Verify: all 6 containers healthy
docker-compose -f docker-compose.dev.yml ps

# 3. Docker Compose dev startup (warm — volumes exist)
docker-compose -f docker-compose.dev.yml down
time docker-compose -f docker-compose.dev.yml up -d --wait
# Record: real time from `time` output

# 4. Docker Compose full stack startup (cold)
docker-compose down -v
time docker-compose up -d --wait
# Record: real time (note ClamAV may take 3-5 min; record with and without)

# 5. Backend test execution
cd backend
pip install -r requirements.txt
time pytest -v 2>&1 | tee pytest-output.txt
# Record: real time, test count (from "X passed, Y failed, Z skipped")
# Memory: /usr/bin/time -v pytest -v 2>&1 | grep "Maximum resident"

# 6. Frontend test execution
cd frontend
npm ci
time npx vitest run 2>&1 | tee vitest-output.txt
# Record: real time, test count (from vitest summary)

# 7. Frontend bundle size

# Record: total KB
```

## 7. Benchmark Design

### Test Setup

| Parameter | Value |
|-----------|-------|
| Environment | Gitea CI runner (spec TBD — document CPU/RAM on Day 1) + developer laptop (Apple Silicon / 16 GB typical) |
| Dataset | Existing test suites (~75K LoC codebase, test count TBM) |
| Concurrency | 1 CI pipeline at a time; 5 parallel CI jobs within pipeline (if runner supports) |
| Duration | Single pipeline run (~8 min); repeat 3x for p50/p95 statistical significance |
| Tool | `time` (wall clock), Gitea CI dashboard (per-job timing), `docker stats` (memory), `du` (disk) |

### Test Scenarios

| Scenario | Description | Key Metric |
|----------|-------------|------------|
| S1: CI pipeline cold run | Full pipeline, no caches (fresh runner workspace) | Total wall-clock time |
| S2: CI pipeline warm run | Full pipeline, pip/npm/Docker caches populated | Total wall-clock time |
| S3: Individual job timing | Each of the 7 CI jobs measured independently | Per-job duration |
| S4: Docker Compose dev cold start | `docker-compose.dev.yml up` with no volumes | Time to all-healthy |
| S5: Docker Compose dev warm start | `docker-compose.dev.yml up` with existing volumes | Time to all-healthy |
| S6: Docker Compose full stack cold start | `docker-compose.yml up` with no volumes | Time to all-healthy (excl. ClamAV) |
| S7: Backend test suite wall time | `pytest` full suite, no parallel | Wall clock + test count |
| S8: Frontend test suite wall time | `vitest run` full suite | Wall clock + test count |
| S9: Sequential CI fallback | All jobs run sequentially (simulates single-threaded runner) | Total wall-clock time |
| S10: Resource usage under full stack | All 10 Docker services running + backend tests executing | Peak memory per container |

### Benchmark Execution Script

```bash
#!/bin/bash
# scripts/benchmark-phase0.sh — Run all Phase 0 performance benchmarks
set -euo pipefail

RESULTS_DIR="benchmarks/phase0/$(date +%Y%m%d-%H%M%S)"
mkdir -p "$RESULTS_DIR"

echo "=== S4: Docker Compose dev cold start ==="
docker-compose -f docker-compose.dev.yml down -v 2>/dev/null || true
{ time docker-compose -f docker-compose.dev.yml up -d --wait ; } 2>"$RESULTS_DIR/s4-cold-start.txt"
docker-compose -f docker-compose.dev.yml ps > "$RESULTS_DIR/s4-services.txt"

echo "=== S5: Docker Compose dev warm start ==="
docker-compose -f docker-compose.dev.yml down
{ time docker-compose -f docker-compose.dev.yml up -d --wait ; } 2>"$RESULTS_DIR/s5-warm-start.txt"

echo "=== S10: Resource usage ==="
docker stats --no-stream --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}\t{{.MemPerc}}" \
  > "$RESULTS_DIR/s10-resource-usage.txt"

echo "=== S7: Backend test suite ==="
cd backend
{ time pytest -v 2>&1 ; } > "$RESULTS_DIR/s7-backend-tests.txt" 2>&1
cd ..

echo "=== S8: Frontend test suite ==="
cd frontend
npm ci --silent
{ time npx vitest run 2>&1 ; } > "$RESULTS_DIR/s8-frontend-tests.txt" 2>&1
cd ..

echo "=== Frontend bundle size ==="
cd frontend
npm run build > "$RESULTS_DIR/bundle-build.txt" 2>&1
du -sk dist/ > "$RESULTS_DIR/bundle-size.txt"
cd ..

echo "=== Cleanup ==="
docker-compose -f docker-compose.dev.yml down

echo "Results saved to $RESULTS_DIR"
ls -la "$RESULTS_DIR"
```

## 8. Regression Thresholds

These thresholds apply to `baseline.json` values compared between sequential `main` merges. In Phase 0, enforcement is manual (diff `baseline.json` files). Starting Phase 1, these feed into CI gates.

| Metric | Allowed Regression | Action if Exceeded |
|--------|-------------------|-------------------|
| CI pipeline total duration | < 15% increase (e.g., 8 min → 9.2 min max) | **Investigate** — identify which job got slower; file ticket |
| `test-backend` job duration | < 20% increase | **Investigate** — check for added slow tests or missing mocks |
| `test-frontend` job duration | < 20% increase | **Investigate** — check for added slow tests or missing mocks |
| `build` job duration (docker-compose build) | < 25% increase | **Investigate** — check for Dockerfile changes invalidating cache layers |
| Backend test count | Must not decrease (0% regression) | **Block merge** — tests must not be deleted without skip+reason |
| Backend coverage % | < 2% decrease | **Investigate** — new code without tests; review in PR |
| Frontend test count | Must not decrease (0% regression) | **Block merge** — tests must not be deleted without skip+reason |
| Frontend coverage % | < 2% decrease | **Investigate** — new code without tests; review in PR |
| Frontend bundle size | < 10% increase | **Investigate** — check for new heavy dependencies or missing tree-shaking |
| Docker Compose dev startup time | < 25% increase | **Investigate** — check for new services or heavier health checks |

### Phase 0 Enforcement Mechanism

Phase 0 does **not** have automated CI regression gates (those are Phase 1+ scope). Enforcement is:

1. `baseline-record` job produces `baseline.json` on every `main` merge
2. Tech Lead manually compares `baseline.json` across runs: `diff <(jq -S . baseline-prev.json) <(jq -S . baseline-curr.json)`
3. Regressions exceeding thresholds above are raised in PR review for the next merge
4. **Phase 1 action**: Add `scripts/check-regression.sh` to CI that auto-compares and blocks merge

## 9. Monitoring in Production

Phase 0 has no production deployment and no monitoring stack (Prometheus/Grafana are Phase 2.5 scope). Monitoring is manual and CI-artifact-based.

| Metric | Dashboard | Alert Condition |
|--------|-----------|-----------------|
| CI pipeline duration | Gitea CI → Actions → Run History (manual review) | Any single run > 10 min — Tech Lead investigates same day |
| CI job failure rate | Gitea CI → Actions → filter by status:failed | > 3 consecutive failures on `main` — investigate runner health |
| `baseline.json` regression | Manual `diff` of JSON artifacts (see §8) | Any metric exceeding regression threshold — raised in next standup |
| Docker Compose startup time | Manual benchmark run (§7 script) | Run weekly; any service consistently > 2x expected startup — investigate |
| CI runner disk usage | `df -h` on runner machine (manual SSH) | > 80% disk — clean old workspaces: `docker system prune -af` |
| CI runner memory | `free -h` on runner machine (manual SSH) | < 1 GB available during pipeline run — investigate OOM risk |

**Phase 1+ migration path**: All manual checks above will be replaced by:
- Grafana dashboards with Gitea CI webhook data (Phase 2.5)
- Automated `baseline.json` comparison in CI (Phase 1)
- Runner health monitoring via node_exporter + Prometheus (Phase 2.5)

## 11. Graceful Degradation Tiers

Phase 0 degradation applies to the CI pipeline and dev environment — not production services.

| Tier | Trigger Condition | What Degrades | User Impact | Auto / Manual |
|------|-------------------|---------------|-------------|---------------|
| 1 — Elevated | CI pipeline > 10 min (NFR-01 exceeded) | Reduce parallelism debugging: combine lint jobs into single job | Engineers wait longer; prioritize cache optimization | Manual — Tech Lead decision |
| 2 — High | CI pipeline > 15 min OR Gitea runner at > 90% CPU for > 5 min | Disable `security-audit` job on PRs (run only on `main` merges) | CVE scanning delayed to merge time; no PR-level audit | Manual — Tech Lead decision |
| 3 — Critical | CI runner offline OR CI pipeline consistently > 20 min | Fall back to local-only testing: engineers run `pytest` + `vitest` locally and attest in PR description | No automated quality gate; high risk of regressions; PR description includes test output | Manual — Tech Lead + all engineers notified |
| 4 — Survival | Gitea runner disk full OR OOM-killed repeatedly | Emergency: `docker system prune -af` on runner; disable `build` job (skip Docker build verification) | Only tests and lints run in CI; Docker build verified locally by PR author | Manual — Tech Lead + DevOps immediate action |

### Dev Environment Degradation

| Tier | Trigger Condition | What Degrades | User Impact | Auto / Manual |
|------|-------------------|---------------|-------------|---------------|
| 1 — Elevated | Docker Compose dev startup > 2 min | Skip ClamAV: `docker-compose.dev.yml` without clamav service | No virus scanning in dev (acceptable for Phase 0) | Manual — engineer choice |
| 2 — High | Docker memory > 80% of available | Stop Neo4j and Qdrant (not needed for most backend unit tests) | Graph and vector features unavailable locally | Manual — engineer choice |
| 3 — Critical | Docker Compose dev fails to start (port conflicts, image pull failures) | Fall back to host-native services: `brew install postgresql redis` | Different versions than Docker; potential parity issues | Manual — engineer documents divergence |

## 12. Optimization Opportunities

Known areas where performance can be improved if budgets are exceeded.

| Opportunity | Expected Improvement | Effort | Trade-off |
|-------------|---------------------|--------|-----------|
| **Pip cache in CI** — cache `~/.cache/pip` between runs | -30–60s on `test-backend` and `lint-backend` cold start | S | Cache invalidation on `requirements.lock` change; stale cache risk |
| **npm cache in CI** — cache `~/.npm` between runs | -20–45s on `test-frontend` and `lint-frontend` cold start | S | Cache invalidation on `package-lock.json` change |
| **Docker layer caching in CI** — mount Docker buildkit cache | -60–180s on `build` job for unchanged layers | M | Requires Gitea runner configuration; cache disk usage |
| **Combine lint jobs** — merge `lint-backend` + `lint-frontend` into single `lint` job | -90s total (saves job startup overhead) | S | Less granular failure reporting (one "lint" status vs two) |
| **pytest-xdist** — parallel test execution across CPU cores | -30–50% on `test-backend` execution time | M | Requires test isolation (no shared state); potential flakiness |
| **vitest thread workers** — increase worker threads | -10–20% on `test-frontend` execution time | S | Already partially parallel by default; diminishing returns |
| **Docker Compose profiles** — define `dev-minimal` profile excluding ClamAV | -60–90s on dev startup (ClamAV is 3–5 min cold) | S | ClamAV unavailable; document scanning untestable locally |
| **Pre-built CI base image** — custom Docker image with Python 3.11 + Node 18 + tools pre-installed | -60–120s on every CI job (skip tool installation) | M | Maintenance burden: rebuild image when tool versions change |
| **Selective security audit** — run `pip-audit`/`npm audit` only when dependency files change | -60–180s on non-dependency PRs | S | CVEs in transitive deps may be missed if new advisory published between dep changes |
