## 1. Service Overview

Phase 0 infrastructure provides the local development environment (Docker Compose with 6 database services) and the CI/CD pipeline (Gitea Actions with 7 jobs). No production deployment exists. The "users" are engineers running local dev and pushing PRs.

### Dependencies

| Dependency | Type | Impact if Unavailable |
|------------|------|----------------------|
| Docker Engine 24+ / Compose v2 | Required | Cannot start any dev services; local development blocked |
| PostgreSQL 15 | Required | Backend cannot start; all CRUD operations fail |
| Redis 7 | Required | No caching, no Celery task queue, sessions fail |
| Neo4j 5 | Required | Graph queries fail; GraphRAG features unavailable |
| Qdrant | Required | Vector search fails; embedding-based retrieval broken |
| MinIO | Required | Document upload/download fails; S3-compatible storage unavailable |
| ClamAV | Degraded | No virus scanning on uploads; dev continues without it |
| Gitea CI Runner | Required (CI only) | CI pipelines queue indefinitely; PRs get no status checks |
| `.env` file | Required | Docker Compose fails to interpolate env vars; all services fail to start |

### Ports & Endpoints

| Endpoint / Service | Port | Purpose |
|----------|------|---------|
| PostgreSQL | `${POSTGRES_PORT}` (default: 5432) | Primary relational database |
| Redis | `${REDIS_PORT}` (default: 6379) | Cache, sessions, Celery broker |
| Neo4j Browser | `${NEO4J_HTTP_PORT}` (default: 7474) | Neo4j web UI |
| Neo4j Bolt | `${NEO4J_PORT}` (default: 7687) | Neo4j driver protocol |
| Qdrant REST | `${QDRANT_PORT}` (default: 6333) | Vector DB HTTP API |
| Qdrant gRPC | `${QDRANT_GRPC_PORT}` (default: 6334) | Vector DB gRPC API |
| MinIO API | `${MINIO_PORT}` (default: 9000) | S3-compatible object storage |
| MinIO Console | `${MINIO_CONSOLE_PORT}` (default: 9001) | MinIO admin web UI |
| ClamAV | `${CLAMAV_PORT}` (default: 3310) | Antivirus scanning daemon |
| Backend API (full stack only) | `${BACKEND_PORT}` (default: 8000) | FastAPI application |
| Frontend (full stack only) | `${FRONTEND_PORT}` (default: 3000) | React dev server |

### Health Check Definition

Phase 0 has no unified `/health` endpoint — health is determined per-service via Docker healthchecks.

**Individual service health checks:**

```bash
# PostgreSQL
docker exec legal-ai-postgres-dev pg_isready -U ${POSTGRES_USER:-postgres} -d ${POSTGRES_DB:-legal_ai}

# Redis
docker exec legal-ai-redis-dev redis-cli ping
# Expected: PONG

# Neo4j
docker exec legal-ai-neo4j-dev cypher-shell -u neo4j -p "${NEO4J_PASSWORD:-password}" "RETURN 1"
# Expected: 1

# Qdrant
curl -sf http://localhost:${QDRANT_PORT:-6333}/healthz
# Expected: (empty 200 OK) or JSON status

# MinIO
curl -sf http://localhost:${MINIO_PORT:-9000}/minio/health/live
# Expected: (empty 200 OK)

# ClamAV
docker exec legal-ai-clamav-dev clamdscan --ping
# Expected: PONG
```

**Status determination rules:**

| Condition | Status | Action |
|-----------|--------|--------|
| All 6 services report healthy via `docker compose ps` | Healthy | None |
| ClamAV unhealthy, others healthy | Degraded | Continue dev; ClamAV is optional |
| Any of PG/Redis/Neo4j/Qdrant/MinIO unhealthy | Unhealthy | Investigate — see [Section 5](#5-common-issues--troubleshooting) |
| Docker daemon not running | Down | Start Docker Desktop or `systemctl start docker` |

## 2. Deployment

### Pre-Deployment Checklist

Phase 0 has no production deployment. "Deployment" = starting the local dev environment or merging CI pipeline changes.

- [ ] Docker Desktop / Docker Engine is running: `docker info > /dev/null 2>&1 && echo OK || echo "Docker not running"`
- [ ] `.env` file exists and is populated: `test -f .env && echo OK || echo "MISSING — run: cp .env.example .env"`
- [ ] Docker Compose v2 available: `docker compose version`
- [ ] Sufficient disk space (≥ 10 GB free): `df -h . | awk 'NR==2{print $4}'`
- [ ] Sufficient memory (≥ 8 GB total for full stack, ≥ 4 GB for dev): `sysctl hw.memsize 2>/dev/null | awk '{printf "%.0f GB\n", $2/1073741824}' || free -h 2>/dev/null | awk '/Mem:/{print $2}'`
- [ ] No port conflicts on required ports: `for p in 5432 6379 7474 7687 6333 6334 9000 9001 3310; do (echo > /dev/tcp/localhost/$p) 2>/dev/null && echo "PORT $p IN USE — conflict!" || true; done`

### Deploy Steps — Dev Environment (docker-compose.dev.yml — databases only)

```bash
# Step 1: Ensure .env exists
test -f .env || cp .env.example .env

# Step 2: Start dev database services
docker compose -f docker-compose.dev.yml up -d

# Step 3: Wait for services to become healthy (up to 90s, excluding ClamAV)
echo "Waiting for services..."
for i in $(seq 1 18); do
  HEALTHY=$(docker compose -f docker-compose.dev.yml ps --format json 2>/dev/null \
    | grep -v clamav \
    | grep -c '"healthy"' 2>/dev/null || echo 0)
  echo "  Healthy (excl. ClamAV): ${HEALTHY}/5 (${i}×5s)"
  [ "${HEALTHY}" -ge 5 ] && echo "All core services healthy!" && break
  sleep 5
done
```

### Deploy Steps — Full Stack (docker-compose.yml — databases + app services)

```bash
# Step 1: Ensure .env exists
test -f .env || cp .env.example .env

# Step 2: Start all services
docker compose up -d

# Step 3: Wait for backend to be healthy (depends on all DBs being healthy first)
echo "Waiting for full stack..."
for i in $(seq 1 30); do
  BE_HEALTH=$(curl -sf http://localhost:${BACKEND_PORT:-8000}/api/v1/health/ready 2>/dev/null && echo "UP" || echo "DOWN")
  echo "  Backend: ${BE_HEALTH} (${i}×10s)"
  [ "${BE_HEALTH}" = "UP" ] && echo "Full stack healthy!" && break
  sleep 10
done
```

### Post-Deployment Verification

```bash
# Run all health checks in one shot
echo "=== Service Health ==="
docker exec legal-ai-postgres-dev pg_isready -U postgres -d legal_ai && echo "✅ PostgreSQL" || echo "❌ PostgreSQL"
docker exec legal-ai-redis-dev redis-cli ping | grep -q PONG && echo "✅ Redis" || echo "❌ Redis"
docker exec legal-ai-neo4j-dev cypher-shell -u neo4j -p "${NEO4J_PASSWORD:-password}" "RETURN 1" > /dev/null 2>&1 && echo "✅ Neo4j" || echo "❌ Neo4j"
curl -sf http://localhost:${QDRANT_PORT:-6333}/healthz > /dev/null && echo "✅ Qdrant" || echo "❌ Qdrant"
curl -sf http://localhost:${MINIO_PORT:-9000}/minio/health/live > /dev/null && echo "✅ MinIO" || echo "❌ MinIO"
docker exec legal-ai-clamav-dev clamdscan --ping > /dev/null 2>&1 && echo "✅ ClamAV" || echo "⚠️ ClamAV (optional — may still be starting)"
echo "=== Done ==="
```

## 4. Known Limits

| Resource | Hard Limit | Symptom When Exceeded | Action |
|----------|-----------|----------------------|--------|
| PostgreSQL connections | 100 (default `max_connections`) | `FATAL: too many connections for role` in PG logs; backend 500 errors | Restart PG container; investigate connection leak. Increase with `POSTGRES_MAX_CONNECTIONS` env var |
| PostgreSQL disk (volume) | Host disk capacity | `FATAL: could not write to file` or `No space left on device` | `docker system prune -af`; expand host disk; run `VACUUM FULL` |
| Redis memory | Host RAM (no `maxmemory` set in dev) | `OOM command not allowed` errors; Redis crash-restarts | Set `maxmemory 256mb` via redis.conf mount; investigate cache size |
| Neo4j heap | 2 GB max (configured in docker-compose.yml) | `java.lang.OutOfMemoryError: Java heap space` in Neo4j logs | Increase `NEO4J_server_memory_heap_max__size` in compose file; restart |
| Neo4j transaction timeout | 30s (default) | `TransactionTimedOut` errors on complex Cypher queries | Optimize query; increase `dbms.transaction.timeout` if needed |
| Qdrant storage | Host disk capacity | Collection operations fail; gRPC errors | Prune unused collections; expand host disk |
| MinIO storage | Host disk capacity | Upload failures; `disk full` errors in MinIO logs | Delete old test uploads; expand host disk |
| ClamAV startup | 5 min (start_period: 300s) | Shows `starting` health for up to 5 min; `unhealthy` after | Wait 5 min; if still failing: `docker compose restart clamav` |
| Docker Desktop memory | Configured in Docker settings (default varies) | Containers OOM-killed; `Exited (137)` status | Increase Docker memory limit in Docker Desktop → Resources → Memory |
| CI runner concurrent pipelines | 1 pipeline (pending verification of runner config) | Queued pipelines; long wait times with no feedback | Add second runner; or wait for current pipeline to finish |
| CI artifact storage | Gitea runner disk capacity | Artifact upload fails; warning in CI logs | Prune old artifacts; reduce retention from 90 to 60 days |
| Host ports | 1 per port number | `Bind for 0.0.0.0:XXXX failed: port is already allocated` | Find conflicting process: `lsof -i :XXXX`; stop it or change port in `.env` |

### Issue 1: `.env` File Missing or Incomplete

**Severity**: P0

**Symptoms**: `docker compose up` fails with errors like `variable is not set` or services start with empty passwords/ports.

**Root Cause**: `.env` file not created from `.env.example`, or key variables are blank.

**Diagnosis**:
```bash
# Check .env exists
test -f .env && echo ".env exists" || echo ".env MISSING"

# Check critical vars are set
for var in POSTGRES_USER POSTGRES_PASSWORD POSTGRES_DB REDIS_PORT NEO4J_PASSWORD MINIO_ROOT_USER MINIO_ROOT_PASSWORD; do
  grep -q "^${var}=" .env 2>/dev/null && echo "✅ ${var}" || echo "❌ ${var} missing"
done
```

**Resolution**:
```bash
# Copy example and use as-is for dev (defaults are safe for local dev)
cp .env.example .env
echo "✅ .env created from .env.example"

# Restart
docker compose -f docker-compose.dev.yml up -d
```

**Prevention**: Add `.env` existence check to `README.md` and any `make dev` / startup scripts. Never commit `.env` to git (it should be in `.gitignore`).

### Issue 2: Port Conflict

**Severity**: P1

**Symptoms**: `Bind for 0.0.0.0:XXXX failed: port is already allocated` during `docker compose up`.

**Root Cause**: Another process (or stale Docker container) is using the required port.

**Diagnosis**:
```bash
# Find what's using the port (replace 5432 with the conflicting port)
lsof -i :5432 | head -5

# Check for stale containers from other compose projects
docker ps -a --format "table {{.Names}}\t{{.Ports}}\t{{.Status}}" | grep -E "5432|6379|7474|7687|6333|9000"
```

**Resolution**:
```bash
# Option A: Kill the conflicting process
kill $(lsof -t -i :5432)

# Option B: Change the port in .env (e.g., use 5433 instead of 5432)
sed -i.bak 's/POSTGRES_PORT=5432/POSTGRES_PORT=5433/' .env

# Option C: Stop stale Docker containers from another project
docker stop $(docker ps -q --filter "publish=5432")

# Restart
docker compose -f docker-compose.dev.yml up -d
```

**Prevention**: Use unique port ranges per project. Stop other Docker projects before starting this one.

### Issue 3: PostgreSQL Restart Loop

**Severity**: P1

**Symptoms**: `legal-ai-postgres-dev` shows `Restarting` or `Exited` status. Logs show `FATAL: password authentication failed` or `data directory has wrong ownership`.

**Root Cause**: Password mismatch between `.env` and initialized volume, or corrupted data volume.

**Diagnosis**:
```bash
# Check container status and logs
docker compose -f docker-compose.dev.yml ps postgres
docker logs legal-ai-postgres-dev --tail=30
```

**Resolution**:
```bash
# If password mismatch (changed POSTGRES_PASSWORD after first init):
# Must wipe the PG volume — data will be lost
docker compose -f docker-compose.dev.yml down
docker volume rm $(docker volume ls -q | grep postgres_dev_data)
docker compose -f docker-compose.dev.yml up -d postgres

# If corrupted data:
docker compose -f docker-compose.dev.yml down
docker volume rm $(docker volume ls -q | grep postgres_dev_data)
docker compose -f docker-compose.dev.yml up -d
```

**Prevention**: Don't change `POSTGRES_PASSWORD` after initial volume creation without also removing the volume. Document this in onboarding.

### Issue 4: Neo4j Unhealthy

**Severity**: P1

**Symptoms**: `legal-ai-neo4j-dev` shows `unhealthy`. Logs show plugin download failures, `OutOfMemoryError`, or `Address already in use`.

**Root Cause**: APOC/GDS plugin download fails (network issue), heap too small for data, or port conflict on 7474/7687.

**Diagnosis**:
```bash
# Check logs for specific error
docker logs legal-ai-neo4j-dev --tail=50 2>&1 | grep -iE "error|exception|oom|plugin|download"

# Check if it's still starting (Neo4j can take 60s+ on first boot with plugin downloads)
docker inspect legal-ai-neo4j-dev --format '{{.State.Health.Status}}'
```

**Resolution**:
```bash
# If plugin download failed (network):
docker compose -f docker-compose.dev.yml restart neo4j
# Wait 60s+ for plugins to download

# If OOM:
# Increase heap — edit docker-compose.yml (full stack) or docker-compose.dev.yml (dev):
# NEO4J_server_memory_heap_max__size=4G
docker compose -f docker-compose.dev.yml restart neo4j

# If volume corrupted:
docker compose -f docker-compose.dev.yml down
docker volume rm $(docker volume ls -q | grep neo4j_dev_data)
docker compose -f docker-compose.dev.yml up -d neo4j
```

**Prevention**: Pre-pull Neo4j image with plugins. Consider using a fixed Neo4j version tag instead of `neo4j:5` (major version float can break).

### Issue 5: Container OOM-Killed (Exit 137)

**Severity**: P1

**Symptoms**: Container shows `Exited (137)`. `docker inspect <container> --format '{{.State.OOMKilled}}'` returns `true`.

**Root Cause**: Container exceeded Docker Desktop memory limit or host is memory-constrained.

**Diagnosis**:
```bash
# Check which container was OOM-killed
docker ps -a --format "table {{.Names}}\t{{.Status}}" | grep "137"

# Confirm OOM
docker inspect legal-ai-neo4j-dev --format '{{.State.OOMKilled}}'

# Check total Docker memory usage
docker stats --no-stream --format "table {{.Name}}\t{{.MemUsage}}\t{{.MemPerc}}"
```

**Resolution**:
```bash
# Immediate: restart the killed container
docker compose -f docker-compose.dev.yml up -d

# If recurring: increase Docker Desktop memory limit
# Docker Desktop → Settings → Resources → Memory → increase to 8 GB+

# If on limited hardware: use dev profile without ClamAV and stop non-essential services
docker compose -f docker-compose.dev.yml up -d postgres redis neo4j qdrant minio
# (skip clamav)
```

**Prevention**: Set Docker Desktop memory to at least 6 GB for dev (8 GB for full stack). Monitor with `docker stats`.

### Issue 6: Qdrant Healthcheck Fails

**Severity**: P2

**Symptoms**: Qdrant shows `unhealthy` in one compose file but works fine when tested manually.

**Root Cause**: `docker-compose.dev.yml` uses `curl -f http://localhost:6333/` while `docker-compose.yml` uses a directory existence check (`test -d /qdrant/storage`). The `curl` method is more reliable.

**Diagnosis**:
```bash
# Test manually
curl -sf http://localhost:${QDRANT_PORT:-6333}/healthz && echo "Qdrant API OK" || echo "Qdrant API FAIL"
curl -sf http://localhost:${QDRANT_PORT:-6333}/collections && echo "Qdrant collections OK" || echo "Qdrant collections FAIL"
```

**Resolution**:
```bash
# If Qdrant is actually running but healthcheck false-alarms:
# Check from inside the container
docker exec legal-ai-qdrant-dev curl -sf http://localhost:6333/healthz && echo OK || echo FAIL

# Restart if genuinely stuck
docker compose -f docker-compose.dev.yml restart qdrant
```

**Prevention**: Standardize healthcheck to `curl -f http://localhost:6333/healthz` across both compose files (Phase 1 scope — Tech Spec open question #6).

### Issue 7: ClamAV Slow Startup

**Severity**: P2

**Symptoms**: ClamAV container shows `health: starting` for 3–5 minutes. Other services are healthy.

**Root Cause**: ClamAV downloads virus signature database on first start (or when signatures are stale). This is a ~300 MB download. `start_period: 300s` gives it 5 minutes.

**Diagnosis**:
```bash
# Check ClamAV logs for download progress
docker logs legal-ai-clamav-dev --tail=20 2>&1 | grep -iE "download|update|fresh|loaded"
```

**Resolution**:
```bash
# Option A: Just wait — ClamAV is optional for dev. 5 min is normal.

# Option B: Skip ClamAV entirely (recommended for most dev work)
docker compose -f docker-compose.dev.yml up -d postgres redis neo4j qdrant minio

# Option C: If ClamAV fails after 5 min, restart with cached signatures
docker compose -f docker-compose.dev.yml restart clamav
```

**Prevention**: ClamAV signature volume (`clamav_dev_data`) persists signatures. After first successful start, subsequent starts take 60–90s. Don't `docker volume rm` the ClamAV volume unnecessarily.

### Issue 8: CI Runner Offline

**Severity**: P0

**Symptoms**: PR pushed > 20 min ago with no CI status checks appearing. Gitea Actions page shows jobs stuck in "Queued" state.

**Root Cause**: Gitea CI runner process crashed, lost network, or machine rebooted.

**Diagnosis**:
```bash
# Check runner status in Gitea admin UI:
# Gitea → Site Administration → Runners
# Look for runner with "Offline" or "Idle" status

# If SSH access to runner machine:
ssh <runner-host>
systemctl status gitea-runner    # or: docker ps | grep runner
ps aux | grep act_runner
```

**Resolution**:
```bash
# Restart runner (systemd)
ssh <runner-host>
sudo systemctl restart gitea-runner

# Restart runner (Docker)
ssh <runner-host>
docker restart gitea-runner

# Verify runner is back online
# Gitea → Site Administration → Runners → should show "Idle"

# Re-trigger stuck pipelines
# Gitea → Repo → Actions → select stuck run → "Re-run all jobs"
```

**Prevention**: Set up a cron watchdog on the runner machine. Minimum: `systemctl enable gitea-runner` for auto-restart on reboot.

### Issue 9: CI `test-backend` Import Error

**Severity**: P1

**Symptoms**: `test-backend` job fails with `ModuleNotFoundError: No module named 'xyz'` or `ImportError`.

**Root Cause**: `requirements.lock` is out of sync with code — a new import was added but `requirements.lock` wasn't regenerated.

**Diagnosis**:
```bash
# Check CI logs for the specific missing module
# Gitea → Actions → failed run → test-backend → find "ModuleNotFoundError"

# Locally verify:
cd backend
source venv/bin/activate
pip install -r requirements.lock
pytest tests/ -v 2>&1 | grep -i "ModuleNotFoundError"
```

**Resolution**:
```bash
# Regenerate lock file
cd backend
source venv/bin/activate
pip install pip-tools
pip-compile requirements.txt \
  --output-file=requirements.lock \
  --generate-hashes \
  --no-emit-index-url \
  --allow-unsafe

# Verify locally
pip install -r requirements.lock
pytest tests/ -v

# Commit and push
git add requirements.lock
git commit -m "fix: regenerate requirements.lock for new dependency"
git push
```

**Prevention**: Always regenerate `requirements.lock` after modifying `requirements.txt`. Add a CI check that `requirements.lock` is in sync (Phase 1 scope).

### Issue 10: CI `npm ci` Failure

**Severity**: P1

**Symptoms**: `test-frontend` or `lint-frontend` job fails on `npm ci` step with `npm ERR! ... package-lock.json`.

**Root Cause**: `package-lock.json` is out of sync with `package.json` — someone ran `npm install` (which updates lock file) but didn't commit it, or there's a merge conflict in the lock file.

**Diagnosis**:
```bash
# Check CI log for specific npm error

# Locally verify:
cd frontend
rm -rf node_modules
npm ci
echo $?
# If non-zero: lock file is out of sync
```

**Resolution**:
```bash
# Regenerate lock file
cd frontend
rm -rf node_modules package-lock.json
npm install
# This creates a fresh package-lock.json in sync with package.json

# Verify
npm ci  # Should succeed now

# Commit
git add package-lock.json
git commit -m "fix: regenerate package-lock.json"
git push
```

**Prevention**: Use `npm ci` locally (not `npm install`) unless intentionally updating deps. Commit `package-lock.json` changes in the same PR as `package.json` changes.

### Issue 11: CI Pipeline Too Slow (> 10 min)

**Severity**: P2

**Symptoms**: Pipeline total time exceeds 10 min (NFR-01). Jobs are running sequentially instead of in parallel, or individual jobs are slow.

**Root Cause**: Gitea runner may not support parallel job execution (open question #1). Cache misses (cold pip/npm cache). Slow runner hardware.

**Diagnosis**:
```bash
# Check if jobs ran in parallel or sequentially
# Gitea → Actions → select run → observe start/end times of parallel jobs
# If all 5 parallel jobs have sequential start times → runner is single-threaded

# Check per-job duration
# Gitea → Actions → select run → each job shows individual timing
```

**Resolution**:
```bash
# If sequential-only runner: combine lint jobs to reduce overhead
# Merge lint-backend + lint-frontend into a single "lint" job
# Estimated savings: ~90s (one fewer job startup)

# If cache misses causing slowness:
# Add caching steps to CI (pip cache, npm cache, Docker layer cache)
# See Perf Spec §12: Optimization Opportunities

# If runner hardware is slow:
# Check runner specs: ssh <runner-host> && nproc && free -h && df -h
# Upgrade runner or add a second runner
```

**Prevention**: Monitor pipeline duration in `baseline.json`. Set regression threshold: investigate if > 15% increase.

### Issue 12: CI Lint Failure

**Severity**: P1

**Symptoms**: `lint-backend` (ruff/mypy) or `lint-frontend` (eslint/tsc) job fails. PR is blocked.

**Root Cause**: New code introduced lint violations or type errors.

**Diagnosis**:
```bash
# Read CI logs — they show exact file:line and rule violation

# Reproduce locally:
# Backend
cd backend
pip install ruff mypy
ruff check .
mypy app/ --ignore-missing-imports

# Frontend
cd frontend
npm ci
npx eslint . --ext ts,tsx --report-unused-disable-directives --max-warnings 0
npx tsc --noEmit
```

**Resolution**:
```bash
# Fix the violations locally, then push

# Backend auto-fix (ruff can fix some issues):
cd backend
ruff check . --fix

# Frontend auto-fix (eslint can fix some issues):
cd frontend
npx eslint . --ext ts,tsx --fix

# For mypy/tsc type errors: fix the type annotations in code
# Commit and push fixes
```

**Prevention**: Run linters locally before pushing. Add pre-commit hooks: `ruff check .` and `npx eslint .`.

### Issue 13: Docker Memory Exhaustion

**Severity**: P1

**Symptoms**: Containers crash-looping with exit code 137. Host machine slow or unresponsive. `docker stats` shows > 8 GB total usage.

**Root Cause**: Full stack (10 services) running on < 8 GB Docker memory. Or Neo4j heap (2 GB) + ClamAV signatures (1.5 GB) consuming too much.

**Diagnosis**:
```bash
# Check per-container memory
docker stats --no-stream --format "table {{.Name}}\t{{.MemUsage}}\t{{.MemPerc}}" | sort -t'/' -k1 -rn

# Check Docker total allocation
docker info --format '{{.MemTotal}}'
```

**Resolution**:
```bash
# Immediate: stop non-essential services
docker stop legal-ai-clamav-dev  # saves ~1.5 GB
docker stop legal-ai-neo4j-dev   # saves ~2 GB (if not needed)

# If running full stack: switch to dev-only compose
docker compose down
docker compose -f docker-compose.dev.yml up -d postgres redis qdrant minio
# Start only services you actually need

# Long-term: increase Docker Desktop memory
# Docker Desktop → Settings → Resources → Memory → set to 8 GB+
```

**Prevention**: Use `docker-compose.dev.yml` (6 services, ~4 GB) instead of full stack. Only run full stack when testing end-to-end.

### Issue 14: Baseline Metrics Incomplete

**Severity**: P2

**Symptoms**: `baseline.json` artifact has `0` values for some metrics after a `main` merge.

**Root Cause**: Artifact download step failed, or metric parsing regex didn't match the test output format.

**Diagnosis**:
```bash
# Download baseline.json from CI artifacts
cat baseline.json | python3 -m json.tool

# Check which fields are 0
python3 -c "
import json
b = json.load(open('baseline.json'))
for k, v in b.items():
    if v == 0 or v == '0':
        print(f'⚠️  {k} = {v} (possibly broken)')
    else:
        print(f'✅ {k} = {v}')
"
```

**Resolution**:
```bash
# Check that test output files were generated in upstream jobs
# Gitea → Actions → select run → test-backend → verify pytest-output.txt in artifacts
# Gitea → Actions → select run → test-frontend → verify vitest-output.txt in artifacts

# If outputs missing: re-run the pipeline
git commit --allow-empty -m "ci: re-collect baseline metrics"
git push origin main

# If parsing regex is wrong: update scripts/record-baseline.sh
# Common issue: vitest or pytest output format changed
```

**Prevention**: After any test framework upgrade, verify `scripts/record-baseline.sh` parsing still works. Add a self-test: `bash -n scripts/record-baseline.sh`.

**Escalation rules:**
- **P0 (dev environment fully down, CI broken on `main`)**: Start at L1. If unresolved after 15 min, escalate to L2. Notify L3 immediately in parallel.
- **P1 (single service unhealthy, CI job flaky)**: Start at L1. Escalate to L2 after 15 min if no progress. L3 notified after 1 hour.
- **P2 (ClamAV slow, CI > 10 min, baseline incomplete)**: L1 owns resolution. Escalate to L2 only if blocked. No after-hours pages for P2.

**Phase 0-specific escalation**:
- If CI runner goes offline and no one has SSH access → escalate to L2 immediately (runner access is a bottleneck).
- If `main` branch CI is red → P0. All hands on deck. No PRs merge until green.
- If Docker Compose changes are needed (outside Phase 0 scope per Tech Spec NG1) → escalate to L2 for scope decision.

## 8. Backup & Recovery

| Data | Backup Method | Frequency | Retention | Recovery Time |
|------|--------------|-----------|-----------|---------------|
| PostgreSQL dev data | Docker volume (persisted automatically) | Continuous (volume mount) | Until `docker volume rm` | Instant (volume exists) |
| Neo4j dev data | Docker volume | Continuous | Until `docker volume rm` | Instant |
| Redis dev data | Docker volume (AOF enabled) | Continuous | Until `docker volume rm` | Instant |
| Qdrant dev data | Docker volume | Continuous | Until `docker volume rm` | Instant |
| MinIO dev data | Docker volume | Continuous | Until `docker volume rm` | Instant |
| CI pipeline config | Git (`.gitea/workflows/ci.yml`) | Every commit | Full git history | `git checkout <commit>` |
| `baseline.json` | CI artifact (Gitea) | Every `main` merge | 90 days | Download from Gitea Actions artifacts |

### Recovery Procedure — Nuclear Reset (Wipe All Dev Data)

```bash
# WARNING: Destroys all local dev data. Use when volumes are corrupted.
docker compose -f docker-compose.dev.yml down -v
docker compose down -v  # If full stack was used

# Verify volumes removed
docker volume ls | grep legal-ai
# Expected: no results

# Recreate
docker compose -f docker-compose.dev.yml up -d

# Re-run database initialization
# PostgreSQL: Alembic migrations
cd backend && source venv/bin/activate
alembic upgrade head

# Neo4j: init script (auto-loaded from volume mount, or manual)
docker exec legal-ai-neo4j-dev cypher-shell -u neo4j -p "${NEO4J_PASSWORD:-password}" < scripts/init-neo4j.cypher
```

### Recovery Procedure — Restore CI Pipeline to Known-Good State

```bash
# Find last green commit on main
git log --oneline main | head -10

# Restore CI config from that commit
git checkout <last-good-sha> -- .gitea/workflows/ci.yml
git commit -m "rollback: restore CI pipeline to last known-good state"
git push origin main
```
