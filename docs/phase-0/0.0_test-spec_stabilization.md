## 1. Scope

Phase 0 is a meta-testing phase: we are testing the test infrastructure itself and establishing a green baseline. There is no application code change — only test fixes, CI pipeline configuration, and dependency management.

### In Scope

- **Broken test remediation**: WorkflowDesignerPage (~59% pass rate), AdminDashboardPage (~83% pass rate), all other failing backend/frontend tests
- **`_green` test variant audit**: 6 backend `_green` files (2,144 LoC total) — disposition: delete, merge, or document
- **CI pipeline validation**: All 7+ CI jobs (test-backend, test-frontend, lint-backend, lint-frontend, security-audit, build, baseline-record) execute correctly
- **Lock file integrity**: `requirements.lock` and `package-lock.json` produce reproducible installs
- **Baseline metrics recording**: `baseline.json` artifact contains valid, non-zero metrics
- **Skip policy enforcement**: Every `pytest.mark.skip` / vitest `.skip` has a documented `reason=`

### Out of Scope

- **Application logic testing** — no business logic changes in Phase 0; existing test cases validate existing behavior
- **Performance benchmarks** — see [Perf Spec](./0.0_perf-spec_baseline-metrics.md) *(to be created)*
- **Visual regression testing** — no UI changes in Phase 0
- **Integration tests against real databases** — Phase 0 tests use mocks/fixtures; real DB integration testing is Phase 1 scope
- **CVE remediation verification** — Phase 0 documents CVEs; remediation testing is Phase 1–2 scope
- **Coverage threshold enforcement** — Phase 0 records baselines only; minimum thresholds are Phase 1+ scope

## 2. How to Run

### Prerequisites

| Requirement | Setup Command |
|-------------|---------------|
| Python 3.11+ | `python --version` (verify ≥ 3.11) |
| Node.js 18+ | `node --version` (verify ≥ 18) |
| Python venv + deps | `cd backend && python -m venv venv && source venv/bin/activate && pip install -r requirements.lock` |
| Frontend deps | `cd frontend && npm ci` |
| Docker (for build validation only) | `docker --version && docker compose version` |
| Test env vars | `cp .env.test .env` (or ensure no live API keys are required) |

### Commands

```bash
# === Backend Tests ===
cd backend
source venv/bin/activate

# Run all backend tests
pytest -v

# Run with coverage (matches CI job)
pytest --cov=app --cov-report=xml:coverage.xml --cov-report=html:htmlcov/ -v

# Run specific test layers
pytest tests/api/ -v                        # API tests only
pytest tests/services/ -v                   # Service tests only
pytest tests/ai/ -v                         # AI module tests only
pytest tests/integration/ -v                # Integration tests only (require mocks in Phase 0)

# Run by priority marker (after markers are added)
pytest -m "priority_p0" -v                  # P0 only (critical path)
pytest -m "priority_p0 or priority_p1" -v   # P0 + P1

# Verify no skips without reason
grep -rn "pytest.mark.skip" tests/ | grep -v "reason="

# === Frontend Tests ===
cd frontend

# Run all frontend tests
npx vitest run

# Run with coverage (matches CI job)
npx vitest run --coverage

# Run specific broken suites
npx vitest run src/pages/workflows/__tests__/WorkflowDesignerPage.test.tsx
npx vitest run src/pages/admin/__tests__/AdminDashboardPage.test.tsx

# === CI Pipeline Validation (local simulation) ===
# Lint - backend
cd backend && ruff check . && mypy app/ --ignore-missing-imports

# Lint - frontend
cd frontend && npx eslint . && npx tsc --noEmit

# Security audit
cd backend && pip-audit -r requirements.lock --format json
cd frontend && npm audit --json

# Build validation
docker compose build

# Baseline metrics (after all above pass)
bash scripts/record-baseline.sh
```

## 3. CI Pipeline Integration

| Test Suite | Trigger | Timeout | Required to Merge? | CI Job Name |
|------------|---------|---------|-------------------|-------------|
| Backend unit + service tests (pytest + coverage) | Every PR, every push to main/develop | 5 min | Yes | `test-backend` |
| Frontend unit + component tests (vitest + coverage) | Every PR, every push to main/develop | 5 min | Yes | `test-frontend` |
| Backend linting (ruff + mypy) | Every PR, every push to main/develop | 3 min | Yes | `lint-backend` |
| Frontend linting (eslint + tsc) | Every PR, every push to main/develop | 3 min | Yes | `lint-frontend` |
| Security audit (pip-audit + npm audit) | Every PR, every push to main/develop | 3 min | No (informational in Phase 0) | `security-audit` |
| Docker build validation | Every PR (after parallel jobs pass) | 5 min | Yes | `build` |
| Baseline metrics recording | On merge to `main` only (after build) | 2 min | No (advisory) | `baseline-record` |

**Total critical path** (parallel execution): `max(5, 5, 3, 3, 3) + 5 + 2 = 12 min` (within NFR-02 12 min feedback latency).
**If sequential-only runner**: `5+5+3+3+3+5+2 = 26 min` — exceeds NFR-01; see Tech Spec §9 for mitigation.

## 10. Flaky Test Policy

| Aspect | Policy |
|--------|--------|
| Definition | A test that fails non-deterministically on the same code. Two unexplained failures in 7 days on the same commit = flaky. |
| Discovery during Phase 0 | Any flaky test found while fixing broken suites must be classified: (1) fix immediately if root cause is clear (mock timing, async race), or (2) skip with `reason="flaky — PROJ-NNN"` and file a tracking ticket |
| Quarantine process | Move to `tests/quarantine/` directory (backend) or add `.skip("flaky — PROJ-NNN")` (frontend). Excluded from merge-blocking CI. Tracking issue filed with: test name, failure frequency, suspected root cause, reproduction steps |
| Retry policy | CI retries failed tests once (`pytest --lf` rerun on failure). If the retry passes, the test is flagged as potentially flaky and a warning is logged. Vitest `--retry=1` for frontend. If a test passes on retry but failed initially, it is added to the flaky watch list |
| Fix ownership | The team that owns the feature area owns the flaky test. Backend Eng owns `backend/tests/`, Frontend Eng owns `frontend/src/`. Must be fixed or permanently removed within 2 weeks of quarantine |
| Monitoring | Weekly manual audit of quarantine directory: `find . -path "*/quarantine/*" -name "*.py" \| wc -l` and `grep -r ".skip.*flaky" frontend/src/ \| wc -l`. Target: 0 quarantined tests by end of Phase 0 |
| Phase 0 specifics | The 6 `_green` test variants are NOT flaky — they are stale duplicates. They follow the audit process (UT-GR-01 through UT-GR-07), not the flaky policy. Tests in WorkflowDesigner (~59%) and AdminDashboard (~83%) that fail deterministically are NOT flaky — they are broken. Only non-deterministic failures during the fix process trigger this policy |

## 12. Known Limitations

- **No integration tests against real databases in Phase 0**: All DB-dependent tests use mocks or are skipped. This means Phase 0 coverage baselines may undercount actual integration coverage. Real integration tests (with `docker-compose.test.yml`) are Phase 1 scope.
- **CI pipeline validation is partially manual**: Negative tests (IT-CI-02, IT-CI-04, etc.) require pushing deliberate failures to test branches — these are one-time manual validations during PR review, not automated recurring tests.
- **`_green` variant audit requires human judgment**: The disposition decision (delete vs. merge vs. rename) cannot be automated — it requires comparing test content and determining which file is canonical. UT-GR-01 through UT-GR-06 define the verification criteria, but the decision itself is manual.
- **Baseline metrics are point-in-time snapshots**: `baseline.json` is a CI artifact, not a time-series database. Trend analysis requires manually comparing artifacts across runs. Automated trend tracking is Phase 2.5 scope (monitoring stack).
- **Flaky test detection is manual in Phase 0**: No automated flaky test detection tool (e.g., `pytest-flaky-detector`, Datadog CI Visibility). Detection relies on developer observation and CI retry behavior. Automated detection tooling is Phase 2.5 scope.
- **Sequential CI runner would exceed NFR-01**: If the Gitea runner does not support parallel job execution, total CI time could reach ~26 min (see §3). This is an open question (Tech Spec §14 #1) to be resolved on Day 1 of Phase 0.
- **Backend model classes location unknown**: Tech Spec §3 notes that `backend/app/models/` does not exist on disk. Test imports that reference `app.models.*` may fail at collection time. This must be investigated during PR 0.1.4 and is a known risk for UT-BE-03.
