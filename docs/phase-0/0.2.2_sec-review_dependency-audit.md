## 1. Change Summary

Phase 0 establishes the foundational infrastructure for the Legal AI Platform: a Docker Compose-based multi-database development environment (PostgreSQL 15, Neo4j 5, Qdrant, Redis 7, MinIO, ClamAV), a FastAPI backend (~75K LoC), a React 18 frontend, and a CI pipeline (Gitea CI) expanding from 3 to 7+ jobs. **No new application features are introduced** — Phase 0 stabilizes the existing codebase, pins dependencies, adds linting and security scanning to CI, and records baseline metrics.

**Security-relevant scope**:
- CI pipeline expansion (secret management, artifact integrity, scanning visibility)
- Docker Compose security posture (6 database services + 3 application services)
- Dependency vulnerabilities: `python-jose` CVE-2024-33663, `passlib` archived, Vite CVE-2024-23331
- Hardcoded credentials in `config.py` and `.env.example`
- Pre-existing Cypher injection vectors in `backend/app/core/graph.py`
- Supply chain risks (unpinned Docker images, gitignored `package-lock.json`)
- PII logged in application code (`oauth.py`, `middleware.py`)

**What Phase 0 improves**: `pip-audit` and `npm audit` added to CI → vulnerability visibility increases. `requirements.lock` with hashes → supply chain integrity improves.

**What Phase 0 does NOT fix**: All application-level vulnerabilities are pre-existing and documented for Phase 1 remediation.

### Threat Scenarios

#### T1: JWT Forgery via python-jose CVE-2024-33663

| Attribute | Value |
|-----------|-------|
| **Attack vector** | Attacker exploits ECDSA key confusion vulnerability in `python-jose==3.3.0` to forge JWT tokens with arbitrary claims (tenant_id, user_id, roles), bypassing authentication and tenant isolation entirely. Public PoC exists. |
| **Likelihood** | 4 (Known CVE with public PoC; low-skill attack if ECDSA keys are ever configured) |
| **Impact** | 5 (Full auth bypass → cross-tenant data access → regulatory violation → complete trust loss) |
| **Risk Score** | **20 (Critical)** |

**Mitigations**:
- Current codebase uses HS256 (`JWT_ALGORITHM = "HS256"` in `config.py:19`), which is not directly affected by the ECDSA key confusion attack vector
- However, the library is unmaintained since May 2022 and may harbor undisclosed vulnerabilities

**Residual risk**: Medium — HS256 mitigates the specific CVE-2024-33663 vector, but using an abandoned JWT library in a legal platform handling PII is unacceptable from a compliance perspective (SOC 2 CC7.1 — vulnerability management). **Must migrate to `PyJWT>=2.8.0` or `joserfc` before Phase 1.** Tracked as LEGAL-SEC-001.

#### T2: Authentication Bypass via Default JWT Secrets

| Attribute | Value |
|-----------|-------|
| **Attack vector** | `config.py:17-18` sets `SECRET_KEY = os.getenv("SECRET_KEY", "your-secret-key-here")` and `JWT_SECRET_KEY = os.getenv("JWT_SECRET_KEY", "your-jwt-secret-key-here")`. If `.env` is missing or incomplete, production runs with predictable JWT signing keys. Attacker signs arbitrary JWTs with the known default, impersonating any user in any tenant. |
| **Likelihood** | 3 (Requires deployment misconfiguration — common in dev→prod copy errors, Docker environment variable omissions) |
| **Impact** | 5 (Complete authentication bypass, arbitrary tenant impersonation, full data exfiltration) |
| **Risk Score** | **15 (High)** |

**Mitigations**:
- None currently implemented
- The `os.getenv()` pattern bypasses Pydantic Settings validation, so there is no startup check for placeholder values
- `.env.example` contains the same placeholder values (`"your-secret-key-here-change-in-production"`), encouraging copy-paste

**Residual risk**: High — must add startup validation that refuses to start with default/placeholder secrets when `ENVIRONMENT != "development"`. Must also migrate all `os.getenv()` calls in `config.py` to native Pydantic Settings field declarations.

#### T3: Cypher Injection via Neo4j Graph Operations

| Attribute | Value |
|-----------|-------|
| **Attack vector** | `backend/app/core/graph.py` constructs Cypher queries by interpolating `label`, `relationship_type`, `direction`, and property `key` names directly into query strings via f-strings. If any of these values originate from user input (API parameters, contract metadata), an attacker can inject arbitrary Cypher — reading/modifying other tenants' graph data or executing destructive `DETACH DELETE` operations. |
| **Likelihood** | 3 (Requires user-controlled data reaching graph functions — depends on service layer validation that does not currently exist) |
| **Impact** | 4 (Cross-tenant data read/write in graph DB; partial data breach; data destruction possible) |
| **Risk Score** | **12 (Medium)** |

**Affected functions** (all in `backend/app/core/graph.py`):

| Function | Line | Injection Point | Pattern |
|----------|------|-----------------|---------|
| `create_node()` | 87 | `label` | `f"CREATE (n:{label} $properties)"` |
| `create_relationship()` | 113 | `relationship_type` | `f"CREATE (a)-[r:{relationship_type} $properties]->(b)"` |
| `find_node()` | 147 | `label`, property `key` | `f":{label}"`, `f"n.{key} = ${key}"` |
| `find_relationships()` | 186 | `relationship_type` | `f":{relationship_type}"` |
| `update_node()` | 222 | property `key` | `f"n.{key} = ${key}"` |
| `get_node_neighbors()` | 297–301 | `relationship_type`, `direction` | Pattern string built from both |
| `find_shortest_path()` | 335 | `max_length` | `""" ... [*..%d] ... """ % max_length` (Python `%` formatting) |
| `find_related_contracts()` | 389 | `relationship_types` list | `"|".join(relationship_types)` interpolated into `f"[:{rel_pattern}]"` |

**Mitigations**:
- Neo4j labels and relationship types cannot be parameterized in Cypher (language limitation)
- Must implement allowlist validation for all label/relationship_type/direction values before interpolation
- Property keys must be validated against `^[a-zA-Z_][a-zA-Z0-9_]*$` regex
- `max_length` in `find_shortest_path()` must be validated as a positive integer before format-string interpolation

**Residual risk**: Medium — requires service-layer allowlist enforcement. No such validation currently exists. Exploitability depends on whether API layers pass user input directly to graph functions.

#### T4: Unauthenticated Redis Access

| Attribute | Value |
|-----------|-------|
| **Attack vector** | Redis runs without authentication — `docker-compose.yml` specifies `redis-server --appendonly yes` with no `--requirepass`. Any process on the Docker network can read/write all Redis keys, including session data and rate limit counters. Redis port is also mapped to the host via `${REDIS_PORT}:6379`. |
| **Likelihood** | 2 (Requires network access to Docker bridge or host; more relevant in production or shared environments) |
| **Impact** | 3 (Session hijacking, rate limit bypass, cache poisoning) |
| **Risk Score** | **6 (Medium)** |

**Mitigations**:
- Docker network provides some isolation in local dev
- Port mapping to host increases exposure surface

**Residual risk**: Low in isolated dev, **High in any shared or production environment**. Must add `--requirepass ${REDIS_PASSWORD}` before any non-local deployment.

#### T5: Supply Chain Compromise via Unpinned Docker Images

| Attribute | Value |
|-----------|-------|
| **Attack vector** | `docker-compose.yml` pulls `qdrant/qdrant:latest` (line 67) and `minio/minio:latest` (line 79). The `latest` tag is mutable — a compromised registry, maintainer account, or upstream build could push a malicious image that is automatically pulled on next `docker-compose pull`. |
| **Likelihood** | 2 (Registry compromise is rare but has precedent — e.g., Docker Hub credential stuffing, Codecov supply chain attack) |
| **Impact** | 5 (Backdoored database service → full data exfiltration, cryptomining, lateral movement) |
| **Risk Score** | **10 (Medium)** |

**Mitigations**:
- Pin all images to specific version tags with SHA256 digests (e.g., `qdrant/qdrant:v1.8.4@sha256:...`)
- Dep Review (§10) already mandates this as a condition for Phase 0 approval

**Residual risk**: Low after image pinning and digest verification.

#### T6: Non-Reproducible Frontend Builds via Gitignored Lock File

| Attribute | Value |
|-----------|-------|
| **Attack vector** | `.gitignore:125` contains `package-lock.json` (there is a negation at line 547 `!package-lock.json`, but the earlier ignore may take precedence depending on git version). Without a committed lock file, `npm install` resolves transitive dependencies at install time, potentially pulling different (compromised or vulnerable) versions across builds and environments. `npm ci` will fail entirely without `package-lock.json`. |
| **Likelihood** | 4 (This is the current state — every frontend build is non-reproducible today) |
| **Impact** | 2 (Inconsistent builds; potential for pulling a compromised transitive dependency) |
| **Risk Score** | **8 (Medium)** |

**Mitigations**:
- Tech Spec §3 Component 8 plans to commit `package-lock.json` in PR 0.2.4
- Must resolve `.gitignore` conflict (remove `package-lock.json` from ignore list or verify negation works)

**Residual risk**: Low after lock file is committed and `.gitignore` is corrected.

### Critical Anti-Pattern: `os.getenv()` Bypass

`backend/app/core/config.py` uses `os.getenv()` to read environment variables **before** Pydantic Settings processes them (lines 17–18, 53–57, 83–84, 93). This bypasses:
- Pydantic's field validation and type coercion
- Pydantic Settings' `.env` file loading
- Any future secret validation logic

This means the `class Config: env_file = ".env"` declaration (line 130) is partially ineffective — several critical secrets are resolved before Pydantic ever reads the `.env` file. **Must refactor all `os.getenv()` calls to native Pydantic Settings field declarations.**
