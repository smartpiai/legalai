## 1. Motivation

Phase 0 establishes the infrastructure and development environment for the Legal AI Platform. Every service, library, and runtime chosen at this stage becomes a load-bearing dependency for all subsequent phases. This review evaluates the full foundational stack — container runtime, databases, backend framework, frontend framework, and supporting tooling — to ensure license compatibility, supply chain health, security posture, and long-term maintenance viability before any feature code is written.

**Triggering event**: Project kickoff — all foundational dependencies must be audited before Phase 1 development begins.
**Tracking issue**: LEGAL-0002 — Infrastructure dependency baseline

## 5. Breaking Changes

> Phase 0 is a greenfield setup. There is no existing code to break. This section documents known migration pitfalls for versions chosen.

| Package | Known Pitfall | Impact | Mitigation |
|---------|--------------|--------|------------|
| SQLAlchemy 2.0 | SQLAlchemy 1.x → 2.0 migration patterns differ significantly | N/A — greenfield | Use 2.0-style exclusively (`select()`, `Session.execute()`). Do not follow 1.x tutorials. |
| Pydantic 2.x | `from_orm()` removed; use `model_validate()` | N/A — greenfield | Follow Pydantic v2 docs from day one. |
| React 18 | Strict Mode double-renders in dev; `createRoot` required | N/A — greenfield | Use `createRoot` API. Design effects to be idempotent. |
| Redis 7 → 7.4 | License change to RSAL/SSPL at 7.4 | Future upgrade risk | Pin `redis:7.2-alpine`. Evaluate Valkey as BSD-licensed fork before upgrading past 7.2. |
| Celery 5.3 → 5.4 | `task_always_eager` behavior change | Future upgrade risk | Pin `==5.3.4`. Review changelog before bumping. |
| Neo4j 5 Community | No clustering, limited APOC in Community | Feature limitation | Sufficient for Phase 0–1. Enterprise evaluation needed by Phase 3 if clustering required. |

### 6.2 Runtime Impact

| Transitive Dep | Cold Start Impact | Memory Overhead | Notes |
|----------------|-------------------|-----------------|-------|
| `cryptography` (OpenSSL) | +100ms | +10 MB | Loads native C extensions at import |
| `botocore` | +300ms | +40 MB | Large JSON models loaded at import; consider lazy loading |

### 6.3 Optimization Recommendations

1. **`botocore` / `boto3`**: Only needed for MinIO S3 compatibility. Consider using `minio` Python client exclusively and dropping `boto3` to save ~80 MB and ~300ms cold start.
2. **`torch` 2.1**: ~800 MB. Not needed in Phase 0. Defer to Phase 2 (AI Integration). Use CPU-only build (`torch+cpu`) when added.
3. Docker multi-stage builds to keep runtime images lean.

## 8. Alternatives Considered

> New dependency adoption — all packages reviewed.

### 8.1 Databases

| Alternative | Why Not Chosen |
|-------------|---------------|
| **MySQL / MariaDB** instead of PostgreSQL | PostgreSQL superior for JSONB, full-text search, RLS (multi-tenancy), and pgvector support |
| **ArangoDB** instead of Neo4j | Neo4j has stronger Cypher query language, larger ecosystem, better graph analytics (GDS library) |
| **Milvus** instead of Qdrant | Qdrant simpler to operate, lower resource footprint, better Rust-based performance for our scale |
| **Memcached** instead of Redis | Redis provides pub/sub, persistence, Celery broker support — Memcached is cache-only |
| **AWS S3** instead of MinIO | MinIO provides S3-compatible API locally for development; production can swap to S3 transparently |

### 8.2 Backend Framework

| Alternative | Why Not Chosen |
|-------------|---------------|
| **Django + DRF** | Heavier ORM coupling, synchronous by default, slower for async workloads |
| **Litestar** | Newer, smaller community, fewer battle-tested production deployments |
| **Flask** | No built-in async support, manual OpenAPI, less productive for API-first development |

### 8.3 Frontend Framework

| Alternative | Why Not Chosen |
|-------------|---------------|
| **Next.js** | SSR not required; adds complexity. Vite + React SPA is simpler for our dashboard-centric UX |
| **Vue 3** | Smaller talent pool for hiring; React ecosystem is larger for enterprise |
| **Angular** | Heavier framework, steeper learning curve, slower iteration for our team size |
| **Svelte** | Smaller ecosystem, fewer enterprise UI component libraries |

### 8.4 State Management

| Alternative | Why Not Chosen |
|-------------|---------------|
| **Redux Toolkit** | More boilerplate than Zustand for equivalent functionality |
| **Jotai** | Atomic model less intuitive for our team's experience |
| **MobX** | Decorator-heavy API, larger bundle |

### 8.5 ORM

| Alternative | Why Not Chosen |
|-------------|---------------|
| **Tortoise ORM** | Less mature async ORM, smaller community, fewer migration tools |
| **Prisma (Python)** | Early stage, not production-ready for Python |
| **Raw SQL (asyncpg only)** | Loses migration tooling, type safety, and query building |

## 9. Rollback Plan

Phase 0 is initial environment setup. Rollback means reverting `docker-compose.yml`, `requirements.txt`, and `package.json` to empty/previous state.

```bash
# Tear down all containers and volumes
docker compose down -v

# Revert dependency files
git revert <phase-0-setup-commit>

# Rebuild from clean state
docker compose up -d --build
```

For individual dependency rollback:

```bash
# Backend — revert a single package
pip install <package>==<previous_version>
pip freeze > requirements.txt

# Frontend — revert a single package
npm install <package>@<previous_version>

# Docker image — revert to previous tag
# Edit docker-compose.yml image tag, then:
docker compose pull <service>
docker compose up -d <service>
```
