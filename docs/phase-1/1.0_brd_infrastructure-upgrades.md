## 1. Executive Summary

1. **What is proposed**: Upgrade the platform's core data infrastructure — PostgreSQL 15 → 17, Redis 7 → Valkey/Redis 8, Qdrant 1.7 → 1.12+, Neo4j 5 to latest patch — and add the pgvector extension to PostgreSQL. This covers 16 PRs across 5 workstreams over Weeks 2–4.
2. **Why it matters**: PostgreSQL 15 is approaching end-of-life community support, Redis's SSPL relicense creates deployment liability, and Qdrant 1.7 lacks named/sparse vector support required for Phase 3's hybrid search and AI agent architecture. Inaction blocks the entire modernization critical path.
3. **The ask**: ~3 person-weeks of SRE/backend engineering effort, zero additional infrastructure budget (same service footprint, upgraded images), and a 2–3 week timeline with Phase 0 as prerequisite.

## 2. Business Problem

### Current State
- **PostgreSQL 15** is running in production. PG 15 reaches end of community support in November 2027, but PG 17 offers `pg_stat_statements` improvements critical for query profiling and the pgvector extension needed for Phase 3.
- **Redis 7 (Alpine)** is deployed under the Redis SSPL license (relicensed in March 2024). For self-hosted deployments, SSPL introduces legal ambiguity around "making the functionality available as a service" — a risk for enterprise customers.
- **Qdrant 1.7** (`qdrant-client==1.7.0`) is 5+ minor versions behind. It lacks named vectors (required for multi-vector-per-document) and sparse vector support (required for BM25 hybrid search in Phase 3.6).
- **Neo4j 5** is on an unspecified patch level (`neo4j:5` Docker tag). APOC and GDS plugin compatibility has not been verified against latest patches.
- **No pgvector extension** exists. The platform currently relies solely on Qdrant for vector similarity search, creating a single point of failure for the embedding pipeline and preventing co-located vector+relational queries.

### Desired State
- PostgreSQL 17 with `pg_stat_statements` enabled and pgvector extension installed, including a `document_embeddings` table with `vector(1536)` column and HNSW/IVFFlat indexing.
- Valkey (BSD-licensed Redis fork) or Redis 8 as the caching/session/pubsub backend — decision captured in ADR-001.
- Qdrant 1.12+ with named vector and sparse vector collection schemas, enabling hybrid search in Phase 3.
- Neo4j 5 latest patch with verified APOC and GDS plugin compatibility.
- All 8 existing Alembic migrations verified clean against PG 17.
- CI updated to test against the new infrastructure versions.

## 4. Proposed Solution

Upgrade all data infrastructure components in a controlled sequence, with each change independently mergeable and tested. This aligns with the Modernization Roadmap Phase 1 (Weeks 2–4).

Reference: [MODERNIZATION_ROADMAP.md](../../MODERNIZATION_ROADMAP.md) — Phase 1.

### In Scope
- **1.1**: PostgreSQL 15 → 17 (docker-compose images, Alembic migration verification, `pg_stat_statements`, CI update) — 4 PRs
- **1.2**: pgvector extension (Alembic migration, Python package, SQLAlchemy model, repository layer, integration tests) — 5 PRs
- **1.3**: Redis → Valkey or Redis 8 (ADR decision, docker-compose swap, integration tests) — 3 PRs
- **1.4**: Qdrant 1.7 → 1.12+ (client bump, docker-compose image, named vectors refactor, sparse vector support, integration tests) — 5 PRs
- **1.5**: Neo4j 5 latest patch (docker-compose update, APOC/GDS plugin verification) — 2 PRs

### Out of Scope
- **Application code changes** — Phase 1 upgrades infrastructure only; backend/frontend code stays unchanged except for new pgvector model/repository and Qdrant `vector_store.py` refactor.
- **Data migration of production data** — Development/staging only; production migration is a deployment decision post-Phase 1 approval.
- **Embedding re-indexing** — Backfilling Qdrant with new embedding formats is Phase 3 (3.5.5) scope.
- **Hybrid search implementation** — Phase 3 (3.6) scope; Phase 1 only adds sparse vector schema support.

## 6. Alternatives Considered

| Alternative | Strategic Rationale | Why Not Chosen |
|-------------|--------------------| ---------------|
| Do nothing — keep PG 15, Redis 7, Qdrant 1.7 | Avoids any migration risk | Unacceptable: blocks entire Phase 3 critical path, leaves SSPL license risk unaddressed, PG 15 EOL approaching |
| Upgrade only PG and Qdrant, keep Redis | Reduces scope by 3 PRs | Insufficient: SSPL license risk remains; enterprise prospects already flagging it |
| Replace Qdrant with pgvector entirely | Eliminates a service dependency | Premature: pgvector lacks Qdrant's named vector, sparse vector, and filtering capabilities needed for Phase 3; revisit after pgvector matures |
| Skip pgvector, rely solely on Qdrant | Simpler — one vector store | Risky: single point of failure; pgvector enables co-located SQL+vector queries that Qdrant cannot (e.g., join embeddings with contract metadata in a single query) |
| Defer to Q3 2026 | Focus on feature delivery first | Unacceptable cascade: every week of Phase 1 delay pushes Phase 3 AI/ML stack by the same amount |
