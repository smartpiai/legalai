# Migration Guide

> **Title**: Phase 1 — Infrastructure Upgrades (PostgreSQL 15→17 + pgvector, Redis 7→Valkey 8, Qdrant 1.7→1.12, Neo4j 5.x pin)
> **Phase**: 1 | **PR(s)**: 1.1.1 – 1.5.3
> **Author**: SRE
> **Date**: 2026-04-07
> **Status**: Draft (NOT YET TESTED IN STAGING — production approval blocked on §5)

---

## 1. Overview

This migration moves the Legal AI Platform data tier from its Phase 0 state to the Phase 1 target defined in the [Tech Spec](./1.0_tech-spec_infrastructure-upgrades.md). It is executed as four sub-migrations (PG, Cache, Qdrant, Neo4j), each independently revertible by image tag. The PG sub-migration is the only one with a Point of No Return.

| Aspect | Before | After |
|--------|--------|-------|
| PostgreSQL | `postgres:15` | `pgvector/pgvector:pg17` |
| `pgvector` extension | not installed | installed; `document_embeddings` table + HNSW index |
| Alembic head | `008_add_refresh_tokens_table` | `009_add_document_embeddings` |
| Cache / Celery broker | `redis:7-alpine` | `valkey/valkey:8-alpine` |
| `qdrant-client` (Python) | `1.7.x` | `1.12.x` |
| Qdrant server | `qdrant/qdrant:v1.7.x` | `qdrant/qdrant:v1.12.4` |
| Qdrant collection schema | single unnamed dense vector | named dense + sparse vectors (back-compat shim) |
| Neo4j | `neo4j:5` (floating) | `neo4j:5.20.0` (pinned, APOC + GDS smoke-tested) |

## 3. Migration Steps

### Parallelism Guidance

Sub-migrations **A (PG)**, **C (Qdrant)**, and **D (Neo4j)** are independent and *could* run concurrently, but at 3am with one executor we run them strictly sequentially in the order **A → B → C → D**. Within sub-migration A, all steps are sequential. Cache (B) is gated on A completing successfully because `cache.py` integration tests run against the live PG.

**Total wall-clock estimate (from staging dry-run): ~40 min.** Maintenance window: 60 min.

### Sub-Migration A — PostgreSQL 15 → 17 + pgvector

#### Step A1: Drain traffic and stop application containers

| Duration | Executor |
|----------|----------|
| ~2 min | SRE |

```bash
cd /opt/legalai
docker compose -f docker-compose.prod.yml stop backend celery-worker celery-beat
```

**Expected output**: Three `Stopped` lines, one per service.

**Verification**:
```bash
docker compose -f docker-compose.prod.yml ps --status running | grep -E 'backend|celery' || echo "OK: app drained"
```
Should print `OK: app drained`.

#### Step A2: Final backup of PostgreSQL 15

| Duration | Executor |
|----------|----------|
| ~5 min (varies with DB size) | SRE |

```bash
TS=$(date -u +%Y%m%dT%H%M%SZ)
docker compose -f docker-compose.prod.yml exec -T postgres \
  pg_dumpall -U postgres --clean --if-exists \
  > /backups/pg/pg15-final-${TS}.sql
ls -lh /backups/pg/pg15-final-${TS}.sql
```

**Expected output**: A `.sql` file >0 bytes whose size is within 10% of yesterday's nightly dump.

**Verification**:
```bash
head -5 /backups/pg/pg15-final-${TS}.sql | grep -q 'PostgreSQL database cluster dump' && echo OK
```

#### Step A3: Stop PostgreSQL 15 container

| Duration | Executor |
|----------|----------|
| ~30 s | SRE |

```bash
docker compose -f docker-compose.prod.yml stop postgres
docker compose -f docker-compose.prod.yml rm -f postgres
```

**Expected output**: `Stopped` then `Removed`.

**Verification**:
```bash
docker ps -a --filter name=legalai-postgres --format '{{.Names}} {{.Status}}'
```
Should be empty.

#### Step A4: Move PG15 data directory aside (preserves rollback path)

| Duration | Executor |
|----------|----------|
| ~10 s | SRE |

> **POINT OF NO RETURN — after Step A5 completes successfully, rollback requires restoring from the dump in Step A2; the original PG15 data directory will still exist on disk as `pgdata.pg15.bak` for 7 days as a safety net, after which it is purged by cron.**

```bash
mv /var/lib/legalai/pgdata /var/lib/legalai/pgdata.pg15.bak
mkdir -p /var/lib/legalai/pgdata
chown 999:999 /var/lib/legalai/pgdata
```

**Expected output**: No output.

**Verification**:
```bash
test -d /var/lib/legalai/pgdata.pg15.bak && test -z "$(ls -A /var/lib/legalai/pgdata)" && echo OK
```

#### Step A5: Start PostgreSQL 17 (with pgvector image) and restore dump

| Duration | Executor |
|----------|----------|
| ~10 min | SRE |

> **POINT OF NO RETURN.** After this step the cluster is PG17. Going back means stopping PG17, swapping the image tag to `postgres:15`, and restoring `pgdata.pg15.bak` (see §7).

```bash
# Image tag is already pinned to pgvector/pgvector:pg17 in docker-compose.prod.yml (PR 1.1.1).
docker compose -f docker-compose.prod.yml up -d postgres

# Wait for readiness
for i in $(seq 1 30); do
  docker compose -f docker-compose.prod.yml exec -T postgres pg_isready -U postgres && break
  sleep 2
done

# Restore the dump from Step A2
docker compose -f docker-compose.prod.yml exec -T postgres \
  psql -U postgres -v ON_ERROR_STOP=1 < /backups/pg/pg15-final-${TS}.sql
```

**Expected output**: `accepting connections`, then a stream of `CREATE`, `ALTER`, `COPY` lines, ending with no `ERROR:` lines.

**Verification**:
```bash
docker compose -f docker-compose.prod.yml exec -T postgres \
  psql -U postgres -tAc "SELECT version();" | grep -q 'PostgreSQL 17' && echo OK
docker compose -f docker-compose.prod.yml exec -T postgres \
  psql -U postgres -d legalai -tAc "SELECT count(*) FROM alembic_version;"
```
Second command should print `1`.

#### Step A6: Enable pgvector + pg_stat_statements

| Duration | Executor |
|----------|----------|
| ~5 s | SRE |

```bash
docker compose -f docker-compose.prod.yml exec -T postgres \
  psql -U postgres -d legalai -c "CREATE EXTENSION IF NOT EXISTS vector; CREATE EXTENSION IF NOT EXISTS pg_stat_statements;"
```

**Expected output**: `CREATE EXTENSION` (or `NOTICE: extension "vector" already exists, skipping`).

**Verification**:
```bash
docker compose -f docker-compose.prod.yml exec -T postgres \
  psql -U postgres -d legalai -tAc "SELECT extname FROM pg_extension WHERE extname IN ('vector','pg_stat_statements') ORDER BY extname;"
```
Should print:
```
pg_stat_statements
vector
```

#### Step A7: Run Alembic upgrade to `009_add_document_embeddings`

| Duration | Executor |
|----------|----------|
| ~30 s (NFR-04) | SRE |

```bash
docker compose -f docker-compose.prod.yml run --rm backend alembic upgrade head
```

**Expected output**: `INFO  [alembic.runtime.migration] Running upgrade 008_add_refresh_tokens_table -> 009_add_document_embeddings`.

**Verification**:
```bash
docker compose -f docker-compose.prod.yml run --rm backend alembic current | grep -q '009_add_document_embeddings' && echo OK

docker compose -f docker-compose.prod.yml exec -T postgres psql -U postgres -d legalai -tAc \
  "SELECT indexname FROM pg_indexes WHERE tablename='document_embeddings' ORDER BY indexname;"
```
Second command must list `idx_doc_emb_hnsw` and `idx_doc_emb_tenant`.

### Sub-Migration C — Qdrant 1.7 → 1.12

#### Step C1: Snapshot every Qdrant collection

| Duration | Executor |
|----------|----------|
| ~5 min (varies with collection size) | SRE |

```bash
for c in $(curl -sf http://qdrant:6333/collections | jq -r '.result.collections[].name'); do
  echo "Snapshotting $c"
  curl -sf -X POST "http://qdrant:6333/collections/${c}/snapshots" | jq -r '.result.name'
done
```

**Expected output**: One snapshot filename per collection.

**Verification**:
```bash
curl -sf http://qdrant:6333/collections | jq -r '.result.collections[].name' | while read c; do
  count=$(curl -sf "http://qdrant:6333/collections/${c}/snapshots" | jq '.result | length')
  echo "$c: $count snapshot(s)"
  test "$count" -gt 0 || { echo "FAIL: $c has no snapshot"; exit 1; }
done
```

#### Step C2: Roll Qdrant to v1.12.4

| Duration | Executor |
|----------|----------|
| ~1 min | SRE |

```bash
docker compose -f docker-compose.prod.yml up -d qdrant
for i in $(seq 1 30); do
  curl -sf http://qdrant:6333/readyz && break
  sleep 2
done
```

**Expected output**: HTTP 200 from `/readyz`.

**Verification**:
```bash
curl -sf http://qdrant:6333/ | jq -r '.version' | grep -q '^1\.12\.' && echo OK
curl -sf http://qdrant:6333/collections | jq '.result.collections | length'
```
Collection count must equal pre-migration count.

| Duration | Executor |

## 7. Rollback Procedure

### Decision Authority

| Role | Authority |
|------|-----------|
| **Can call rollback** | Migration executor (SRE on-call), Backend on-call |
| **Must be notified** | Engineering Manager (Eastern Time), Tech Lead (Eastern Time) |
| **Must approve re-attempt** | Tech Lead + SRE Lead jointly |

### Trigger Conditions

Roll back if **any** of:
- Any step fails verification and the failure is not resolvable in <5 min
- Live monitoring (§4) hits an Abort threshold
- pgvector p95 > 100 ms sustained for 5 min after traffic returns
- `/health` returns non-200 for >60 s after Step E
- Any new Sentry P1/P2 tagged `release:phase-1`
- Tenant-isolation regression discovered (any tenant sees another tenant's data)

### Rollback Steps

Choose the matching scenario.

#### Scenario R1 — Failure **before** Step A5 completes (no Point of No Return crossed)

Re-tag images to previous and restart. PG15 data dir is still in place at `/var/lib/legalai/pgdata.pg15.bak` only if Step A4 ran; otherwise it is still at `/var/lib/legalai/pgdata`.

```bash
# Restore image tags via the previous compose file kept at /opt/legalai/docker-compose.prod.yml.prev
cp /opt/legalai/docker-compose.prod.yml.prev /opt/legalai/docker-compose.prod.yml

# If Step A4 ran, restore the data dir
if [ -d /var/lib/legalai/pgdata.pg15.bak ] && [ -z "$(ls -A /var/lib/legalai/pgdata 2>/dev/null)" ]; then
  rmdir /var/lib/legalai/pgdata
  mv /var/lib/legalai/pgdata.pg15.bak /var/lib/legalai/pgdata
fi

docker compose -f docker-compose.prod.yml up -d postgres redis qdrant neo4j
docker compose -f docker-compose.prod.yml up -d backend celery-worker celery-beat
```

#### Scenario R2 — Failure **after** Step A5 (PG17 is live; Point of No Return crossed)

```bash
# 1. Stop everything
docker compose -f docker-compose.prod.yml stop backend celery-worker celery-beat postgres

# 2. Move the PG17 data dir aside
mv /var/lib/legalai/pgdata /var/lib/legalai/pgdata.pg17.failed

# 3. Restore PG15 data dir from the safety copy
mv /var/lib/legalai/pgdata.pg15.bak /var/lib/legalai/pgdata

# 4. Revert compose file (image tag → postgres:15)
cp /opt/legalai/docker-compose.prod.yml.prev /opt/legalai/docker-compose.prod.yml

# 5. Bring PG15 back up
docker compose -f docker-compose.prod.yml up -d postgres
for i in $(seq 1 30); do
  docker compose -f docker-compose.prod.yml exec -T postgres pg_isready -U postgres && break
  sleep 2
done

# 6. If pgdata.pg15.bak was already purged (>7 days) OR is corrupt, restore from dump instead:
#    docker compose ... exec -T postgres psql -U postgres -v ON_ERROR_STOP=1 < /backups/pg/pg15-final-${TS}.sql

# 7. Restart app
docker compose -f docker-compose.prod.yml up -d backend celery-worker celery-beat
```

docker compose -f docker-compose.prod.yml rm -f redis
