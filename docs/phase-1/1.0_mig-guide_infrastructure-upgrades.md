## 1. Overview

This phase-level Migration Guide consolidates per-component datastore migration procedures for Phase 1 infrastructure upgrades. It is the single source of truth the on-call executor follows during the maintenance window; component-level Mig Guides (1.1.1, 1.2.1, 1.3.2, 1.4.2, 1.5.2) defer to this document for ordering, snapshots, and rollback policy.

Upstream: [1.0 Tech Spec — Infrastructure Upgrades](./1.0_tech-spec_infrastructure-upgrades.md), [ADR-002: Valkey vs Redis 8](./1.3.1_adr_valkey-vs-redis8.md).

| Aspect | Before | After |
|--------|--------|-------|
| PostgreSQL | 15.x | 17.x |
| pgvector extension | Not installed | 0.7.x (HNSW default) |
| Cache / broker | `redis:7.2` | `valkey/valkey:8` |
| Qdrant | 1.7.x | 1.12.x+ (named + sparse vectors) |
| Neo4j | 5.x (stale patch) | 5.latest + matched APOC/GDS |
| Client libs | pinned | `qdrant-client ≥1.12`, `pgvector` (new), `neo4j` latest 5.x, `redis-py` unchanged |
| `document_embeddings` table | N/A | Created (empty; backfill is Phase 3) |

> **Assumption update (2026-04-09)**: ADR-002 Assumption #1 ("Valkey 8 reads Redis 7.2 RDB directly") holds **only** when the on-disk RDB version byte is ≤ v11. During PR 1.3.2 implementation we discovered the dev `redis_data` volume contained an AOF base file written at **RDB v12** (produced by a prior Redis ≥ 7.4 image), which Valkey 8 refuses to load — the container enters a crash loop on boot. A mandatory RDB preflight (Step 5.0 below) is now required before any Valkey image swap, and ADR-002 has been amended to record this constraint. See also R-X in §7.

### 1.1 Upgrade Order (Canonical)

Order is driven by blast radius, lowest first, so each component can be verified and rolled back independently before touching the next:

1. **Neo4j** patch — isolated from SQL stack.
2. **Qdrant 1.7 → 1.12+** — isolated vector store.
3. **PostgreSQL 15 → 17** — base engine upgrade; **point of no return**.
4. **pgvector extension** — requires PG 17.
5. **Valkey (Redis → Valkey 8)** — highest blast radius (cache + Celery broker + pub/sub).

Rationale and per-component breaking changes: Tech Spec §3.

### 1.2 PostgreSQL: `pg_upgrade` vs dump/restore Decision

| Criterion | `pg_upgrade --link` | `pg_dumpall` + restore |
|-----------|---------------------|------------------------|
| Duration on current prod DB (~38 GB) | ~6–8 min (dry run measured) | ~55–70 min (dry run measured) |
| Disk overhead | Near zero (hard links) | 2× DB size during cut-over |
| Rollback after run | **Not possible** — old cluster hard-links are destroyed; must restore from pre-upgrade `pg_dumpall` | Old cluster untouched; stop new, start old |
| Handles extensions | Yes, but pgvector must be installed in new cluster **after** upgrade | Yes |
| Risk of silent corruption | Very low; well-trodden path | Very low |
| Complexity | Medium (requires matching binaries in a tooling container) | Low |

**Decision**: Use **`pg_upgrade --link`** for the production upgrade. It fits inside the approved 45-minute maintenance window; `pg_dumpall` does not. The mandatory pre-upgrade `pg_dumpall` snapshot (Step 4.3 below) is our rollback artifact and preserves the full dump/restore escape hatch.

> **Dump/restore fallback is invoked only if `pg_upgrade --check` fails in Step 4.4.** In that case we abort, do **not** run `pg_upgrade`, and either (a) fix the reported issue and retry in the same window, or (b) postpone and re-plan. We do **not** mix strategies mid-window.

Resolves Tech Spec Open Question #1.

## 3. Migration Steps

### Parallelism Guidance

**All five components must run sequentially in the order listed in §1.1.** No two datastores are upgraded concurrently. Within a single component's sub-steps, snapshot → stop → upgrade → verify are strictly sequential. The only parallelism permitted is pre-pulling images (done in the checklist).

### Step 1 — Neo4j patch upgrade

| Duration Estimate | Executor |
|-------------------|----------|
| ~8 min (measured 6m12s in staging) | SRE |

```bash
# 1.1 Snapshot
docker compose exec -T neo4j neo4j-admin database dump neo4j \
  --to-path=/backups --overwrite-destination=true
docker compose cp neo4j:/backups/neo4j.dump ./backups/neo4j-preupgrade-$(date +%F).dump

# 1.2 Stop and upgrade
docker compose stop neo4j
# Edit docker-compose.yml: image: neo4j:5.<LATEST_PATCH>

# 1.3 Wait for ready
until docker compose exec -T neo4j cypher-shell -u neo4j -p "$NEO4J_PASSWORD" \
  "RETURN 1;" >/dev/null 2>&1; do sleep 2; done
```

**Expected output**: `neo4j` container `healthy`; `cypher-shell` returns `1`.

**Verification**:
```bash

  "CALL dbms.components() YIELD name, versions RETURN name, versions;"

  "RETURN apoc.version() AS apoc;"
docker compose exec -T neo4j cypher-shell -u neo4j -p "$NEO4J_PASSWORD" \
  "CALL gds.version();"
```

> **POINT OF NO RETURN** — Running `pg_upgrade --link` hard-links the PG 15 data files into the PG 17 cluster. From this moment, the only rollback is **restore from the `pg15-preupgrade-*.sql.gz` dump taken in Step 3.1 into a freshly re-initialized PG 15 cluster.** In-place downgrade is not possible. ```bash

  -v legalai_pgdata:/var/lib/postgresql/15/data \
  -v legalai_pgdata17:/var/lib/postgresql/17/data \

**Expected output**: `Upgrade Complete`, `Optimizer statistics are not transferred...`

**Verification**: exit code 0 and new data directory populated.

### Step 5 — Redis → Valkey 8

| Duration Estimate | Executor |
|-------------------|----------|
| ~6 min (measured 4m22s in staging) | SRE |

#### Step 5.0 — Valkey RDB Preflight (MANDATORY)

> **Do not skip.** Valkey 8 only reads RDB format versions **up to v11**. Redis ≥ 7.4 writes **v12**. If the persisted file on the existing `redis_data` volume is v12, the image swap will crash-loop and Step 5 will fail after the point where Celery has already been stopped. This was caught in PR 1.3.2 on the dev volume.

Inspect the RDB magic/version bytes of both the AOF base file and any `dump.rdb` on the live volume:

```bash
# Replace <project> with the compose project name (e.g. legalai)
docker run --rm -v <project>_redis_data:/d busybox sh -c \
  'od -An -c /d/appendonly.aof.1.base.rdb 2>/dev/null | head -1; \
   od -An -c /d/dump.rdb 2>/dev/null | head -1'
```

**How to read the output**: The first 9 bytes of any RDB payload are the ASCII magic `REDIS` followed by a 4-digit zero-padded version, i.e. `REDIS00XX`. `od -c` renders this as `R   E   D   I   S   0   0   0   B` for v11, `... 0   0   0   C` for v12, etc. The version is the **last two ASCII digits** of the `REDIS00XX` magic, interpreted as **hexadecimal** (`0B` = 11, `0C` = 12).

**Abort criteria**: if **either** file reports a version byte **> 0x0B (v11)** — i.e. `REDIS000C` or higher — **do NOT proceed with the in-place Valkey 8 upgrade.** Halt Step 5 and follow remediation path R-X in §7. Note that if the AOF is present, Redis loads the AOF in preference to `dump.rdb`, so an AOF at v12 is disqualifying even if `dump.rdb` is absent or older.

**Pass criteria**: all present files show `REDIS000B` or lower (or neither file exists, i.e. an empty volume). Record the observed magic string in the war-room channel before proceeding to 5.1.

#### Step 5.1 onward — Image swap

```bash
# 5.1 Flush and snapshot RDB (broker data is ephemeral; cache is reconstructible)
docker compose exec -T redis redis-cli BGSAVE
sleep 3
docker compose cp redis:/data/dump.rdb ./backups/redis72-preupgrade-$(date +%F).rdb

# 5.2 Stop Celery workers FIRST (drain queue)
docker compose stop celery-worker celery-beat

# 5.3 Swap image
docker compose stop redis
# Edit docker-compose.yml:
#   image: valkey/valkey:8
#   healthcheck test: ["CMD","valkey-cli","ping"]
#   volume still mounts /data — Valkey 8 reads the Redis 7.2 RDB directly
docker compose up -d redis

# 5.4 Wait for ready, then restart workers
until docker compose exec -T redis valkey-cli ping 2>/dev/null | grep -q PONG; do sleep 1; done
docker compose up -d celery-worker celery-beat
```

**Expected output**: `PONG` from `valkey-cli`; Celery workers reconnect (log line `celery@... ready`).

**Verification**:
```bash
docker compose exec -T redis valkey-cli INFO server | grep -E 'valkey_version|redis_version'
docker compose exec -T redis valkey-cli DBSIZE
docker compose exec -T backend python -c \
  "import redis; r=redis.Redis.from_url('$REDIS_URL'); r.set('mig_probe','ok'); print(r.get('mig_probe'))"
# Celery round trip
docker compose exec -T backend celery -A app.core.celery_app inspect ping
```
`valkey_version` present; `DBSIZE` ≥ pre-upgrade; probe returns `b'ok'`; Celery inspect returns `pong`.

## 6. Data Migration

All Phase 1 upgrades are schema-preserving. No row-level data movement between stores; only in-place format preservation via `pg_upgrade --link`, Qdrant snapshot/recover, Valkey RDB re-read, and Neo4j store-file preservation.

| Data | Source | Destination | Method | Estimated Duration |

## 7. Rollback Procedure

### Decision Authority

| Role | Authority |
|------|-----------|
| **Can call rollback (pre-PG-PONR)** | On-call SRE, unilaterally |
| **Can call rollback (post-PG-PONR, i.e., after Step 3.3)** | SRE Lead **and** Tech Lead, jointly, in writing (Slack `#incident-phase1`) |
| **Must be notified** | Engineering Manager, Customer Success on-call |
| **Must approve re-attempt** | SRE Lead + Tech Lead + DBA |

### Trigger Conditions

Call rollback if any of:
- A step's verification command fails and cannot be remediated in ≤ 5 min.
- Any abort condition in §4 fires.
- `pg_upgrade --check` fails in Step 3.2.
- Alembic `check` (Step 3.5) reveals unexpected drift.
- Cumulative elapsed time exceeds 50 minutes at the end of Step 3 (no time left to safely finish Steps 4–5).

### Rollback by Component

Rollback **only the component in flight plus any completed after it, in reverse order.** Do not rewind successfully-verified earlier components unless they are implicated.

#### R-X: RDB version > v11 detected in Step 5.0

Triggered when the preflight shows `REDIS000C` or higher on `appendonly.aof.1.base.rdb` and/or `dump.rdb`. Valkey 8 cannot load the file. Choose **one** remediation path based on environment and data criticality:

1. **Dev environment — flush volume and start clean on Valkey 8.** Acceptable because cache + broker data is reconstructible.
   ```bash
   docker compose stop celery-worker celery-beat redis
   docker volume rm <project>_redis_data
   # Edit docker-compose.yml: image: valkey/valkey:8
   docker compose up -d redis
   until docker compose exec -T redis valkey-cli ping 2>/dev/null | grep -q PONG; do sleep 1; done
   docker compose up -d celery-worker celery-beat
   ```

2. **Staging / prod — Redis 7.4 sidecar bridge.** Stand up a temporary Redis 7.4 container bound to the existing volume, replicate or `MIGRATE` the keyspace into a fresh Valkey 8 instance on a clean volume, cut over, retire the sidecar.
   ```bash
   # Sketch — full procedure in Runbook 1.0 §Valkey-RDB-Bridge (pending)
   docker run -d --name redis-bridge --network <project>_default \
     -v <project>_redis_data:/data redis:7.4 redis-server --appendonly yes
   docker run -d --name valkey-new --network <project>_default \
     -v <project>_redis_data_new:/data valkey/valkey:8
   # Option A: replicaof from valkey-new → redis-bridge until in-sync, then promote
   # Option B: scripted MIGRATE of all keys from redis-bridge to valkey-new
   # Then swap the compose 'redis' service to point at valkey-new's volume
   ```
   Decision authority: SRE Lead + Tech Lead. Adds ~20–40 min; will likely blow the maintenance window — communicate per §11.

3. **Any environment with critical, non-reconstructible data — abort upgrade and escalate.** Do not attempt an in-place swap and do not flush. Roll the Valkey step back (system remains on Redis), open an ADR-002 amendment ticket, and re-plan the Valkey migration as its own windowed change with the sidecar bridge as the pre-approved strategy.

In all three paths, record the observed RDB version, the path chosen, and the decision-maker in the incident channel before executing.

## 9. Downtime Estimate

| Component | Downtime | Mitigation |
|-----------|----------|------------|
| Neo4j | ~60s container swap | Maintenance window; graph is not on login critical path |
| Qdrant | ~90s (12 min if fallback) | Maintenance window; vector features degraded gracefully via pgvector once populated (Phase 3) |
| PostgreSQL | ~15 min (whole stack effectively down) | Scheduled 60-min maintenance window; status page banner |
| pgvector | ~0s (DDL only, inside PG downtime) | Overlaps PG window |
| Valkey | ~30s Valkey swap + ~60s Celery drain/restart | Cache misses permitted; Celery jobs re-queued on reconnect |
| **Total user-visible** | **~45 min worst case (single window)** | One 60-min maintenance window Sat 02:00–03:00 UTC |
