## 1. Overview

This Perf Spec defines latency, throughput, cost, and capacity budgets for the Phase 1 data-tier upgrade described in the [Phase 1 Tech Spec](./1.0_tech-spec_infrastructure-upgrades.md). The user-facing impact is indirect: Phase 1 is infrastructure plumbing, but every downstream phase (RAG, agents, frontend) inherits the performance envelope set here. A regression in Valkey GET latency or Qdrant k-NN throughput silently taxes every feature shipped after this phase.

The single new code path with measurable application latency is **pgvector k-NN** through `EmbeddingRepository.knn` (Tech Spec §3). Everything else is "do no harm" relative to the Phase 0 baseline.

### SLO vs. SLA Definitions

- **SLO**: Internal target. Missing it triggers investigation, dashboard alert, and (where wired) a CI gate.
- **SLA**: Customer contractual commitment. **No SLAs exist in Phase 1** — the platform is not yet customer-facing. All targets below are SLOs and become SLA candidates in Phase 3+.

| Backend image size | < +50 MB vs Phase 0 | Investigate; flag in PR review. |

CI action mapping:
- **Block merge**: `compare-baselines` job exits non-zero, branch protection enforces no merge.

## 11. Graceful Degradation Tiers

Phase 1 has no user-facing surface, so degradation is defined for the **shared infrastructure** that Phase 2+ will rely on. These tiers are wired into Phase 3 (alerting + auto-shed). The point of writing them now is so the runbook ([Migration Guide](./1.0_mig-guide_infrastructure-upgrades.md), Runbook to be authored) has the failure modes enumerated before code lands.

| Tier | Trigger Condition | What Degrades | User Impact | Auto / Manual |
|------|-------------------|---------------|-------------|---------------|
| 1 — Elevated | pgvector p95 > 1.5x target (75 ms) for 5 min, **OR** Valkey p99 > 2 ms for 5 min, **OR** Qdrant p95 > 1.5x Phase 0 baseline | Disable background re-embed jobs; pause non-critical Celery beat tasks; shed ingestion concurrency from 10→5 workers | Slower ingestion; analytics dashboards lag by 1–2 min | **Auto** (Celery beat schedule swap + worker autoscaler signal) |
| 2 — High | pgvector p95 > 2x target (100 ms) for 5 min, **OR** Valkey memory > 90% maxmemory, **OR** Qdrant p95 > 2x baseline, **OR** PG CPU > 90% for 10 min | Switch k-NN reads to **Qdrant only** via repository feature flag (pgvector becomes write-only); shed ingestion to 2 workers; serve cached search results (TTL 60s) | Search results may be up to 60s stale; ingestion backlog grows | **Auto** for read-path swap (flag flip); **manual approval** to extend cache TTL beyond 60s |
| 3 — Critical | pgvector p95 > 3x target (150 ms), **OR** PG CPU > 95% for 5 min, **OR** Valkey unresponsive, **OR** Qdrant OOM-killed | Reject new ingestion requests with 503 + Retry-After; queue them in S3 (or local disk in Phase 1) for replay; shed all non-essential reads; flip `PGVECTOR_INDEX_TYPE=ivfflat` and schedule reindex in maintenance window | "Ingestion paused — try again in 10 min" banner; existing queries succeed from cache | **Manual approval** (SRE on-call). Decision authority: SRE on-call; escalate to Tech Lead if degradation > 30 min. |
| 4 — Survival | PG at risk of OOM/disk-full, **OR** Valkey OOM-killed and not recovering, **OR** Qdrant container crash-looping | Shed all AI/search endpoints; serve only contract metadata reads from PG (no embeddings, no graph); enable read-only mode on PG | AI features unavailable; users see read-only banner; ingestion fully paused | **Manual** (SRE on-call + Tech Lead). Decision authority: Tech Lead. Recovery requires runbook §rollback. |

**Decision authority** for each manual tier is named so a tired on-call at 3am knows exactly who can pull the trigger without paging the world.

## 12. Optimization Opportunities

Known levers if budgets are exceeded. Listed in order of effort; pick the cheapest that closes the gap.

| Opportunity | Expected Improvement | Effort | Trade-off |
|-------------|---------------------|--------|-----------|
| Tune HNSW `ef_search` (runtime) lower (default 40 → 20) | -30% k-NN p95 latency | XS | -5 to -10% recall@10 |
| Tune HNSW `m=16, ef_construction=64` upward (`m=32, ef_construction=128`) | +5–10% recall@10 | S | 2x index size, 3x build time |
| Switch `PGVECTOR_INDEX_TYPE` to `ivfflat` | -50% index size, -40% build time | S | Higher tail latency, requires post-bulk REINDEX (already documented as fallback in Tech Spec) |
| Pre-warm pgvector index pages on app startup | Eliminates cold-start +200 ms | S | +5 s container startup |
| Pin `maintenance_work_mem=4GB` for migration session only | -50% Alembic `009` time on large datasets | XS | None (session-scoped) |
| Batch `EmbeddingRepository.insert` via PG `COPY` instead of `executemany` | +3–5x insert throughput | M | New code path, more failure modes |
| Add Valkey connection multiplexing (one pool per process, not per worker) | -20% connection overhead | S | Tighter pool sizing required |
| Repoint Celery result backend off Valkey to PG (results table) | Frees ~30% Valkey memory | M | Slower result lookup |
| Qdrant: enable on-disk payload storage for cold tenants | -40% RSS | M | +20 ms p95 cold reads |
| Qdrant: shard collections by tenant once any tenant > 5M vectors | Linear scale-out | L | Operational complexity; deferred to Phase 5 |
| Move PG WAL to dedicated volume | -20% write p95 under bulk insert | M | New volume, new backup target |
| Run pgvector and Qdrant query in parallel, return whichever is faster ("hedged read") | -30% p99 (tail) | L | 2x query cost; needs Phase 2 design |
| Pre-compute and cache top-K for hot queries in Valkey (60 s TTL) | -90% latency for repeated queries | M | Cache invalidation on re-embed |
