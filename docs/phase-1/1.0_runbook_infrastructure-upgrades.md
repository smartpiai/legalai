## 1. Service Overview

The Phase 1 datastore stack provides persistent state for the Legal AI Platform:

- **PostgreSQL 17 + pgvector 0.8** — primary OLTP, multi-tenant data, embeddings (HNSW indexes)
- **Valkey 8** — Redis-protocol cache, Celery broker, session store (drop-in for Redis 7)
- **Qdrant 1.12+** — high-dimensional vector search for document chunks (named vectors)
- **Neo4j 5.latest** — knowledge graph for clauses/parties/obligations (GraphRAG)

### Dependencies

| Dependency | Type | Impact if Unavailable |
|------------|------|----------------------|
| PostgreSQL 17 | Required | Backend cannot start; all API requests 503 |
| Valkey 8 | Required | Celery jobs queue stalls; sessions invalidated; auth flow degraded |
| Qdrant 1.12+ | Degraded | Vector search returns empty; falls back to PG pgvector if `FALLBACK_VECTOR=pg` |
| Neo4j 5 | Degraded | GraphRAG disabled; plain RAG still works |
| MinIO | Required (separate runbook) | Document upload/download fails |
| Docker / container runtime | Required | Stack down |

### Ports & Endpoints

| Endpoint | Port | Purpose |
|----------|------|---------|
| PostgreSQL | 5432 | SQL / pgvector |
| Valkey | 6379 | RESP protocol |
| Qdrant HTTP | 6333 | REST API + `/healthz`, `/readyz`, `/metrics` |
| Qdrant gRPC | 6334 | gRPC client |
| Neo4j Bolt | 7687 | Driver protocol |
| Neo4j HTTP | 7474 | Browser / REST |
| Backend `/health` | 8000 | Aggregated stack health |

### Health Check Definition

Backend `/health` aggregates each datastore. Schema:

```json
{
  "status": "healthy | degraded | unhealthy",
  "version": "1.2.3",
  "uptime_seconds": 86400,
  "checks": {
    "postgres":  { "status": "up", "latency_ms": 5,  "version": "17.2", "pgvector": "0.8.0" },
    "valkey":    { "status": "up", "latency_ms": 1,  "version": "8.0.1" },
    "qdrant":    { "status": "up", "latency_ms": 8,  "version": "1.12.4", "collections": 3 },
    "neo4j":     { "status": "up", "latency_ms": 12, "version": "5.26.0", "plugins": ["apoc","gds"] }
  }
}
```

**Status determination rules:**

| Condition | `status` | HTTP |
|-----------|----------|------|
| All required (`postgres`, `valkey`) up AND all degraded-tier up | `healthy` | 200 |
| Any degraded-tier (`qdrant`, `neo4j`) down | `degraded` | 200 |
| Any required down | `unhealthy` | 503 |
| Endpoint timeout (>2s) | N/A | N/A — page immediately |

## 2. Deployment

### Pre-Deployment Checklist

- [ ] Migration Guide dry-run completed in staging within last 7 days
- [ ] Snapshot of all datastores taken (`ops/backup/snapshot-all.sh`)
- [ ] Backup verified by test-restore in staging
- [ ] Maintenance window communicated (≥48h notice)
- [ ] Feature flag `READ_ONLY_MODE=true` ready to toggle
- [ ] Rollback script tested against snapshot
- [ ] Monitoring dashboards open (Grafana: Datastores, Backend Health)

### Deploy Order (cold start or full restart)

> Order matters: PG → Valkey → Qdrant → Neo4j → backend. Backend depends on all four.

```bash
# 1. PostgreSQL (longest startup; runs WAL recovery)
docker compose up -d postgres
until docker compose exec -T postgres pg_isready -U legalai -d legalai; do sleep 2; done

# 2. Valkey
docker compose up -d valkey
until docker compose exec -T valkey valkey-cli ping | grep -q PONG; do sleep 1; done

# 3. Qdrant
docker compose up -d qdrant
until curl -sf http://localhost:6333/readyz; do sleep 2; done

# 4. Neo4j (slowest; plugin load)
docker compose up -d neo4j
until docker compose exec -T neo4j cypher-shell -u neo4j -p "$NEO4J_PASSWORD" "RETURN 1" >/dev/null 2>&1; do sleep 3; done

# 5. Backend
docker compose up -d backend
```

### Post-Deployment Verification

```bash
# Aggregated health
curl -sf http://localhost:8000/health | jq .

# Per-component versions
docker compose exec -T postgres psql -U legalai -d legalai -c "SELECT version();"
docker compose exec -T postgres psql -U legalai -d legalai -c "SELECT extname, extversion FROM pg_extension WHERE extname='vector';"
docker compose exec -T valkey valkey-cli INFO server | grep valkey_version
curl -s http://localhost:6333/ | jq .version
docker compose exec -T neo4j cypher-shell -u neo4j -p "$NEO4J_PASSWORD" "CALL dbms.components() YIELD name, versions RETURN name, versions;"

# Smoke tests
docker compose exec -T backend pytest tests/smoke/ -q
```

## 4. Known Limits

| Resource | Hard Limit | Symptom When Exceeded | Action |
|----------|-----------|----------------------|--------|
| PG `max_connections` | 200 | `FATAL: sorry, too many clients`; backend 503 | Restart PgBouncer; investigate leaked sessions; raise limit only with memory headroom |
| PG data volume | 500 GB | Writes fail; DB enters read-only | Expand volume; `VACUUM FULL`; archive old tenants |
| pgvector HNSW build memory | `maintenance_work_mem` = 2 GB | OOM during `CREATE INDEX` | Build off-hours; raise `maintenance_work_mem` for session |
| Valkey `maxmemory` | 4 GB | Evictions begin (`allkeys-lru`); cache misses spike | Scale memory; review TTLs; check for unbounded keys |
| Valkey RDB fork memory | ~2× working set | `fork failed` in logs; no snapshots | Lower `save` frequency or enable `vm.overcommit_memory=1` |
| Qdrant disk | 200 GB | Writes rejected; `out of disk space` | Expand volume; delete stale collections; check WAL retention |
| Qdrant collection count | ~50 (soft) | Slow startup, high RAM | Consolidate via named vectors; archive |
| Neo4j heap | 4 GB | `OutOfMemoryError`; query failures | Increase `server.memory.heap.max_size`; tune queries |
| Neo4j page cache | 4 GB | Hit ratio drops; query latency spikes | Increase `server.memory.pagecache.size` |
| Neo4j max open files | 65535 | `Too many open files` | Raise ulimit on host |

### Issue 1: PostgreSQL 17 fails to start after upgrade

**Severity**: P0

**Symptoms**:
- `docker compose ps postgres` shows `Restarting` or `Exited`
- Logs: `database files are incompatible with server`, `could not load library "vector.so"`, or WAL replay errors
- Backend `/health`: `postgres: down`

**Root Cause**: Either (a) `pg_upgrade` didn't run / data dir is from PG 15/16, or (b) the `pgvector.so` shared library inside the new container is built against the wrong PG ABI, or (c) WAL from a crashed pre-upgrade instance.

**Diagnosis**:
```bash
# Check what version the data dir expects
docker compose exec -T postgres cat /var/lib/postgresql/data/PG_VERSION

# Check installed server version
docker compose exec -T postgres postgres --version

# Check pgvector lib
docker compose exec -T postgres ls -la /usr/lib/postgresql/17/lib/vector.so

# Tail startup logs
docker compose logs --tail 200 postgres
```

**Resolution**:
```bash
# A) Data dir is older than server → restore snapshot and re-run pg_upgrade per Migration Guide
docker compose stop postgres
bash ops/rollback/restore-postgres.sh --snapshot pre-upgrade-$(date +%F)

# B) pgvector ABI mismatch → pin known-good image and recreate
#   Edit docker-compose.yml: image: pgvector/pgvector:0.8.0-pg17
docker compose up -d --force-recreate postgres

# C) Crash recovery loop → start in single-user mode for inspection
docker compose run --rm --entrypoint "" postgres \
  postgres --single -D /var/lib/postgresql/data legalai
```

**Prevention**: Always run `pg_upgrade --check` in dry-run; pin pgvector image tag; take snapshot immediately before cutover.

### Issue 2: Valkey RDB incompatibility

**Severity**: P0

**Symptoms**:
- Valkey container restart loop
- Logs: `Bad file format reading the RDB file: Unexpected EOF reading RDB file` or `Can't handle RDB format version N`
- Celery workers report broker connection errors

**Root Cause**: An RDB snapshot written by Redis 7 with module data, or a forward-incompatible RDB version from a newer Redis fork, cannot be read by Valkey 8. Less commonly, the AOF file is partial.

**Diagnosis**:
```bash
docker compose logs --tail 100 valkey
docker compose exec -T valkey ls -la /data
docker compose exec -T valkey valkey-check-rdb /data/dump.rdb || true
docker compose exec -T valkey valkey-check-aof /data/appendonly.aof || true
```

**Resolution**:
```bash
# Option A — preferred: replay from a fresh logical dump taken pre-cutover
docker compose stop valkey
docker run --rm -v legalai_valkey_data:/data busybox sh -c 'rm -f /data/dump.rdb /data/appendonly.aof'
docker compose up -d valkey
# Then re-warm cache from source of truth (sessions will require re-login):
bash ops/cache/rewarm-from-postgres.sh

# Option B — repair AOF (if dump.rdb is the bad one and AOF is intact)
docker compose exec -T valkey valkey-check-aof --fix /data/appendonly.aof
docker compose restart valkey

# Option C — full rollback to Redis 7 snapshot
bash ops/rollback/restore-valkey.sh --snapshot pre-upgrade-$(date +%F)
```

**Prevention**: Before cutover, run `BGREWRITEAOF` and capture both RDB and AOF; verify `valkey-check-rdb` against the snapshot in staging. Disable Redis modules pre-migration.

### Issue 3: Qdrant collection format mismatch

**Severity**: P0 (if all collections affected) / P1 (single collection)

**Symptoms**:
- `/readyz` returns 503 or specific collection 404
- Logs: `failed to load segment`, `unknown format version`, `Wal: failed to open`
- Backend reports `qdrant: degraded`; vector search returns empty

**Root Cause**: On-disk segment/WAL format from a Qdrant version more than one minor behind 1.12 requires migration. Or the storage volume was restored from a snapshot taken mid-write.

**Diagnosis**:
```bash
curl -s http://localhost:6333/collections | jq .
curl -s http://localhost:6333/collections/contracts | jq '.result.status'
docker compose logs --tail 200 qdrant | grep -iE 'segment|wal|format|panic'
docker compose exec -T qdrant ls -la /qdrant/storage/collections/
```

**Resolution**:
```bash
# A) Single collection corrupt → recreate from snapshot
curl -X POST http://localhost:6333/collections/contracts/snapshots/recover \
  -H 'Content-Type: application/json' \
  -d '{"location":"file:///qdrant/snapshots/contracts-pre-upgrade.snapshot"}'

# B) Format-version too old → spin up intermediate Qdrant version, let it migrate, then upgrade
docker run --rm -v legalai_qdrant_data:/qdrant/storage qdrant/qdrant:1.10.1
# Wait for "loaded successfully" in logs, stop it, then bring 1.12 back up
docker compose up -d qdrant

# C) Last resort: drop and re-index from PostgreSQL source of truth
curl -X DELETE http://localhost:6333/collections/contracts
bash ops/qdrant/reindex-from-postgres.sh contracts
```

**Prevention**: Take Qdrant snapshots via the Snapshot API (not raw volume copy) before upgrades. Never skip more than one minor version.

### Issue 4: Neo4j plugin version mismatch

**Severity**: P1

**Symptoms**:
- Neo4j won't start or starts then exits
- Logs: `Plugin 'apoc' is not compatible with this version of Neo4j` or `NoSuchMethodError` during `CALL gds.*`
- Backend `/health`: `neo4j: down`; GraphRAG features fail

**Root Cause**: APOC and GDS jars are tightly coupled to a specific Neo4j minor version. Bumping Neo4j without bumping plugins (or vice versa) breaks startup.

**Diagnosis**:
```bash
docker compose exec -T neo4j ls -la /var/lib/neo4j/plugins/
docker compose logs --tail 200 neo4j | grep -iE 'plugin|incompatible|version'
docker compose exec -T neo4j cypher-shell -u neo4j -p "$NEO4J_PASSWORD" \
  "CALL dbms.components() YIELD name, versions RETURN name, versions;"
```

**Resolution**:
```bash
# Determine the running Neo4j minor (e.g., 5.26.0), then download matching plugins
NEO4J_VER=5.26.0
APOC_VER=5.26.0
GDS_VER=2.13.2  # check GDS compatibility matrix

docker compose stop neo4j
mkdir -p ./data/neo4j/plugins
curl -L -o ./data/neo4j/plugins/apoc-${APOC_VER}-core.jar \
  https://github.com/neo4j/apoc/releases/download/${APOC_VER}/apoc-${APOC_VER}-core.jar
curl -L -o ./data/neo4j/plugins/neo4j-graph-data-science-${GDS_VER}.jar \
  https://github.com/neo4j/graph-data-science/releases/download/${GDS_VER}/neo4j-graph-data-science-${GDS_VER}.jar

docker compose up -d neo4j
docker compose logs -f neo4j  # watch for "Started."
```

**Prevention**: Pin plugin versions in `docker-compose.yml` env (`NEO4J_PLUGINS`) or bake into a custom image. Add a CI check that the plugin versions match the Neo4j tag.

### Issue 5: pgvector extension not upgraded

**Severity**: P1

**Symptoms**:
- Queries fail with `ERROR: operator does not exist: vector <=> vector`
- `\dx vector` shows old version
- Backend logs: `UndefinedFunction` from SQLAlchemy

**Root Cause**: After a PG major upgrade, `ALTER EXTENSION vector UPDATE;` was not run.

**Diagnosis**:
```bash
docker compose exec -T postgres psql -U legalai -d legalai -c "\dx vector"
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "SELECT default_version, installed_version FROM pg_available_extensions WHERE name='vector';"
```

**Resolution**:
```bash
docker compose exec -T postgres psql -U legalai -d legalai -c "ALTER EXTENSION vector UPDATE;"
# Verify
docker compose exec -T postgres psql -U legalai -d legalai -c "\dx vector"
# Restart backend to clear any cached prepared statements
docker compose restart backend
```

**Prevention**: Add `ALTER EXTENSION vector UPDATE;` to the post-upgrade Alembic migration step in the Migration Guide.

- **PG**: Add read replicas via streaming replication (separate runbook). Writes remain single-primary in Phase 1.

## 8. Backup & Recovery

| Data | Backup Method | Frequency | Retention | Recovery Time |
|------|--------------|-----------|-----------|---------------|
| PostgreSQL | `pg_dump -Fc` + WAL archive | Daily full + continuous WAL | 30 days | ~30 min |
| Valkey | `BGSAVE` + AOF copy | Hourly | 7 days | ~5 min |
| Qdrant | Snapshot API per collection | Daily | 14 days | ~30 min/collection |
| Neo4j | `neo4j-admin database dump` | Daily | 30 days | ~20 min |
| All (cutover) | Volume snapshot | Pre-upgrade only | 14 days | ~10 min |

### Recovery Procedure

```bash
# Full stack rollback (use only with explicit authority — see Rollback Triggers)
bash ops/rollback/rollback-datastores.sh --snapshot pre-upgrade-$(date +%F)

# Per-component restore
bash ops/restore/restore-postgres.sh  --backup /backups/pg/legalai-YYYY-MM-DD.dump
bash ops/restore/restore-valkey.sh    --backup /backups/valkey/dump-YYYY-MM-DD.rdb
bash ops/restore/restore-qdrant.sh    --collection contracts --snapshot contracts-YYYY-MM-DD.snapshot
bash ops/restore/restore-neo4j.sh     --backup /backups/neo4j/neo4j-YYYY-MM-DD.dump
```

**Decision authority for full rollback**: Service Owner (Platform Eng Lead) OR on-call SRE during a P0. Document reason in incident channel before invoking.
