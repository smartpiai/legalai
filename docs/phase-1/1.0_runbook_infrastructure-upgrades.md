## 1. Service Overview

Phase 1 upgrades the platform's data tier. This runbook covers the four data services (PostgreSQL, Valkey, Qdrant, Neo4j) and the backend application's interaction with them. Application logic is unchanged; risk is concentrated in image/version pins, the new `pgvector` extension, and the new `document_embeddings` table.

### Dependencies

| Dependency | Type | Impact if Unavailable |
|------------|------|----------------------|
| PostgreSQL 17 (`postgres` container) | Required | Backend will not start; `/health` returns 503; all CRUD fails. |
| pgvector extension | Required | Embedding writes via `EmbeddingRepository` raise `UndefinedObject`; document ingest fails. |
| Valkey 8 (`valkey` container, ex-`redis`) | Required | Celery broker down → no async jobs; cache misses degrade latency but reads still work *briefly* before sessions break. |
| Qdrant 1.12 (`qdrant` container) | Required | Vector search returns 503; RAG endpoints fail. |
| Neo4j 5.x pinned (`neo4j` container) | Degraded | GraphRAG queries fail; lexical/vector retrieval still works. |
| MinIO | Required | Document upload fails; existing reads OK if cached. |
| Anthropic / OpenAI APIs | Required | LLM-backed endpoints fail; ingest still works. |

### Ports & Endpoints

| Endpoint | Port | Purpose |
|----------|------|---------|
| `GET /health` | 8000 | Aggregate health (DB, cache, qdrant, neo4j) |
| `GET /metrics` | 8000 | Prometheus metrics |
| PostgreSQL | 5432 | Primary DB + pgvector |
| Valkey | 6379 | Cache + Celery broker (wire-compatible with Redis) |
| Qdrant HTTP / gRPC | 6333 / 6334 | Vector store |
| Neo4j Bolt / HTTP | 7687 / 7474 | Graph DB |

### Health Check Definition

The `/health` endpoint returns:

```json
{
  "status": "healthy | degraded | unhealthy",
  "version": "1.2.3",
  "uptime_seconds": 86400,
  "checks": {
    "database":   { "status": "up", "latency_ms": 12 },
    "pgvector":   { "status": "up", "extension_version": "0.7.4" },
    "cache":      { "status": "up", "latency_ms": 2,  "backend": "valkey-8.0.1" },
    "qdrant":     { "status": "up", "latency_ms": 8,  "version": "1.12.4" },
    "neo4j":      { "status": "up", "latency_ms": 18, "version": "5.20.0" }
  }
}
```

**Status determination rules:**

| Condition | `status` Value | HTTP Code |
|-----------|---------------|-----------|
| All checks `up` | `healthy` | 200 |
| `neo4j` down (only non-required) | `degraded` | 200 |
| Any of `database` / `pgvector` / `cache` / `qdrant` down | `unhealthy` | 503 |
| Health endpoint itself unreachable | N/A (timeout) | N/A |

### Deploy Steps

```bash
# Step 1: Pull pinned images on every host
docker compose -f docker-compose.prod.yml pull

# Step 2: Apply DB migration 009 (creates document_embeddings + HNSW index)
docker compose run --rm backend alembic upgrade head

# Step 3: Bring up the data tier first (ordered)
docker compose -f docker-compose.prod.yml up -d postgres valkey qdrant neo4j

# Step 4: Wait for all data services to report healthy
for s in postgres valkey qdrant neo4j; do
  until [ "$(docker inspect -f '{{.State.Health.Status}}' $s)" = "healthy" ]; do
    echo "waiting for $s..."; sleep 3;
  done
done

# Step 5: Roll backend + workers
docker compose -f docker-compose.prod.yml up -d backend celery-worker
```

## 4. Known Limits

| Resource | Hard Limit | Symptom When Exceeded | Action |
|----------|-----------|----------------------|--------|
| PostgreSQL `max_connections` | 200 | "FATAL: sorry, too many clients", 503s on `/health` | Restart PgBouncer (or backend); investigate connection leaks via `pg_stat_activity` |
| PostgreSQL data volume | 100 GB | Writes fail, DB enters read-only | Expand volume; `VACUUM FULL` only if safe; archive old `document_embeddings` rows |
| pgvector HNSW index build memory | `maintenance_work_mem` = 2 GB | `out of memory` during `CREATE INDEX` | Increase `maintenance_work_mem` session-locally; rebuild off-peak |
| Valkey `maxmemory` | 4 GB | Evictions spike, sessions invalidate, login storms | Scale Valkey memory; verify `maxmemory-policy=allkeys-lru`; check for unbounded keys |
| Qdrant disk | 50 GB | Upserts return 5xx, collection turns red | Expand volume; delete unused collections; trigger snapshot + prune |
| Qdrant max concurrent searches | 64 | p95 latency spikes, request queuing | Scale qdrant vertically; reduce `hnsw_ef` for read path |
| Neo4j heap | 4 GB | OOM, container restart loop | Increase `NEO4J_server_memory_heap_max__size`; investigate runaway Cypher |
| Backend container memory | 2 GB | OOMKilled, pod restart loop | Bump memory limit; check for embedding-batch leaks |
| Anthropic API rate limit | 1000 RPM (tier-dependent) | Spike of 429s in backend logs | Backoff already in client; if sustained, page L2 to request limit increase |

### Issue 1: PostgreSQL down

**Severity**: P0

**Symptoms**: `/health` returns 503 with `database.status=down`. Backend logs show `OperationalError: could not connect to server`.

**Root Cause**: Container crash, OOM, disk full, or `max_connections` exhausted.

**Diagnosis**:
```bash
docker compose ps postgres
docker compose logs --tail 200 postgres
docker compose exec -T postgres psql -U legalai -d legalai -c "SELECT 1;"
df -h $(docker volume inspect legalai_postgres_data -f '{{.Mountpoint}}')
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "SELECT count(*), state FROM pg_stat_activity GROUP BY state;"
```

**Resolution**:
```bash
# If container is down: restart
docker compose restart postgres

# If connections exhausted: kill idle and restart backend
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "SELECT pg_terminate_backend(pid) FROM pg_stat_activity
    WHERE state='idle' AND state_change < now() - interval '10 min';"
docker compose restart backend

# If disk full: expand volume, then VACUUM
# (Coordinate with L2 — see Section 6)
```

**Prevention**: Connection pool monitoring (alert at >170); weekly VACUUM ANALYZE (Section 9); disk alert at 80 %.

### Issue 2: pgvector extension missing

**Severity**: P0

**Symptoms**: Backend logs show `psycopg2.errors.UndefinedObject: type "vector" does not exist` or `extension "vector" is not available`. Document ingest fails.

**Root Cause**: Migration 009 not applied, or Postgres image was rolled back without re-applying init script.

**Diagnosis**:
```bash
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "SELECT extname, extversion FROM pg_extension WHERE extname='vector';"
docker compose exec -T backend alembic current
```

**Resolution**:
```bash
# Create extension manually (idempotent)
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "CREATE EXTENSION IF NOT EXISTS vector;"

# Re-run alembic to head
docker compose exec -T backend alembic upgrade head

# Verify table + index
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "\d+ document_embeddings"
```

**Prevention**: CI smoke job already asserts `\dx vector`; add post-deploy gate (Section 2 Step 5 already calls this).

### Issue 3: Qdrant unreachable or collection red

**Severity**: P0

**Symptoms**: `/health` shows `qdrant.status=down`. Vector search endpoints return 503. Grafana → Qdrant shows collection status `red`.

**Root Cause**: Container OOM, disk full, corrupted segment, or version mismatch between client (`1.12`) and server.

**Diagnosis**:
```bash
docker compose ps qdrant
docker compose logs --tail 200 qdrant
curl -fsS http://localhost:6333/ | jq '.version'
curl -fsS http://localhost:6333/collections | jq
curl -fsS http://localhost:6333/collections/document_chunks | jq '.result.status'
df -h $(docker volume inspect legalai_qdrant_data -f '{{.Mountpoint}}')
```

**Resolution**:
```bash
# Restart
docker compose restart qdrant

# If still red: restore from latest snapshot (see Section 8)
curl -X POST "http://localhost:6333/collections/document_chunks/snapshots/recover" \
  -H 'Content-Type: application/json' \
  -d '{"location":"file:///qdrant/snapshots/document_chunks/latest.snapshot"}'

# If client/server mismatch: re-pull image
docker compose pull qdrant && docker compose up -d qdrant
```

**Prevention**: Disk alert at 80 %; weekly snapshot verification (Section 9); pin `qdrant-client==1.12.x` in `requirements.txt`.

### Issue 4: Valkey connection refused

**Severity**: P0

**Symptoms**: Backend logs `ConnectionError: Error 111 connecting to valkey:6379`. Sessions invalidated; Celery workers crashloop.

**Root Cause**: Container down, OOM eviction storm, or hostname not migrated (compose still references `redis`).

**Diagnosis**:
```bash
docker compose ps valkey
docker compose logs --tail 200 valkey
docker compose exec -T valkey valkey-cli PING
docker compose exec -T valkey valkey-cli INFO memory | grep -E 'used_memory_human|maxmemory_human|evicted_keys'
grep -rE 'redis://|REDIS_HOST' backend/app/ docker-compose*.yml
```

**Resolution**:
```bash
# Restart (wire-compatible with Redis client per ADR-001)
docker compose restart valkey

# If hostname mismatch (legacy `redis` references): hotfix env var
# Set REDIS_HOST=valkey in .env then:
docker compose up -d backend celery-worker

# If memory pressure: temporary maxmemory bump
docker compose exec -T valkey valkey-cli CONFIG SET maxmemory 6gb
```

**Prevention**: Eviction-rate alert; integration test asserts `valkey-cli PING` post-deploy; remove all `redis://` literals (tracked in PR 1.3.x).

### Issue 5: Slow vector search

**Severity**: P1

**Symptoms**: pgvector or Qdrant p95 > 250 ms; user reports of slow document search.

**Root Cause**: HNSW index missing/unbuilt, statistics stale, or `ef_search` too high.

**Diagnosis**:
```bash
# pgvector — verify HNSW index exists and is being used
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "\d+ document_embeddings"
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "EXPLAIN ANALYZE SELECT id FROM document_embeddings
    ORDER BY embedding <=> '[0,0,...]'::vector LIMIT 10;"

# Qdrant — collection config
curl -fsS http://localhost:6333/collections/document_chunks | jq '.result.config'
```

**Resolution**:
```bash
# Refresh stats
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "ANALYZE document_embeddings;"

# Rebuild HNSW (off-peak, takes minutes; locks writes briefly)
docker compose exec -T postgres psql -U legalai -d legalai -c \
  "REINDEX INDEX CONCURRENTLY document_embeddings_embedding_hnsw_idx;"

# Qdrant — lower ef_search at query time (application config)
# Coordinate with backend on-call; revert before next deploy.
```

**Prevention**: Latency SLO alert at 200 ms; nightly `ANALYZE`; perf regression CI gate (see [Perf Spec](./1.0_perf-spec_infrastructure-upgrades.md)).

### Issue 6: Anthropic 429 spike

**Severity**: P2

**Symptoms**: Backend logs show `429 Too Many Requests` from `api.anthropic.com`. Some agent calls slow due to backoff.

**Root Cause**: Upstream tier limit hit; usually a runaway loop or genuine traffic spike.

**Diagnosis**:
```bash
docker compose logs --since 30m backend | grep -c '429'
docker compose logs --since 30m backend | grep -i 'anthropic\|claude' | tail -50
```

**Resolution**:
```bash
# Confirm backoff is engaging (look for retry logs)
# If sustained > 30 min: enable agent-call circuit breaker (feature flag)
curl -X POST http://localhost:8000/internal/flags/llm_circuit_breaker -d 'on=true'
```

**Prevention**: Cost + RPM dashboards; per-tenant agent budget caps.

### Issue 7: Celery queue backlog

**Severity**: P1

**Symptoms**: Grafana → Workers shows queue depth growing; user-visible delay in async ops (ingest, indexing).

**Root Cause**: Worker crashloop (often pgvector/Valkey-related), or insufficient worker concurrency.

**Diagnosis**:
```bash
docker compose ps celery-worker
docker compose logs --tail 200 celery-worker
docker compose exec -T valkey valkey-cli LLEN celery
```

**Resolution**:
```bash
# Restart workers
docker compose restart celery-worker

# Scale up
docker compose up -d --scale celery-worker=4
```

**Prevention**: Worker liveness probe; alert at queue depth > 1000.

### Issue 8: Neo4j down (degraded)

**Severity**: P2

**Symptoms**: GraphRAG endpoints fail; `/health` returns 200 with `status=degraded`, `neo4j.status=down`. All other features work.

**Root Cause**: Neo4j OOM, APOC/GDS plugin load failure after pin change.

**Diagnosis**:
```bash
docker compose ps neo4j
docker compose logs --tail 200 neo4j
docker compose exec -T neo4j cypher-shell -u neo4j -p "$NEO4J_PASSWORD" "CALL dbms.components();"
```

**Resolution**:
```bash
docker compose restart neo4j
# If plugin load fails: re-pull pinned image
docker compose pull neo4j && docker compose up -d neo4j
```

**Prevention**: APOC + GDS smoke test in CI; heap alert at 85 %.

## 7. Scaling

## 8. Backup & Recovery

| Data | Backup Method | Frequency | Retention | Recovery Time |
|------|--------------|-----------|-----------|---------------|
| PostgreSQL (incl. `document_embeddings`) | `pg_dump -Fc` to S3 | Daily 02:00 UTC | 30 days | ~30 min for 100 GB |
| Qdrant collections | Snapshot API → S3 | Daily 03:00 UTC | 14 days | ~1 hour per collection |
| Neo4j | `neo4j-admin database dump` | Weekly Sunday 04:00 UTC | 4 weeks | ~30 min |
| Valkey | None — broker/cache is ephemeral by design | N/A | N/A | N/A |
| MinIO documents | S3 cross-region replication | Continuous | 90 days | Immediate |

### Recovery Procedure

```bash
# PostgreSQL — restore from latest dump
aws s3 cp s3://legalai-backups/postgres/latest.dump /tmp/latest.dump
docker compose exec -T postgres pg_restore -U legalai -d legalai --clean --if-exists /tmp/latest.dump
# Re-create extension if needed
docker compose exec -T postgres psql -U legalai -d legalai -c "CREATE EXTENSION IF NOT EXISTS vector;"

# Qdrant — restore collection from snapshot
curl -X POST "http://localhost:6333/collections/document_chunks/snapshots/recover" \
  -H 'Content-Type: application/json' \
  -d '{"location":"s3://legalai-backups/qdrant/document_chunks/latest.snapshot"}'

# Neo4j — restore from dump
docker compose stop neo4j
docker compose run --rm neo4j neo4j-admin database load --from-path=/backups neo4j --overwrite-destination=true
docker compose start neo4j
```
