# Test Specification

## 1. Scope

This spec defines the integration test strategy that validates every datastore upgrade in Phase 1: PostgreSQL 15→17 (+ pgvector), Redis 7.2 → Valkey 8, Qdrant 1.7 → 1.12+, and Neo4j latest patch. The goal is to prove that (a) the upgraded datastore accepts existing data, (b) existing client libraries continue to work, and (c) baseline performance does not regress.

### In Scope
- Version assertion for each upgraded component (running image matches target).
- Schema/data compatibility: pre-upgrade snapshot replays successfully on new version.
- Alembic migration replay on PG 17 (all 8 existing migrations, clean DB → head).
- `pgvector` extension install + basic CRUD on `document_embeddings` table.
- Valkey 8 compatibility across cache, session, Celery broker, pub/sub.
- Qdrant 1.12+ compatibility: collection create/read, named vectors, sparse vectors, snapshot restore.
- Neo4j + APOC + GDS plugin version compatibility.
- Fault-injection: graceful behavior during upgrade-step failures and snapshot-restore rollback.
- Post-upgrade end-to-end smoke (login → upload → query → logout).
- Regression check against Phase 0 performance baseline.

### Out of Scope
- Detailed performance benchmarking — reuses [Phase 0 baseline harness](../phase-0/0.0_perf-spec_baseline-metrics.md); no new Perf Spec required for Phase 1.
- Application business logic tests — unchanged, covered by existing Phase 0 suite.
- Security penetration testing — not mandated by the Document Matrix for Phase 1.
- Load/stress testing at production scale — deferred to Phase 2.
- Managed-cloud (RDS / ElastiCache / Aura) compatibility — out of phase scope.
- UI tests — no UI changes in Phase 1.

## 2. How to Run

### Prerequisites

| Requirement | Setup Command |
|-------------|---------------|
| Docker + Compose v2 | `docker compose version` (≥2.20) |
| Python venv + dev deps | `python -m venv venv && source venv/bin/activate && pip install -r backend/requirements-dev.txt` |
| Test env file | `cp .env.test .env` |
| Pre-upgrade snapshots (for rollback tests) | `./scripts/phase1/make_baseline_snapshots.sh` |
| Phase 1 target images pulled | `docker compose -f docker-compose.yml -f docker-compose.phase1.yml pull` |

### Commands

```bash
# Full Phase 1 integration suite (all components)
pytest tests/phase1/ -v

# Per-component suites
pytest tests/phase1/postgres/     -v    # 1.1 + 1.2 (PG 17 + pgvector)
pytest tests/phase1/valkey/       -v    # 1.3
pytest tests/phase1/qdrant/       -v    # 1.4
pytest tests/phase1/neo4j/        -v    # 1.5

# Alembic migration replay (1.1.2)
pytest tests/phase1/postgres/test_alembic_replay.py -v

# Post-upgrade smoke (run after all components upgraded)
pytest tests/phase1/smoke/ -v -m smoke

# Fault injection (slower, uses docker kill)
pytest tests/phase1/ -v -m fault_injection

# Performance regression vs Phase 0 baseline
./scripts/phase1/compare_to_baseline.sh
```

## 3. CI Pipeline Integration

| Test Suite | Trigger | Timeout | Required to Merge? |
|------------|---------|---------|-------------------|
| Version assertion (all components) | Every PR touching `docker-compose*.yml` or `backend/pyproject.toml` | 2 min | Yes |
| Per-component integration (P0) | Every PR in the component's scope | 10 min | Yes |
| Alembic replay on PG 17 | Every PR touching `backend/alembic/` or PG image | 8 min | Yes |
| Full Phase 1 integration (all components together) | PR to `main` with `phase-1` label | 25 min | Yes |
| Fault-injection suite | Nightly on `main` | 20 min | No (nightly advisory) |
| Post-upgrade smoke (E2E) | Nightly + pre-release | 15 min | Yes on release |
| Perf regression vs Phase 0 baseline | Nightly on `main` | 30 min | No (alerts on >15% regression) |

Budget: full Phase 1 CI suite must complete in **≤ 45 min** wall-clock to stay within the team's merge SLA.

| Layer | Target | Measurement |
|-------|--------|-------------|
| Unit | N/A — infrastructure phase, no new application logic at phase level | — |

## 10. Flaky Test Policy

| Aspect | Policy |
|--------|--------|
| Definition | Two unexplained failures in 7 days on identical code = flaky. Fault-injection tests using `docker kill` are high-risk; any failure that does not reproduce locally within 3 runs counts. |
| Quarantine process | Move to `tests/quarantine/phase1/`, excluded from merge-blocking CI, tracking issue filed within 24h. |
| Retry policy | CI retries failed tests once; retry-pass is flagged and surfaces in the weekly flaky report. Fault-injection tests are **not** auto-retried — a flaky fault-injection test usually indicates a real timing bug. |
| Fix ownership | SRE owns fault-injection flakes; feature engineer owns component-specific flakes; QA Lead owns cross-component flakes. 2-week fix-or-remove SLA. |
| Monitoring | Weekly flaky-test report from CI dashboard, reviewed at Friday QA sync. |

## 11. Pass / Fail Criteria

| Criteria | Threshold |
|----------|-----------|
| All P0 tests pass | Required for merge |
| All P1 tests pass | Required for merge |
| Alembic replay on PG 17 | 100% of 8 migrations apply clean |
| Snapshot-restore drill for every component | Data integrity (row/key/point/node counts) matches pre-upgrade |
| Post-upgrade smoke (E2E login→upload→query→logout) | Passes on upgraded stack |
| Performance regression vs Phase 0 baseline | ≤ 15% on any tracked metric (reuses [Phase 0 Perf Spec](../phase-0/0.0_perf-spec_baseline-metrics.md)) |
| No data loss in any fault-injection scenario | Zero silent losses; all failures produce loud errors |

## 12. Known Limitations

- Fault-injection tests require local Compose (not CI service containers) because they `docker kill` containers; they run in a dedicated CI job with Docker-in-Docker.
- `pg_upgrade` test uses a fixture-sized DB; production-sized `pg_upgrade` timing is validated separately during staging dress-rehearsal, not in CI.
- Performance regression check uses Phase 0 baselines captured on Phase 0 hardware; results are advisory when CI runners differ. Hardware-normalized comparison happens in the nightly `compare_to_baseline.sh`.
- Valkey RDB compatibility is the single highest-risk assumption; if IT-VK-07 fails, ADR-002 assumption breaks and the decision must be re-evaluated.
- Neo4j plugin version matrix (APOC/GDS) depends on upstream release timing; occasional skips are acceptable and tracked in 1.5.2.
