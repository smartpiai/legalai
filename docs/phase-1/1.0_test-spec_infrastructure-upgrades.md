## 1. Scope

Validates that the Phase 1 infrastructure upgrades land without functional, performance, or licensing regressions, and that the new capabilities (pgvector, Qdrant named/sparse vectors, pinned Neo4j) are usable by Phase 3 builders.

### In Scope
- PostgreSQL 15 → 17 upgrade, including all 8 existing Alembic migrations and `pg_stat_statements`
- pgvector extension, `document_embeddings` table, SQLAlchemy model, repository, and HNSW index
- Cache backend swap (Valkey or Redis 8 per ADR-001) — Celery broker, sessions, pub/sub, generic cache
- `qdrant-client` 1.7 → 1.12.x bump and Qdrant server 1.12+ upgrade
- Qdrant collection schema for **named vectors** and **sparse vectors**
- Neo4j 5 pinned-patch upgrade with APOC and GDS plugin verification
- Full application regression suite on the upgraded stack
- Rollback procedures (per Runbook) — verified via dry run
- Fault injection on each upgraded dependency

### Out of Scope
- Application feature code beyond `vector_store.py` refactor and the embedding repo (Phase 3)
- Production data migration / backfill (Phase 3.5.5; deployment runbook)
- Hybrid search implementation logic (Phase 3.6)
- End-user UI, accessibility, analytics — no UI in Phase 1
- Detailed perf SLOs beyond NFR-01..05 — covered here, no separate Perf Spec authored for Phase 1
- Visual regression — N/A (no UI)

## 2. How to Run

### Prerequisites

| Requirement | Setup Command |
|-------------|---------------|
| Docker + Compose v2 | `docker compose version` (≥ 2.20) |
| Upgraded stack running | `docker compose -f docker-compose.yml up -d postgres valkey qdrant neo4j` |
| Python venv + dev deps | `python -m venv venv && source venv/bin/activate && pip install -r backend/requirements-dev.txt` |
| Test env file | `cp backend/.env.test backend/.env` |
| Phase 0 baselines available | `ls tests/baselines/phase-0/*.json` (Qdrant, Redis, CI runtime) |
| Apply migrations | `cd backend && alembic upgrade head` |

### Commands

```bash
# Full Phase 1 suite (P0 + P1)
pytest backend/tests/infra/ -v -m "p0 or p1"

# By layer
pytest backend/tests/infra/unit/         -v        # Unit
pytest backend/tests/infra/integration/  -v        # Integration (requires docker compose up)
pytest backend/tests/infra/regression/   -v        # Phase 0 regression suite re-run on new stack

# With coverage (embedding repo target ≥ 90%)
pytest backend/tests/infra/ \
  --cov=backend/app/repositories/embedding_repository \
  --cov=backend/app/models/document_embedding \
  --cov=backend/app/services/vector_store \
  --cov-report=html --cov-fail-under=90

# Fault injection (toxiproxy + docker kill)
pytest backend/tests/infra/fault/ -v -m "fault"

# Performance gates (NFR-01..05)
pytest backend/tests/infra/perf/ -v -m "perf" --baseline=tests/baselines/phase-0
```

## 10. Flaky Test Policy

| Aspect | Policy |
|--------|--------|
| Definition | A test that fails non-deterministically on the same commit. Two unexplained failures within 7 days = flaky. |
| Quarantine process | Move to `backend/tests/quarantine/`, exclude from merge-blocking CI, file a P1 issue tagged `flaky`, link the failing CI runs. |
| Retry policy | CI retries failed tests **once**. A passing retry flags the test as *potentially flaky* and posts to `#eng-platform-flaky`. |
| Fix ownership | Owned by the PR author whose change introduced the test, or the team owning the touched module. |
| Fix deadline | **2 weeks** from quarantine. After 2 weeks: test is either fixed and re-enabled or **deleted** (with justification in the issue). No indefinite quarantine. |
| Monitoring | Weekly flaky test report from CI dashboard, reviewed in Friday platform sync. |
| Phase 1 hard rule | Fault injection (FI-*) and perf (PT-*) tests are **not** allowed to be quarantined during Phase 1 — they directly gate the upgrade. |

## 12. Known Limitations

- **Synthetic embeddings only.** PT-01 uses random 1536-d vectors, not real document embeddings. Real-data perf is validated in Phase 3.5.5 (re-indexing) test spec.
- **No production data migration tested.** Dev/staging only; production cutover is validated separately by the deployment runbook execution, not this spec.
- **Toxiproxy approximates network failure.** Real network partitions in cloud environments may behave differently; covered by staging validation, not unit/integration CI.
- **License scanning relies on `pip-licenses` + image scan.** Transitive native libraries may be missed; mitigated by manual review during dep-review phase, not by automated tests.
- **HNSW fallback path (EC-08b) is documented but not the default.** If NFR-01 fails, the fallback is exercised but not benchmarked at the same depth.
- **No AI behavior tests** — by design; Phase 1 is infra-only.
