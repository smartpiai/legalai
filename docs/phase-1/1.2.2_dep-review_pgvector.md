## 1. Motivation

The Legal AI Platform requires vector similarity search for embeddings (contract clause retrieval, RAG pipelines). While Qdrant is the primary vector store, PostgreSQL + `pgvector` is needed for:

- Co-located tenant metadata + small-scale embeddings (avoid cross-DB joins)
- Transactional guarantees on embedding writes alongside contract rows
- Simpler dev/test footprint (no separate vector DB for unit tests)

The `pgvector` Python package provides SQLAlchemy type bindings (`Vector`, `HALFVEC`, `SPARSEVEC`) and async support required for our SQLAlchemy 2.0 + asyncpg stack.

**Triggering event**: Implementation of embedding persistence layer (progress.md §1.2.2 — "Add pgvector extension + ORM bindings").

## 6. Transitive Dependencies

The `pgvector` Python package has **zero runtime dependencies** for the core install. Optional extras:

| Extra | Pulled In | Using? |
|-------|-----------|--------|
| `pgvector[sqlalchemy]` | `sqlalchemy>=2` (already a direct dep) | Yes |
| `pgvector[asyncpg]` | `asyncpg` (already a direct dep) | Yes |
| `pgvector[numpy]` | `numpy` (already a direct dep via embeddings stack) | Yes |
| `pgvector[psycopg]` | `psycopg>=3` | No |
| `pgvector[django]` | `django` | No |

**Total bundle impact**: +~80 KB backend site-packages. No new transitive deps introduced.

### Runtime Impact

| Aspect | Impact | Notes |
|--------|--------|-------|
| Cold start | Negligible (<5 ms import) | Pure Python, no native extension in client package |
| Memory overhead | <1 MB RSS | Type adapters only |
| Connection pool | None | Uses existing asyncpg pool |

### SQLAlchemy 2.0 Async + PG 17 Compatibility Verification

| Component | Version | Compatible? | Evidence |
|-----------|---------|-------------|----------|
| SQLAlchemy | 2.0.x (async) | Yes | `pgvector.sqlalchemy.Vector` registered as SQLAlchemy `TypeDecorator`; async tested in upstream CI against SA 2.0 |
| asyncpg | 0.29+ | Yes | Codec registered via `register_vector(conn)`; async example in pgvector README |
| PostgreSQL server | 17.x | Yes | pgvector server extension ≥0.7.4 supports PG 17; our Docker image `pgvector/pgvector:pg17` ships 0.8.0 |
| Alembic | 1.13+ | Yes | `Vector` type serializes in autogenerate via `pgvector.sqlalchemy` import in `env.py` (documented in README) |

Verified locally: async `insert` / `select` with cosine distance operator (`<=>`) works against `pgvector/pgvector:pg17` with SA 2.0.30 + asyncpg 0.29.0 + pgvector 0.3.6.

## 8. Alternatives Considered

| Alternative | Why Not Chosen |
|-------------|---------------|
| Raw SQL + custom asyncpg codec | Reinvents SQLAlchemy type integration; higher maintenance burden, no Alembic autogen support |
| `sqlalchemy-pgvector` (third-party) | Unmaintained (last release 2022), fewer downloads, not recommended by SA community |
| Qdrant-only (no PG vector) | Loses transactional co-location with tenant/contract rows; complicates small-scale dev/test |
| `pg_embedding` (Neon) | Narrower ecosystem, weaker SA integration, Neon-focused |

## 9. Rollback Plan

```bash
# Remove Python package
poetry remove pgvector
# Drop the extension (destructive — only in dev)
psql -c "DROP EXTENSION IF EXISTS vector CASCADE;"
# Revert Alembic migration that added vector columns
alembic downgrade -1
```

Production rollback: revert the PR, run `alembic downgrade` to the pre-pgvector revision. Embedding data will be dropped; backfill from Qdrant (source of truth) if needed.
