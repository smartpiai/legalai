# ADR-001: Cache/Broker Backend — Valkey vs Redis 8

> **Date**: 2026-04-07
> **Decision Date**: Pending
> **Review By**: 2026-10-07
> **Decision Owner**: Tech Lead (SRE delegate: SRE Lead)
> **Status**: Draft
> **Supersedes**: None
> **Deciders**: Tech Lead, SRE Lead, Backend Lead, Security
> **Phase**: 1 | **PR**: 1.3.1

## Context

The platform currently runs **Redis 7 (Alpine)** as the backing store for three workloads:

1. **Celery broker + result backend** (background contract processing, AI pipelines)
2. **Session store** (FastAPI auth, JWT denylist)
3. **Application cache + pub/sub** (tenant cache, real-time notifications)

In **March 2024, Redis Inc. relicensed Redis 7.4+ under SSPL/RSALv2**, dropping the BSD license. The SSPL's "service provider" clause introduces legal ambiguity for self-hosted enterprise deployments where the platform is "made available as a service" to tenants. Two enterprise prospects (~$200K combined ARR) have already raised SSPL concerns during procurement review (per [BRD §3](./1.0_brd_infrastructure-upgrades.md)).

We must choose a forward path for the cache/broker layer as part of Phase 1 infrastructure upgrades. The choice is between **Valkey** (the Linux Foundation BSD-licensed Redis 7.2.4 fork) and **Redis 8 OSS** (Redis Inc.'s newly-announced AGPLv3 re-open-sourcing, May 2025).

## Assumptions

- **A1**: Valkey 8.x maintains wire-protocol and command compatibility with Redis 7.2 such that `redis-py` ≥ 5.0 and `celery[redis]` work without code changes.
  *Monitor*: SRE Lead — verified by integration tests in PR 1.3.3.
- **A2**: Redis 8 OSS (AGPLv3) is acceptable to Legal/Security for our self-hosted enterprise distribution model. AGPLv3's network-copyleft does not contaminate our application code because Redis is accessed only as a network service, not linked.
  *Monitor*: Security — async review of this ADR.
- **A3**: Neither Valkey nor Redis 8 introduces a regression in p99 latency for our workloads (Celery enqueue, session GET/SET, pub/sub fanout) versus Redis 7.
  *Monitor*: SRE Lead — baseline captured in Phase 0; re-measured in PR 1.3.3.
- **A4**: At least one of {Valkey, Redis 8} will remain actively maintained with security patches over the next 24 months.
  *Monitor*: Tech Lead — review at the Review-By date.
- **A5**: Our deployment footprint stays single-node for Phase 1 (no clustering). Cluster-mode parity differences between Valkey and Redis 8 are not blocking.
  *Monitor*: SRE Lead — re-evaluate before Phase 4 scale work.

## Scope

- `docker-compose.yml` — `redis` service image
- `docker-compose.prod.yml` — production overrides
- `backend/app/core/config.py` — connection URL env vars (no value change expected)
- `backend/app/core/cache.py`, `backend/app/core/celery_app.py` — client init (no code change expected)
- CI workflow `.github/workflows/ci.yml` — service container image
- **Out of scope**: Redis Cluster, Redis Sentinel, RedisJSON / RediSearch modules (we use none of these today).

## Decision Drivers

- **D1 — License cleanliness**: Must remove SSPL exposure to unblock the two enterprise prospects flagged in the BRD.
- **D2 — Zero application code change**: Phase 1 is infrastructure-only; any option that requires refactoring `cache.py` or `celery_app.py` falls out of scope.
- **D3 — Long-term maintenance signal**: We need confidence the project will receive security patches through 2028+.
- **D4 — Operational familiarity**: SRE team has deep Redis 7 ops experience; steep relearning is a cost.

## Options Considered

### Option A: Valkey 8.x (Linux Foundation fork)

Adopt [Valkey](https://valkey.io/), the Linux Foundation–hosted BSD-3-Clause fork of Redis 7.2.4, backed by AWS, Google Cloud, Oracle, and Ericsson. Swap the docker image from `redis:7-alpine` to `valkey/valkey:8-alpine`.

**Pros:**
- BSD-3-Clause license — zero ambiguity, identical to pre-2024 Redis terms.
- Wire-protocol and command compatible with Redis 7.2; `redis-py` and Celery work unchanged.
- Strong governance: Linux Foundation neutral steward, multi-vendor contributor base reduces single-vendor capture risk.
- Active release cadence (Valkey 7.2.5, 7.2.6, 8.0 GA in 2024–2025).
- Already adopted by AWS ElastiCache and Google MemoryStore as default — large operator base de-risks long-term support.

**Cons:**
- Younger project (forked April 2024); operational track record at our scale is ~18 months.
- Diverging feature set from Redis 8 (e.g., Redis 8's vector sets, hash field TTL) — we don't need these today but lose optionality.
- Some observability tooling (RedisInsight) is Redis-Inc.-branded and may not officially support Valkey.

**Effort**: S (image swap + integration test)
**Risk**: Low

### Option B: Redis 8 OSS (AGPLv3)

Upgrade to [Redis 8.0](https://redis.io/blog/redis-8-ga/), released May 2025 under AGPLv3 (Redis Inc. re-open-sourced after community pressure). Swap the docker image from `redis:7-alpine` to `redis:8-alpine`.

**Pros:**
- Single upstream, continuous lineage with our existing Redis 7 — zero relearning for SRE.
- AGPLv3 is OSI-approved and Free Software Foundation–endorsed; cleaner story than SSPL.
- Newer feature set: vector sets, hash field TTL, improved `CLIENT NO-EVICT` — potential future use.
- Full compatibility with RedisInsight, redis-cli, all existing observability tooling.

**Cons:**
- **AGPLv3 network-copyleft requires Legal/Security sign-off** for our enterprise self-hosted distribution. While we believe A2 holds, it is a non-trivial review and a residual reputational risk with enterprise procurement teams who have a blanket "no AGPL" policy.
- Single-vendor governance — Redis Inc. relicensed once and could do so again.
- Some enterprise customers explicitly disallow AGPL anywhere in their stack regardless of linkage.

**Effort**: S (image swap + integration test)
**Risk**: Medium (legal review dependency)

### Option C: Stay on Redis 7.2.4 (last BSD release)

Pin to `redis:7.2.4-alpine`, the final BSD-licensed Redis release.

**Pros:**
- Zero migration work.
- Known-good license.

**Cons:**
- **No security patches** — 7.2.4 is frozen; any future CVE is unpatched upstream.
- Violates Phase 1 BRD success metric ("Cache backend license: BSD or OSS-compatible *with active maintenance*").
- Just defers the decision 6–12 months under worse conditions.

**Effort**: S
**Risk**: High (unpatched CVE exposure)

### Option D: Replace with KeyDB or Dragonfly

Considered and ruled out inline:
- **KeyDB**: Snap acquired and effectively unmaintained since 2023; last release Feb 2023. Fails D3.
- **Dragonfly**: BSL license (not OSI-approved), and a from-scratch reimplementation — higher behavioral-divergence risk for Celery broker semantics. Fails D1 and D2.

## Decision

**Chosen option**: **Option A — Valkey 8.x**, because it is the only option that satisfies all four decision drivers simultaneously: BSD license (D1), zero code change (D2), multi-vendor Linux Foundation governance with major cloud-provider adoption signaling long-term maintenance (D3), and full Redis 7.2 protocol compatibility preserving SRE operational knowledge (D4). Redis 8's AGPLv3 introduces a Legal review dependency and a residual enterprise-procurement risk we have no business reason to absorb.

## Consequences

### Positive
- Eliminates SSPL exposure; unblocks the two flagged enterprise prospects.
- Aligns with the broader industry pivot (AWS, GCP, Oracle defaulting to Valkey).
- No application code changes; Phase 1 stays infrastructure-only as scoped.
- Future managed-service migration path is straightforward (ElastiCache Valkey, MemoryStore Valkey).

### Negative
- We commit to a project with ~18 months of independent track record.
- We forgo Redis 8's newer features (vector sets, hash field TTL) for now. If we later need them, migrating Valkey → Redis 8 is non-trivial because of the AGPL question.
- RedisInsight may not officially support Valkey; we may need to switch to `valkey-cli` + Prometheus exporters for observability.

### Trade-offs & Accepted Risks
- **Accepted**: Slightly younger upstream in exchange for license cleanliness and multi-vendor governance.
- **Accepted**: Loss of feature optionality (Redis 8 modules) in exchange for avoiding AGPLv3 enterprise friction.
- **Accepted**: One-time observability tooling migration cost (small).
