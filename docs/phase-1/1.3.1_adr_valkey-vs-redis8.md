# ADR-002: Valkey vs Redis 8 for Cache, Session Store, and Task Broker

> **Date**: 2026-04-09
> **Decision Date**: Pending
> **Review By**: 2026-10-09
> **Decision Owner**: Tech Lead (Platform)
> **Status**: Proposed
> **Supersedes**: None
> **Deciders**: Tech Lead, SRE Lead, Backend Eng Lead
> **Phase**: 1 | **PR**: 1.3.1

## Context

In March 2024, Redis Ltd. relicensed Redis from the permissive BSD-3-Clause license to a dual SSPLv1 / RSALv2 source-available license starting with Redis 7.4. This change introduces legal ambiguity for Legal AI Platform, which ships Redis as part of its self-hosted deployment footprint and offers a managed multi-tenant SaaS — both potentially implicated by SSPL's "managed service" clauses.

In response, the Linux Foundation forked Redis 7.2.4 as **Valkey**, maintained under the original BSD-3-Clause license with backing from AWS, Google Cloud, Oracle, and Ericsson. Separately, Redis Ltd. released **Redis 8** (May 2025) under a tri-license that now includes AGPLv3 as an OSI-approved option alongside SSPL/RSAL.

Phase 1 requires us to move off the pinned Redis 7.2 image (EOL, no security patches) before downstream phases depend on newer client features. We must choose one path: Valkey or Redis 8. This decision affects the cache layer, session store, Celery broker (until replaced in Phase 2.4), and pub/sub for real-time updates.

## Assumptions

- **Wire protocol compatibility**: Both Valkey 8.x and Redis 8.x remain wire-compatible with Redis 7.2 clients (`redis-py`, `celery[redis]`). Monitor: Backend Eng Lead via integration tests in 1.3.3.
- **AGPLv3 in Redis 8 is not triggered** by our usage pattern (we do not modify Redis source, we run it as an unmodified container). Monitor: Tech Lead + Legal review before go-live.
- **Valkey's governance remains healthy** through 2026: ≥3 active corporate maintainers, release cadence ≥2 minor versions/year. Monitor: Tech Lead quarterly.
- **Managed cloud availability is not required in Phase 1** (we run containers in all environments). If we move to managed ElastiCache/MemoryStore later, both AWS and GCP already offer Valkey. Monitor: SRE Lead.
- **No dependency on Redis Stack modules** (RediSearch, RedisJSON, RedisTimeSeries). Qdrant handles vector search; pgvector added in 1.2. Monitor: Tech Lead during 1.3.3.
- **RDB on-disk format compatibility**: Valkey 8 can read existing Redis 7.2 RDB files in place, enabling a simple image-swap migration. Monitor: SRE Lead during 1.3.2 dry-run.

### Assumption Validation Update (2026-04-09)

- **FALSIFIED — RDB on-disk format compatibility**: During PR 1.3.2 implementation, the existing dev `redis_data` volume was found to contain `appendonly.aof.1.base.rdb` written at **RDB format version 12**, produced by a Redis ≥7.4 binary during prior experimentation on the volume. **Valkey 8.1.6 only reads up to RDB v11** and refuses to start against a v12 file. The assumption that "Valkey 8 reads existing Redis RDB files directly" does not hold whenever any Redis ≥7.4 has ever touched the data directory.
  - **Evidence**: Valkey 8.1.6 startup failure against the dev volume; file header inspection of `appendonly.aof.1.base.rdb` showing RDB v12 magic.
  - **Impact on decision**: None — Valkey remains the chosen option. Only the migration mechanics change.
  - **Resolution (dev)**: Flush the `redis_data` volume and start clean on Valkey 8. Acceptable because dev holds no production data (only ephemeral cache/session/broker state).
  - **Resolution (staging/prod, when applicable)**: Deploy a Redis 7.4 → Valkey 8 online migration sidecar (e.g., `RIOT` or equivalent replication-based tool) rather than attempting an in-place RDB swap. To be specified in the 1.3.2 Mig Guide before staging rollout.

## Scope

Affected components:
- `docker-compose.yml` — `redis` service image
- `backend/app/core/cache.py` — cache client
- `backend/app/core/session.py` — session store
- `backend/app/core/celery_app.py` — broker + result backend (until Phase 2.4)
- `backend/app/services/realtime.py` — pub/sub
- CI fixtures in `.github/workflows/` and `tests/conftest.py`
- Runbook and Mig Guide for 1.3.2

Out of scope: Celery replacement (Phase 2.4), managed-cloud migration, Redis Cluster mode.

## Decision Drivers

1. **License safety for SaaS offering**: SSPL's "managed service" clause creates legal risk for our multi-tenant deployment; Legal has flagged SSPL as unacceptable for production.
2. **Long-term maintenance and patch availability**: Redis 7.2 is EOL; we need a vendor/community committed to security patches through at least 2027.
3. **Zero application code churn**: Phase 1 is infrastructure-only. Whichever option minimizes code and operational change wins ties.
4. **Alignment with cloud providers**: AWS ElastiCache and GCP MemoryStore have both adopted Valkey as their default — relevant if/when we offer managed deployments.

## Options Considered

### Option A: Valkey 8.x (Linux Foundation fork)

Replace the `redis:7.2` image with `valkey/valkey:8`. Keep `redis-py` client (Valkey is wire-compatible). Update config file naming (`valkey.conf`) and health-check command (`valkey-cli ping`).

**Pros:**
- BSD-3-Clause license — zero legal ambiguity, matches Legal's guidance.
- Backed by Linux Foundation with multi-vendor governance (AWS, GCP, Oracle, Ericsson) — no single-vendor capture risk.
- Default choice on AWS ElastiCache and GCP MemoryStore as of 2024–2025, smoothing any future managed-cloud migration.
- Wire-compatible with existing `redis-py` and `celery[redis]` clients — no application code changes.
- Active release cadence (8.0 GA Apr 2025, 8.1 late 2025) with security backports to 7.2 line.

**Cons:**
- Ecosystem tooling (monitoring dashboards, Helm charts, Terraform modules) still catching up; some references still say "redis".
- Lacks Redis Stack modules (RediSearch, RedisJSON) — not needed for our use case but closes that door.
- Slightly younger fork; long-term divergence from Redis protocol is possible (low probability given wire-compat commitment).

**Effort**: S (image swap, config rename, health-check update, CI fixture refresh)
**Risk**: Low

### Option B: Redis 8 under AGPLv3

Upgrade to `redis:8` and elect the AGPLv3 license option from the tri-license.

**Pros:**
- Continuity: same vendor, same image name, same documentation corpus; team familiarity is highest.
- AGPLv3 is OSI-approved, which resolves the hard blocker from the original SSPL-only relicense.
- Access to Redis Stack modules (RediSearch, RedisJSON) if we ever want them.
- Redis Ltd. remains the primary maintainer with a large full-time engineering team.

**Cons:**
- **AGPLv3 network-copyleft clause** (§13) creates residual legal review burden for our SaaS — Legal must confirm our deployment model doesn't trigger source-disclosure obligations. Even a "likely fine" verdict is more expensive than Valkey's unambiguous BSD.
- Single-vendor governance: Redis Ltd. has now relicensed twice (BSD → SSPL → tri-license); a third change is not impossible.
- Diverging from cloud providers' default (both AWS and GCP promote Valkey), increasing friction for future managed migration.
- Tri-license complexity means every future dependency review must re-verify which license applies.

**Effort**: S (image swap only)
**Risk**: Medium (legal review dependency)

### Option C: Stay on Redis 7.2 (status quo)

Pin to the last BSD-licensed Redis release.

**Ruled out**: Redis 7.2 reached EOL in 2024; no further security patches from upstream. Unacceptable for a platform handling legal data.

## Decision

**Chosen option**: **Option A — Valkey 8.x**, because it eliminates the license ambiguity that drove this decision in the first place (Driver 1), provides a vendor-neutral maintenance path (Driver 2), and requires zero application code changes thanks to wire compatibility (Driver 3), while aligning with the managed-cloud direction of AWS and GCP (Driver 4).

## Consequences

### Positive
- Removes SSPL/AGPL legal review from the critical path for Phase 1 deployment.
- Keeps `redis-py` and `celery[redis]` unchanged — 1.3 becomes a pure infrastructure PR.
- Forward-compatible with AWS ElastiCache for Valkey and GCP MemoryStore for Valkey if we offer managed deployments later.
- Establishes a precedent (Valkey) that the team can cite when reviewing future source-available relicenses.

### Negative
- **Existing environments with RDB v12 on disk (from any Redis ≥7.4 contact) cannot be in-place upgraded to Valkey 8**; they require either flush-and-restart (dev) or a replication-based sidecar migration (staging/prod). This invalidates the original "simple image swap" migration story for any environment whose data directory has ever been touched by Redis ≥7.4.
- Some observability tooling (Grafana dashboards, exporters) still labelled "redis"; SRE must verify `valkey_exporter` or confirm `redis_exporter` works unchanged.
- Team documentation and runbooks must be updated to reference Valkey, creating a short-term search/discoverability cost.
- If Valkey governance deteriorates (assumption breaks), we must re-evaluate — potentially migrating again.

### Trade-offs & Accepted Risks
- **Accepting** a younger fork in exchange for license clarity and multi-vendor governance.
- **Accepting** loss of access to Redis Stack modules (RediSearch, RedisJSON) — acceptable because Qdrant and pgvector cover our search needs.
- **Accepting** short-term tooling/documentation friction in exchange for long-term legal certainty.
