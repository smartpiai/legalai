## 1. Motivation

The Legal AI Platform's vector search layer currently pins `qdrant-client==1.7.x`, which predates several capabilities required for Phase 1 RAG and GraphRAG features:

- **Sparse vectors** (BM25-style hybrid search) for legal-text retrieval where lexical precision matters (clause/citation matching).
- **Named vectors** per collection to co-locate dense embeddings (OpenAI `text-embedding-3-large`) with legal-domain embeddings without duplicating collections.
- **Query API** (`client.query_points`) unifying search, recommend, and hybrid fusion — required by `app/services/retrieval_service.py`.
- Server compatibility: our Qdrant server image is being bumped to 1.12 in the same infra PR; client ≥ 1.12 is required to avoid gRPC schema drift warnings.

**Triggering event**: Phase 1 Week 5–6 "Basic AI Integration" requires hybrid (dense+sparse) retrieval; the 1.7 client cannot express sparse vector configs.
**Tracking issue**: [LAI-141](https://tracker.internal/LAI-141)

## 5. Breaking Changes

API surface changes between 1.7 and 1.12 that affect our code:

| Breaking Change | Impact on Our Code | Fix Required | Tracked In |
|-----------------|-------------------|-------------|------------|
| **Named vectors**: `VectorParams` can no longer be passed directly to `create_collection(vectors_config=...)` as a bare object when using multiple vectors; must use `dict[str, VectorParams]`. Legacy single-vector form still works but emits `DeprecationWarning`. | `app/repositories/vector_repository.py` — `create_collection()` helper (1 file) | Wrap in `{"dense": VectorParams(...)}` dict and update read paths to pass `using="dense"`. | [LAI-142](https://tracker.internal/LAI-142) |
| **Sparse vectors**: new `sparse_vectors_config` parameter on `create_collection`; requires `SparseVectorParams` and `SparseIndexParams`. No drop-in from 1.7 (feature did not exist). | New code in `vector_repository.py` and `retrieval_service.py` (hybrid search path) | Additive; add `"sparse": SparseVectorParams(index=SparseIndexParams(on_disk=False))`. | [LAI-143](https://tracker.internal/LAI-143) |
| **Collection config schema**: `optimizers_config`, `hnsw_config`, `quantization_config` moved under typed Pydantic v2 models. Dict-form kwargs raise `ValidationError` in 1.11+. | `app/repositories/vector_repository.py`, `tests/repositories/test_vector_repository.py` (2 files) | Replace dict literals with `OptimizersConfigDiff(...)`, `HnswConfigDiff(...)`. | LAI-142 |
| **`search()` deprecated** in favor of `query_points()`. Still functional in 1.12 with warning; scheduled for removal in 2.0. | `retrieval_service.py` (3 call sites) **and** `app/services/embedding.py` — the legacy `VectorSearchService` (`create_collection`, `upsert_embeddings`, `search_similar`, `upsert_with_sparse`, `search_sparse`, `hybrid_search`) which the original review missed. Migrated in PR 1.4.3. | Migrate to `query_points(collection_name, query=..., using="dense", limit=k)`. Enables hybrid fusion later. | [LAI-144](https://tracker.internal/LAI-144) |
| **`recreate_collection()` removed** (was deprecated in 1.8, removed in 1.11). | Test fixtures in `tests/conftest.py` (1 file) | Replace with `delete_collection()` + `create_collection()`. | LAI-142 |

> **FU-5 amendment (2026-04-08):** the `search()` row originally only listed `retrieval_service.py`. During the 1.4.3 named-vectors sweep we found `backend/app/services/embedding.py::VectorSearchService` holds a parallel, older set of Qdrant call sites this Dep Review missed. Those sites were already migrated to `query_points` + named vectors in PR 1.4.3; the row above is corrected so a future grep for "files touched by the qdrant-client upgrade" returns the complete set.
| **Pydantic v2 required** (client models migrated in 1.10). Our project is already on Pydantic v2, so no impact — noted for record. | None | None | — |
| **Async client**: `AsyncQdrantClient` method signatures now mirror sync exactly; a couple of positional args became keyword-only. | `retrieval_service.py` (async paths) | Convert positional `collection_name, query_vector` → keyword args. | LAI-144 |
| **gRPC protocol**: requires Qdrant server ≥ 1.9 for full feature parity; ≥ 1.12 recommended. | Infra (Docker compose + Helm values) | Bump server image to `v1.12.1`. | [LAI-145](https://tracker.internal/LAI-145) |

**Python version compatibility**: qdrant-client 1.12 supports **Python 3.9–3.13**. Our backend runs Python 3.11, well within the support window. Dropped support for Python 3.8 occurred in 1.10 — not relevant to us. No change required to `pyproject.toml` `requires-python`.

### Migration Effort Estimate

| Area | Files Touched | Est. Effort |
|------|---------------|-------------|
| Repository layer refactor (named + sparse vectors, config models) | 2 | 0.5 day |
| Service layer migration to `query_points` + hybrid | 2 | 0.5 day |
| Test fixtures and repository tests | 3 | 0.5 day |
| Alembic-equivalent: Qdrant collection re-creation script (dev + staging re-index) | 1 new script | 0.5 day |
| Infra bump + smoke tests | — | 0.25 day |
| **Total** | **~8 files** | **~2.25 engineer-days** |

Re-indexing: collections must be dropped and recreated because sparse vector config cannot be added to an existing collection in-place. Dev/staging can re-embed from source; **production has no data yet** (Phase 1 Week 5), so no prod migration window is needed. If this review is revisited post-launch, a blue/green collection swap via aliases will be required (documented in LAI-145 follow-up).

| `protobuf` 5.28.x (bumped from 4.25.x) | grpcio | BSD-3-Clause | ~400 KB | **Major bump** — verified compatible with our other consumers (`opentelemetry-proto`, `google-cloud-storage`). |
| `portalocker` ≥ 2.7 | qdrant-client (local mode) | BSD-3-Clause | ~20 KB | New (1.10+) |
| `pydantic` ≥ 2.5 | qdrant-client | MIT | — | Already present, no bump forced |
| `httpx` ≥ 0.27 | qdrant-client (REST fallback) | BSD-3-Clause | — | Already present |
| `numpy` ≥ 1.26 | qdrant-client (vector helpers) | BSD-3-Clause | — | Already present |

**Total bundle impact**: ~+4 MB installed size in the backend container (net, after removing old grpcio/protobuf). Negligible for a server-side dep.

### Runtime Impact

| Dep | Cold Start Impact | Memory Overhead | Notes |
|-----|-------------------|-----------------|-------|
| `grpcio` 1.66 | ~+30 ms at import | +3–5 MB RSS | Native extension; already loaded today, just newer. |
| `protobuf` 5.x | negligible | negligible | Pure re-gen of descriptors at import. |
| `qdrant-client` 1.12 | ~+15 ms at import vs 1.7 | +2 MB RSS | Additional Pydantic v2 models for sparse/named vectors. |

No measurable request-path latency change in local benchmarks (p50 vector search within ±2%).

**⚠️ protobuf 5.x compatibility note**: We verified that `opentelemetry-proto` (currently `~=1.25`) and `google-cloud-storage` both declare `protobuf>=4.21,<6`, so the 5.x bump is within their supported range. A compatibility test is added to CI (`tests/deps/test_protobuf_compat.py`) as a guardrail.

## 9. Rollback Plan

```bash
# Revert client pin
poetry add "qdrant-client@~1.7.3"
poetry lock --no-update
# Revert server image in docker-compose.yml / Helm values
# qdrant/qdrant:v1.7.4

# Drop & recreate collections in dev/staging (sparse configs are not backward readable)
python -m app.scripts.qdrant_recreate_collections --profile=legacy_1_7
```

Rollback is only viable **before** sparse vectors are written to production (post-LAI-143 merge, rollback requires full re-embed from source documents in object storage — budget ~30 min for expected Phase 1 corpus size).

**Tracking issue for rollback verification**: [LAI-146](https://tracker.internal/LAI-146)

## 10. Decision

- [x] **Approved with conditions**
  - Conditions:
    1. LAI-145 (server image bump to v1.12.1) must merge **before or with** this client upgrade — mixed 1.7-client / 1.12-server combination is supported but not the reverse.
    2. CI guardrail `tests/deps/test_protobuf_compat.py` must be green.
    3. Digest-pinning of the Qdrant Docker image (LAI-152) tracked as follow-up; not blocking this PR.
    4. `DeprecationWarning` from `qdrant_client` promoted to error in the vector repository test module (prevents silent regressions on the `search()` → `query_points()` migration).
