# Migration Guide

> **Title**: Qdrant 1.7 → 1.12.x Server Upgrade (docker-compose)
> **Phase**: 1 | **PR(s)**: 1.4.2
> **Author**: SRE / Platform Engineering
> **Date**: 2026-04-09
> **Status**: Tested in Staging

---

## 1. Overview

PR 1.4.2 bumps the Qdrant server image used by `docker-compose.yml` from the previously-pinned `qdrant/qdrant:latest@sha256:9472857…` (Qdrant 1.7.x line) to a digest-pinned `qdrant/qdrant:v1.12.6`, introduces a `QDRANT_IMAGE` environment override, and preserves the existing healthcheck contract. This is the **server-side half** of the upgrade tracked in [1.4.1 Dep Review](./1.4.1_dep-review_qdrant-client.md); the client bump and `vector_store.py` refactor land in PR 1.4.3.

This guide is the **component-level procedure** referenced from §3 Step 2 of the [Phase-Level Mig Guide 1.0](./1.0_mig-guide_infrastructure-upgrades.md). When run as part of the Phase 1 maintenance window, follow the phase-level ordering and rollback authority. When run standalone (dev/staging refresh, hotfix), this document is self-contained.

| Aspect | Before | After |
|--------|--------|-------|
| Compose image | `qdrant/qdrant:latest@sha256:9472857…` (1.7.x) | `${QDRANT_IMAGE:-qdrant/qdrant:v1.12.6}` (digest pin pre-merge) |
| Override mechanism | None (image hardcoded) | `QDRANT_IMAGE` env var (in `.env.example`) |
| Healthcheck | `test -d /qdrant/storage` | `test -d /qdrant/storage` (unchanged — Qdrant minimal image lacks `curl`) |
| External readiness probe | `GET :6333/readyz` (manual) | `GET :6333/readyz` (manual + smoke script) |
| On-disk format | 1.7 segment format | 1.12 segment format (forward-compatible read; **rewritten on first compaction**) |
| `vector_store.py` | Uses `qdrant-client==1.7.x` API | **Unchanged in this PR** — refactored in PR 1.4.3 |

**Out of scope for 1.4.2**: client library bump, named/sparse vector schema, `vector_store.py` refactor, collection re-creation. Those are PR 1.4.3.

## 3. Migration Steps

### Parallelism Guidance

All steps in this guide are **strictly sequential**. When run inside the Phase 1 window, this entire guide executes as Step 2 of the phase-level Mig Guide and must complete (verification green) before Step 3 (PostgreSQL) begins. Do not run pgvector, Valkey, or Neo4j upgrades concurrently with this procedure.

### Step 1 — Snapshot every collection (pre-upgrade rollback artifact)

| Duration Estimate | Executor |
|-------------------|----------|
| ~3 min (staging: 1m48s on 420K points) | SRE |

```bash
# 1.1 Record pre-upgrade inventory (counts go into the verification checklist)
mkdir -p ./backups
PRECOUNTS=./backups/qdrant-precounts-$(date +%F).txt
: > "$PRECOUNTS"
for col in $(curl -sf http://localhost:6333/collections | jq -r '.result.collections[].name'); do
  cnt=$(curl -sf -X POST "http://localhost:6333/collections/${col}/points/count" \
    -H 'Content-Type: application/json' -d '{"exact":true}' | jq -r .result.count)
  printf '%s\t%s\n' "$col" "$cnt" | tee -a "$PRECOUNTS"
done

# 1.2 Trigger a snapshot for every collection
for col in $(curl -sf http://localhost:6333/collections | jq -r '.result.collections[].name'); do
  curl -sf -X POST "http://localhost:6333/collections/${col}/snapshots" | jq .
done

# 1.3 Copy the entire snapshots tree out of the volume
SNAPDIR=./backups/qdrant-preupgrade-$(date +%F)
docker compose cp qdrant:/qdrant/storage/snapshots "$SNAPDIR"
ls -lhR "$SNAPDIR" | head
```

**Expected output**: One `.snapshot` file per collection under `$SNAPDIR/<collection>/`. `$PRECOUNTS` lists each collection and its exact point count.

**Verification**:
```bash
test -s "$PRECOUNTS" && echo "precounts: OK"
find "$SNAPDIR" -name '*.snapshot' -type f -size +0c | wc -l   # must be ≥ number of collections
```

If `wc -l` is `0`, **stop**. The snapshot API silently succeeded but the volume cp picked up nothing — almost always a wrong volume mount path. Re-run with `docker compose exec qdrant ls /qdrant/storage/snapshots` first.

### Step 2 — Stop Qdrant cleanly

| Duration Estimate | Executor |
|-------------------|----------|
| ~30s | SRE |

```bash
docker compose stop qdrant
docker compose ps qdrant
```

**Expected output**: `STATE: exited (0)`. Any non-zero exit code → capture `docker compose logs --tail=200 qdrant` and pause; do not proceed until the stop is clean (a dirty stop risks segment corruption that the 1.12 loader will reject).

**Verification**:
```bash
docker compose logs --tail=20 qdrant | grep -i 'shutdown\|stopped\|gracefully'
```

### Step 3 — Swap the image and start Qdrant 1.12.x (in-place attempt)

| Duration Estimate | Executor |
|-------------------|----------|
| ~90s (staging: 52s) | SRE |

> **POINT OF NO RETURN**: Starting Qdrant 1.12 against the existing `qdrant_data` volume will, on first successful boot, begin rewriting segment files into the 1.12 on-disk format during background compaction. Once segments have been rewritten, Qdrant 1.7 will refuse to read the volume. From this moment, the rollback path is **R-1 (snapshot restore into a fresh 1.7 volume)**, not a simple image revert. Decision authority to cross this gate: **on-call SRE** (Phase 1 has no production data on Qdrant); inside the Phase 1 window, defer to the joint authority defined in the phase-level Mig Guide §7.

```bash
# 3.1 Confirm the compose file already pins the new image (this PR)
grep -nE 'qdrant/qdrant:v1\.12' docker-compose.yml

# 3.2 (Optional, recommended) override the digest at runtime if not yet committed
# export QDRANT_IMAGE='qdrant/qdrant:v1.12.6@sha256:<digest>'

# 3.3 Pull and start
docker compose pull qdrant
docker compose up -d qdrant

# 3.4 Wait for /readyz from the host (qdrant image lacks curl, so probe externally)
for i in $(seq 1 60); do
  if curl -sf http://localhost:6333/readyz >/dev/null; then echo "ready"; break; fi
  sleep 2
done
```

**Expected output**: `ready` printed within 60s; container status `Up (healthy)` after the docker healthcheck interval (≤30s + start_period).

**Verification**:
```bash
docker compose ps qdrant                                    # STATE healthy
curl -s http://localhost:6333/ | jq -r .version             # 1.12.x
docker compose logs --tail=100 qdrant | grep -iE 'error|panic|migrat' || echo "no errors"
```

If `docker compose ps` shows `unhealthy` or the loop above times out → go to **§7 Fallback (Snapshot Restore Path)**.

### Step 4 — Verify collections, point counts, and healthcheck stability

| Duration Estimate | Executor |
|-------------------|----------|
| ~2 min | SRE |

```bash
# 4.1 Collections list still matches pre-upgrade
curl -s http://localhost:6333/collections | jq -r '.result.collections[].name' | sort > /tmp/qdrant-postcols.txt
awk -F'\t' '{print $1}' "$PRECOUNTS" | sort > /tmp/qdrant-precols.txt
diff /tmp/qdrant-precols.txt /tmp/qdrant-postcols.txt && echo "collections: OK"

# 4.2 Point counts unchanged
while IFS=$'\t' read -r col before; do
  after=$(curl -sf -X POST "http://localhost:6333/collections/${col}/points/count" \
    -H 'Content-Type: application/json' -d '{"exact":true}' | jq -r .result.count)
  if [ "$before" = "$after" ]; then
    echo "OK   $col $before"
  else
    echo "DIFF $col before=$before after=$after"; FAIL=1
  fi
done < "$PRECOUNTS"
[ -z "$FAIL" ] && echo "counts: OK" || { echo "counts: MISMATCH — abort"; exit 1; }

# 4.3 Docker healthcheck has not flapped since boot
docker inspect --format='{{json .State.Health}}' legal-ai-qdrant | jq '{Status, FailingStreak, Log: (.Log[-3:] | map({End, ExitCode}))}'
```

**Expected output**: `collections: OK`, `counts: OK`, `Status: "healthy"`, `FailingStreak: 0`. **This satisfies acceptance criterion IT-QD-01** (`GET /` returns version ≥ 1.12) and the healthcheck contract.

**Verification**: Re-run the inspect command after waiting 2 × healthcheck interval (60s) and confirm `FailingStreak` is still 0.

## 6. Data Migration

No row-level data movement. The 1.12 server reads existing 1.7-format segments on boot, then rewrites them into 1.12 format during background compaction. The `qdrant_data` volume is the same volume before and after — no copy, no re-index.

| Data | Source | Destination | Method | Estimated Duration |
|------|--------|------------|--------|-------------------|
| All collections (in-place path) | `qdrant_data` (1.7 segments) | same volume (1.12 segments after compaction) | In-place container swap | ~90s downtime; compaction async |
| `embeddings_v1` (fallback path, if in-place fails) | `./backups/qdrant-preupgrade-*/embeddings_v1/*.snapshot` | freshly-initialized `qdrant_data` volume | `PUT /collections/{name}/snapshots/recover` | ~9 min for 420K points (staging) |

## 7. Rollback Procedure

### Decision Authority

| Role | Authority |
|------|-----------|
| **Can call rollback (before Step 3 boot succeeds)** | On-call SRE, unilaterally |
| **Can call rollback (after Step 3 boot succeeds — segments may have been rewritten)** | On-call SRE for staging/dev; **SRE Lead + Tech Lead jointly** for production (in writing in `#incident-phase1`) |
| **Must be notified** | Backend Platform on-call, Engineering Manager |
| **Must approve re-attempt** | SRE Lead + Tech Lead |

### Trigger Conditions

Rollback if any of:
- Step 2 produces a non-clean exit and a retry stop also fails.
- Step 3 boot loop exceeds 60s without `/readyz` returning 200, **and** logs show `failed to load segment` or `panic`.
- Step 4 point counts mismatch by even one point on any collection.
- Healthcheck `FailingStreak ≥ 3` within 5 minutes of boot.
- Backend `/health/ready` returns 503 for > 2 consecutive checks attributable to qdrant.
- Cumulative elapsed time exceeds the 12-minute Phase 1 §3 Step 2 budget.

### R-1: Snapshot Restore into Fresh Volume (the only viable post-PONR path)

```bash
# 1. Stop the failing 1.12 container
docker compose stop qdrant

# 2. Revert the image pin (in compose, OR override at runtime)
export QDRANT_IMAGE='qdrant/qdrant:latest@sha256:94728574965d17c6485dd361aa3c0818b325b9016dac5ea6afec7b4b2700865f'
# (Or: git checkout HEAD~1 -- docker-compose.yml)

# 3. Wipe the (possibly half-rewritten) volume
docker compose rm -f qdrant
docker volume rm legalai_qdrant_data

# 4. Boot Qdrant 1.7 clean
docker compose up -d qdrant
until curl -sf http://localhost:6333/readyz >/dev/null; do sleep 2; done

# 5. Recover every collection from the Step 1 snapshots
SNAPDIR=./backups/qdrant-preupgrade-$(date +%F)
# Snapshots must be reachable from inside the qdrant container — copy them in:
docker compose cp "$SNAPDIR" qdrant:/qdrant/snapshots-restore
for snap in "$SNAPDIR"/*/*.snapshot; do
  col=$(basename "$(dirname "$snap")")
  inside="/qdrant/snapshots-restore/$(basename "$(dirname "$snap")")/$(basename "$snap")"
  curl -sf -X PUT "http://localhost:6333/collections/${col}/snapshots/recover" \
    -H 'Content-Type: application/json' \
    -d "{\"location\":\"file://${inside}\"}" | jq .
done
```

### R-2: Forward-Recovery to 1.12 via Snapshot (the documented "snapshot→restore path if in-place fails")

If in-place at Step 3 fails but the goal is still to land on 1.12 (preferred when the underlying issue is segment-format incompatibility, not a 1.12 bug):

```bash
# 1. Stop and wipe (same as R-1 steps 1, 3)
docker compose stop qdrant
docker compose rm -f qdrant
docker volume rm legalai_qdrant_data

# 2. Boot 1.12 against an EMPTY volume
docker compose up -d qdrant
until curl -sf http://localhost:6333/readyz >/dev/null; do sleep 2; done

# 3. Recover snapshots — Qdrant 1.12 reads 1.7-era snapshots correctly
SNAPDIR=./backups/qdrant-preupgrade-$(date +%F)
docker compose cp "$SNAPDIR" qdrant:/qdrant/snapshots-restore
for snap in "$SNAPDIR"/*/*.snapshot; do
  col=$(basename "$(dirname "$snap")")
  inside="/qdrant/snapshots-restore/$(basename "$(dirname "$snap")")/$(basename "$snap")"
  curl -sf -X PUT "http://localhost:6333/collections/${col}/snapshots/recover" \
    -H 'Content-Type: application/json' \
    -d "{\"location\":\"file://${inside}\"}" | jq .
done

# 4. Re-run Step 4 verification (counts must match precounts file)
```

**This R-2 path is what satisfies acceptance criterion IT-QD-03** ("Snapshot of 1.7 collection restores into 1.12"). It is exercised in CI (see §8) and was used for `embeddings_v1` in dry run 1.

### Rollback Verification

```bash
# After R-1 (rolled back to 1.7)
curl -s http://localhost:6333/ | jq -r .version                 # 1.7.x
docker compose ps qdrant                                        # healthy
while IFS=$'\t' read -r col before; do
  after=$(curl -sf -X POST "http://localhost:6333/collections/${col}/points/count" \
    -H 'Content-Type: application/json' -d '{"exact":true}' | jq -r .result.count)
  [ "$before" = "$after" ] && echo "OK $col" || echo "FAIL $col"
done < ./backups/qdrant-precounts-$(date +%F).txt
curl -sf http://localhost:8000/api/v1/health/ready | jq .
```

## 9. Downtime Estimate

| Component | Downtime | Mitigation |
|-----------|----------|------------|
| Qdrant container (in-place path) | ~90s (Step 2 stop → Step 3 ready) | Phase 1: no production traffic on vector path; backend retries on reconnect |
| Qdrant container (fallback R-2 path) | ~10 min | Same — Phase 1 has no SLA on vector reads yet |
| Backend `/health/ready` | ~10s flap during qdrant restart | Liveness != readiness; load balancer drains naturally |
| **User-visible** | **0 min in Phase 1** | No user-facing feature depends on Qdrant in Phase 1; vector search is dark-launched in Phase 3 |
