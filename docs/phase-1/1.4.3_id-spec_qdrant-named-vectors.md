# Interface Design Specification

> **Title**: Qdrant Named-Vectors Collection Schema & Repository Interface
> **Phase**: Phase 1 — Infrastructure Upgrades | **PR(s)**: 1.4.3
> **Author**: Tech Lead (Backend)
> **Date**: 2026-04-09
> **Status**: Draft
> **Version**: 1.0

> This ID Spec defines an **internal backend interface**, not a public HTTP API. Qdrant is consumed exclusively through `backend/app/services/vector_store.py` by other backend services; there is no direct frontend or tenant-facing surface in Phase 1. HTTP-specific sections (CORS, rate limit headers, SSE, WebSocket, webhooks, pagination envelope) are therefore marked `N/A`. The contract defined here is the **Python repository API** plus the **Qdrant wire schema** (collection config, point payload, sparse vector format) that the repository produces and consumes.

---

## 1. Overview

PR 1.4.3 refactors the Qdrant integration from a legacy single-unnamed-vector layout to Qdrant 1.12+ **named vectors** with an additional **sparse vector** channel, unblocking Phase 3 hybrid (dense + lexical) retrieval. This spec is the contract between:

| Role | Component |
|------|-----------|
| **Producer** | `backend/app/services/vector_store.py` (`QdrantVectorStore` class) — owns collection lifecycle, upsert, search, delete. |
| **Consumer (internal)** | `backend/app/services/embedding_service.py`, `backend/app/services/retrieval_service.py`, future Phase 3 `HybridRetriever`, ingestion Celery tasks. |
| **Wire peer** | `qdrant/qdrant:v1.12.x` via `qdrant-client ≥ 1.12.0` (gRPC preferred, HTTP fallback). |
| **Reviewers** | Tech Lead, Phase 3 Retrieval owner (consumer-reviewer), SRE (ops impact). |

**Scope of this document**
1. Named-vector collection configuration (dense + sparse) and collection naming convention.
2. Point payload schema, including mandatory `tenant_id` for multi-tenant isolation.
3. Python repository API contracts: `ensure_collection`, `upsert`, `search`, `delete`, `count`.
4. Backward-compatibility strategy for existing single-unnamed-vector collections created under Qdrant 1.7.
5. Sparse vector wire format and construction rules.
6. Versioning and deprecation plan for the legacy collection layout.

**Out of scope**
- Actual hybrid fusion (RRF / weighted) logic — Phase 3.
- Sparse encoder choice (SPLADE vs. BM25/IDF tokenizer) — this spec pins the *format*, not the *model*. Phase 1 ships with a deterministic IDF-style sparse encoder as a placeholder.
- Qdrant server upgrade mechanics — covered by 1.4.2 Mig Guide.

## 2. Global API Policies

This is an internal Python interface over a gRPC/HTTP client. Traditional web API policies (CORS, public rate limits) do not apply. The equivalent operational policies are:

| Policy | Value | Notes |
|--------|-------|-------|
| Transport | gRPC (preferred), HTTP fallback | `qdrant-client` auto-selects; gRPC for bulk upsert, HTTP for admin. |
| Endpoint | `QDRANT_URL` env var | Existing; unchanged by this PR. |
| Auth | `QDRANT_API_KEY` env var (optional, required in staging/prod) | Passed as gRPC metadata / `api-key` header. |
| Max upsert batch | **256 points** per `upsert` call | Client-side chunking; larger inputs are split transparently. |
| Max vector payload size | 1 MB per point (payload only, excluding vectors) | Enforced client-side to avoid server rejections. |
| Default RPC timeout | 10 s (search), 60 s (upsert), 120 s (collection admin) | Tunable via `QdrantVectorStore(timeout=...)`. |
| Retries | 3 attempts, exponential backoff (100 ms → 400 ms → 1.6 s), idempotent ops only | Upsert is idempotent by point `id`; search/count are safe; delete is idempotent. |
| Tenant isolation | **Mandatory** `tenant_id` payload filter on every `search` and `delete` | Enforced in repository, not caller — see §3.3. |
| Concurrency | Async (`AsyncQdrantClient`) | One shared client per FastAPI app instance. |

### Rate Limiting Strategy

N/A — Qdrant is an internal dependency, not a tenant-facing API. Backpressure is handled by (a) Celery ingestion worker concurrency limits and (b) the per-tenant retrieval rate limits that live on the outer `/api/v1/search` endpoint (defined in the retrieval service ID Spec, not here).

### CORS / Headers

N/A — no browser clients.

## 3. API Endpoints

The "endpoints" below are **Python repository methods** on `QdrantVectorStore`. Each subsection follows the same shape as an HTTP endpoint spec: contract, inputs, outputs, errors, example.

### 3.1 Collection Naming Convention

Collections are named:

```
{env}__legalai__{logical_name}__v{schema_version}
```

| Segment | Example | Rules |
|---------|---------|-------|
| `env` | `dev`, `staging`, `prod` | From `APP_ENV`. |
| `legalai` | literal | Namespace prefix to avoid collision with any co-tenant on a shared Qdrant instance. |
| `logical_name` | `document_chunks`, `contract_clauses` | `[a-z0-9_]+`, ≤ 40 chars. |
| `schema_version` | `v2` | Bumped on **breaking** schema changes (new vector name, dim change, distance change). Phase 1 ships `v2` (dense+sparse named); the legacy single-vector layout is retroactively `v1`. |

**Tenancy model**: **one collection per logical purpose, tenant-scoped via payload filter** — *not* one collection per tenant. This is a deliberate choice (see §10 / rejected approaches in the upstream Tech Spec): tenant-per-collection does not scale past ~1k tenants on a single Qdrant instance. All multi-tenant access goes through an enforced `tenant_id` filter (§3.3, §7).

Example: `prod__legalai__document_chunks__v2`.

### 3.2 `ensure_collection(name, recreate=False) -> None`

**Description**: Idempotent collection creation. If the collection does not exist, creates it with the named-vectors config in §7. If it exists with a compatible config, returns silently. If it exists with an **incompatible** config (different dim, distance, or missing named vectors), raises `CollectionSchemaMismatch` unless `recreate=True`.

**Authentication**: Inherits client credentials.

**Authorization**: Caller must be running under the backend app role; collection creation is a bootstrap / migration operation and is not exposed to request-path code.

**Idempotency**: Yes — safe to call on every app startup.

#### Request (Python signature)

```python
async def ensure_collection(
    self,
    name: str,
    *,
    recreate: bool = False,
) -> None:
    ...
```

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | `str` | Yes | Full collection name per §3.1 convention. |
| `recreate` | `bool` | No (default `False`) | If `True` and an incompatible collection exists, drop and recreate. **Destructive** — only used by explicit migration scripts, never on the request path. |

#### Wire request (sent to Qdrant)

```json
{
  "vectors": {
    "dense": {
      "size": 1536,
      "distance": "Cosine",
      "on_disk": false
    }
  },
  "sparse_vectors": {
    "sparse": {
      "index": { "on_disk": false, "full_scan_threshold": 5000 },
      "modifier": "idf"
    }
  },
  "optimizers_config": { "default_segment_number": 2 },
  "hnsw_config":       { "m": 16, "ef_construct": 128 },
  "on_disk_payload":   true
}
```

#### Response

`None` on success. State is observable via `client.get_collection(name)`.

#### Errors

| Exception | Raised when |
|-----------|-------------|
| `CollectionSchemaMismatch` | Existing collection has different dense dim, distance, missing `dense` or `sparse` named vector, or has an unnamed default vector (legacy v1). |
| `QdrantUnavailable` | Client cannot connect / auth fails after retries. |
| `QdrantServerError` | Qdrant returns 5xx or gRPC `INTERNAL`. |

### 3.3 `upsert(name, points) -> UpsertResult`

**Description**: Insert or update points. Each point carries a dense vector and optionally a sparse vector under the named channels defined in §7. `tenant_id` is **mandatory** in the payload; points missing it are rejected client-side before reaching Qdrant.

**Idempotency**: Yes — Qdrant upsert is idempotent by point `id`. Retries on transient failures are safe.

#### Request (Python signature)

```python
async def upsert(
    self,
    name: str,
    points: list[VectorPoint],
) -> UpsertResult:
    ...
```

**`VectorPoint` Pydantic schema:**

```python
from pydantic import BaseModel, ConfigDict, Field
from uuid import UUID

class SparseVector(BaseModel):
    indices: list[int] = Field(..., description="Sorted, unique term IDs (non-negative int32).")
    values:  list[float] = Field(..., description="IDF-weighted values, same length as indices, all > 0.")

class VectorPayload(BaseModel):
    tenant_id:     UUID          = Field(..., description="Mandatory tenant isolation key.")
    document_id:   UUID
    chunk_index:   int           = Field(..., ge=0)
    source_uri:    str
    content_hash:  str           = Field(..., description="sha256 of chunk text, for dedupe.")
    model_name:    str           = Field(..., description="e.g. 'text-embedding-3-small'.")
    model_version: str
    token_count:   int           = Field(..., ge=0)
    language:      str           = Field(default="en", max_length=8)
    created_at:    str           = Field(..., description="RFC3339 UTC.")
    # Free-form domain metadata; must not collide with reserved keys above.
    metadata:      dict[str, str | int | float | bool] = Field(default_factory=dict)

class VectorPoint(BaseModel):
    id:      UUID                = Field(..., description="Stable point id; reused across re-embeds of the same chunk.")
    dense:   list[float]         = Field(..., min_length=1536, max_length=1536)
    sparse:  SparseVector | None = None
    payload: VectorPayload

    model_config = ConfigDict(
        json_schema_extra={
            "example": {
                "id": "f3a1c0b2-2c4a-4b8f-9a27-2d3e1e7a9b11",
                "dense": [0.0123, -0.0456, "... (1536 floats)"],
                "sparse": {
                    "indices": [17, 204, 998, 15342],
                    "values":  [2.31, 1.84, 3.02, 0.97]
                },
                "payload": {
                    "tenant_id":     "b8e6b3a4-1111-4222-8333-444444444444",
                    "document_id":   "c9d7c4b5-5555-4666-8777-888888888888",
                    "chunk_index":   12,
                    "source_uri":    "s3://legalai-prod/contracts/abc.pdf",
                    "content_hash":  "9f86d081884c7d659a2feaa0c55ad015...",
                    "model_name":    "text-embedding-3-small",
                    "model_version": "2024-02-01",
                    "token_count":   384,
                    "language":      "en",
                    "created_at":    "2026-04-09T12:00:00Z",
                    "metadata":      {"contract_type": "MSA"}
                }
            }
        }
    )
```

**Wire shape sent to Qdrant (per point):**

```json
{
  "id": "f3a1c0b2-2c4a-4b8f-9a27-2d3e1e7a9b11",
  "vector": {
    "dense":  [0.0123, -0.0456, "..."],
    "sparse": { "indices": [17, 204, 998, 15342], "values": [2.31, 1.84, 3.02, 0.97] }
  },
  "payload": { "tenant_id": "...", "document_id": "...", "chunk_index": 12, "...": "..." }
}
```

Note: dense and sparse share the same top-level `vector` object because they are sibling **named** vectors on the same collection. Sparse is *optional per point* — points may omit it during dense-only ingestion, but the collection must have the `sparse` channel declared.

#### Response

```python
class UpsertResult(BaseModel):
    upserted_count: int
    operation_id:   int               # Qdrant operation id
    status:         Literal["acknowledged", "completed"]
```

#### Errors

| Exception | When |
|-----------|------|
| `MissingTenantId` | Any point has `payload.tenant_id` unset. **Fail fast, no partial write.** |
| `VectorDimensionMismatch` | Any `dense` vector length ≠ 1536. |
| `SparseVectorInvalid` | `indices`/`values` length mismatch, negative indices, non-positive values, or unsorted indices. |
| `PayloadTooLarge` | Payload > 1 MB after JSON encoding. |
| `CollectionNotFound` | Collection does not exist (caller must `ensure_collection` first). |
| `QdrantUnavailable` / `QdrantServerError` | Transport / server faults (retried before surfacing). |

### 3.4 `search(name, query, *, tenant_id, top_k, filters=None, vector_name="dense") -> list[SearchHit]`

**Description**: Single-vector search against one named channel. Hybrid search (fusing `dense` + `sparse`) is **not** provided in Phase 1 — callers use this method twice and fuse in application code, or wait for Phase 3's `HybridRetriever`. This keeps the Phase 1 interface minimal and the fusion decision deferred.

**Tenant isolation (mandatory)**: `tenant_id` is a required keyword argument. The repository **always** injects a `must: [{ key: "tenant_id", match: { value: tenant_id }}]` filter, merged (AND) with any caller-supplied filter. Callers cannot bypass this.

#### Request

```python
async def search(
    self,
    name: str,
    query: list[float] | SparseVector,
    *,
    tenant_id: UUID,
    top_k: int = 10,
    filters: dict | None = None,
    vector_name: Literal["dense", "sparse"] = "dense",
    score_threshold: float | None = None,
) -> list[SearchHit]:
    ...
```

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `str` | — | Collection name. |
| `query` | `list[float]` or `SparseVector` | — | Must match `vector_name`: dense=list of 1536 floats, sparse=`SparseVector`. |
| `tenant_id` | `UUID` | — | **Required.** Enforced filter key. |
| `top_k` | `int` | `10` | 1 ≤ top_k ≤ 200. |
| `filters` | `dict \| None` | `None` | Additional Qdrant filter, AND-merged with tenant filter. |
| `vector_name` | `"dense" \| "sparse"` | `"dense"` | Selects the named channel. |
| `score_threshold` | `float \| None` | `None` | Drop hits below threshold. |

#### Response

```python
class SearchHit(BaseModel):
    id:      UUID
    score:   float
    payload: VectorPayload
    # Vectors not returned by default (bandwidth); use a dedicated retrieve() call if needed.
```

**Example response:**

```json
[
  {
    "id": "f3a1c0b2-2c4a-4b8f-9a27-2d3e1e7a9b11",
    "score": 0.8734,

      "tenant_id":   "b8e6b3a4-1111-4222-8333-444444444444",
      "document_id": "c9d7c4b5-5555-4666-8777-888888888888",
      "chunk_index": 12,
      "source_uri":  "s3://legalai-prod/contracts/abc.pdf",
      "content_hash":"9f86d081...",
      "model_name":  "text-embedding-3-small",
      "model_version":"2024-02-01",
      "token_count": 384,
      "language":    "en",
      "created_at":  "2026-04-09T12:00:00Z",
      "metadata":    {"contract_type": "MSA"}
    }
  }
]
```

#### Errors

| Exception | When |
|-----------|------|
| `VectorDimensionMismatch` | `vector_name="dense"` and `len(query) != 1536`. |
| `SparseVectorInvalid` | `vector_name="sparse"` and query fails §6 validation. |
| `InvalidFilter` | Caller filter uses reserved key `tenant_id` (would conflict with enforced filter) or is structurally invalid. |
| `CollectionNotFound` | Collection missing. |
| `QdrantUnavailable` / `QdrantServerError` | Transport / server faults. |

### 3.5 `delete(name, *, tenant_id, point_ids=None, filter=None) -> DeleteResult`

**Description**: Deletes points by id or by filter. **Exactly one** of `point_ids` or `filter` must be provided. Tenant filter is always AND-merged, so callers cannot delete across tenants even with a wide filter.

**Idempotency**: Yes — deleting an unknown id is a no-op at Qdrant.

#### Request

```python
async def delete(
    self,
    name: str,
    *,
    tenant_id: UUID,
    point_ids: list[UUID] | None = None,
    filter: dict | None = None,
) -> DeleteResult:
    ...
```

#### Response

```python
class DeleteResult(BaseModel):
    operation_id: int
    status:       Literal["acknowledged", "completed"]
```

#### Errors

| Exception | When |
|-----------|------|
| `InvalidDeleteRequest` | Both or neither of `point_ids` / `filter` supplied; or caller filter references `tenant_id`. |
| `CollectionNotFound` | Collection missing. |
| `QdrantUnavailable` / `QdrantServerError` | Transport / server faults. |

### 3.6 `count(name, *, tenant_id, filter=None, exact=False) -> int`

**Description**: Returns the number of points matching the tenant filter (plus optional extra filter). `exact=False` (default) uses Qdrant's fast approximate count; `exact=True` forces a full scan (use sparingly).

Errors: `CollectionNotFound`, `QdrantUnavailable`, `QdrantServerError`.

## 6. Sparse Vector Format

Sparse vectors represent lexical / term-weighted signals. In Phase 1 we ship a deterministic IDF-weighted format; the *encoder* can be swapped later (SPLADE, BM25) without a schema change as long as the wire format holds.

### Wire format

```python
class SparseVector(BaseModel):
    indices: list[int]     # int32, non-negative, strictly sorted ascending, unique
    values:  list[float]   # float32, > 0.0, same length as indices
```

### Validation rules (enforced in `QdrantVectorStore` before send)

1. `len(indices) == len(values)`.
2. `1 ≤ len(indices) ≤ 10_000`.
3. All `indices[i] >= 0` and `indices[i] < 2**31`.
4. `indices` strictly ascending (no duplicates) — client sorts on caller's behalf if needed, and rejects duplicate term IDs.
5. All `values[i] > 0.0` and finite (no NaN/Inf).
6. Term ID space: the vocabulary mapping is owned by the sparse encoder module, versioned under `payload.model_name = "idf-v1"`. Points from different encoder versions must live in the same collection only if their term ID spaces are compatible; otherwise bump the collection `schema_version`.

### Collection-level configuration

Declared under `sparse_vectors.sparse` with `modifier: "idf"`, instructing Qdrant to apply IDF weighting on top of stored per-term values. The effective score at query time is `sum_over_i (query.value_i * doc.value_i * idf_i)`; see Qdrant 1.10+ sparse docs.

### Dense vs sparse at a glance

| Aspect | `dense` | `sparse` |
|--------|---------|----------|
| Dimensionality | Fixed 1536 | Variable per point |
| Distance | Cosine | Dot product (+ IDF modifier) |
| Encoder | `text-embedding-3-small` (Phase 1 default) | IDF tokenizer (Phase 1 placeholder) |
| Required per point? | **Yes** | **No** (optional at upsert time) |
| Search method | `search(..., vector_name="dense")` | `search(..., vector_name="sparse")` |

## 7. Data Models / Schemas

### 7.1 Collection configuration (Phase 1, `v2`)

```python
from qdrant_client.http.models import (
    VectorParams, Distance, SparseVectorParams, SparseIndexParams, Modifier,
    HnswConfigDiff, OptimizersConfigDiff,
)

COLLECTION_CONFIG_V2 = {
    "vectors_config": {
        "dense": VectorParams(size=1536, distance=Distance.COSINE, on_disk=False),
    },
    "sparse_vectors_config": {
        "sparse": SparseVectorParams(
            index=SparseIndexParams(on_disk=False, full_scan_threshold=5000),
            modifier=Modifier.IDF,
        ),
    },
    "hnsw_config":       HnswConfigDiff(m=16, ef_construct=128),
    "optimizers_config": OptimizersConfigDiff(default_segment_number=2),
    "on_disk_payload":   True,
}
```

### 7.2 Payload indexes (created by `ensure_collection`)

Every collection gets the following payload indexes to make filtered search fast and, critically, to make the mandatory `tenant_id` filter O(log n):

| Field | Index type | Rationale |
|-------|-----------|-----------|
| `tenant_id` | `keyword` | **Mandatory on every query** — must be indexed. |
| `document_id` | `keyword` | Per-document deletes / updates. |
| `content_hash` | `keyword` | Dedupe lookups during ingestion. |
| `model_name` | `keyword` | Filter out points from stale embedding models after re-embed. |
| `language` | `keyword` | Language routing. |
| `created_at` | `datetime` | Time-bounded queries, TTL-like cleanup. |

### 7.3 Reserved payload keys

These keys are owned by the repository; callers must not put their own values under them inside `payload.metadata`:

```
tenant_id, document_id, chunk_index, source_uri, content_hash,
model_name, model_version, token_count, language, created_at
```

Using any of these as a key in `VectorPayload.metadata` raises `ReservedPayloadKey` at validation time.

### 7.4 Error model

```python
class QdrantError(Exception):
    code: str
    detail: str

class MissingTenantId(QdrantError): ...          # code="MISSING_TENANT_ID"
class VectorDimensionMismatch(QdrantError): ...  # code="DIM_MISMATCH"
class SparseVectorInvalid(QdrantError): ...      # code="SPARSE_INVALID"
class PayloadTooLarge(QdrantError): ...          # code="PAYLOAD_TOO_LARGE"
class ReservedPayloadKey(QdrantError): ...       # code="RESERVED_KEY"
class InvalidFilter(QdrantError): ...            # code="INVALID_FILTER"
class InvalidDeleteRequest(QdrantError): ...     # code="INVALID_DELETE"
class CollectionNotFound(QdrantError): ...       # code="COLLECTION_NOT_FOUND"
class CollectionSchemaMismatch(QdrantError): ... # code="SCHEMA_MISMATCH"
class QdrantUnavailable(QdrantError): ...        # code="UNAVAILABLE" (transient)
class QdrantServerError(QdrantError): ...        # code="SERVER_ERROR" (transient/fatal)
```

Errors are classified into three tiers for the upstream Tech Spec's error policy:

| Tier | Errors |
|------|--------|
| **Transient** (retry) | `QdrantUnavailable`, `QdrantServerError` (5xx only) |
| **Recoverable** (surface to caller, do not retry) | `CollectionNotFound`, `MissingTenantId`, `VectorDimensionMismatch`, `SparseVectorInvalid`, `PayloadTooLarge`, `ReservedPayloadKey`, `InvalidFilter`, `InvalidDeleteRequest` |
| **Fatal** (page SRE) | `CollectionSchemaMismatch` on app startup |

## 10. Versioning & Backward Compatibility

### 10.1 Versioning Strategy

| Aspect | Approach |
|--------|----------|
| Versioning method | **Collection suffix** `__v{N}` per §3.1. The Python interface is versioned via normal semver on `vector_store.py`. |
| Breaking change definition (collection) | Adding/removing a named vector, changing dense dim, changing distance metric, changing sparse modifier, removing a reserved payload key. |
| Non-breaking changes | Adding a new optional metadata key, adding a new payload index, adding a new optional method, adding a new reserved key (only if unused by any caller). |
| Breaking change definition (Python API) | Removing or renaming a method, changing a required arg, tightening validation that previously accepted inputs. |

### 10.2 Version History

| Version | Layout | Changes | Backward compatible |
|---------|--------|---------|---------------------|
| `v1` (legacy) | Single unnamed vector, `size=1536`, Cosine, ad-hoc payload, no enforced `tenant_id` index | Pre-PR-1.4.3 state under Qdrant 1.7 | N/A |
| `v2` (this PR) | Named `dense` (1536, Cosine) + named `sparse` (IDF), enforced payload schema, enforced `tenant_id` index | Named vectors, sparse channel, mandatory tenant filter | **No** — separate collection name |

### 10.3 Backward-Compatibility Strategy for existing `v1` collections

The legacy single-unnamed-vector collections created under Qdrant 1.7 **cannot be upgraded in place** — Qdrant does not support adding a named vector to a collection that was created with an unnamed default vector. Strategy:

1. **Detection**. On app startup, `QdrantVectorStore.ensure_collection` inspects each configured logical collection. If it finds a collection of the old name (no `__v2` suffix) containing an unnamed default vector config, it logs `qdrant.legacy_v1_detected{collection=...}` at WARN and emits a metric `qdrant_legacy_v1_collections`.
2. **Dual-read shim (Phase 1 only, ≤ 60 days)**. `QdrantVectorStore.search` accepts a constructor flag `legacy_fallback=True`. When set, and the `v2` collection is empty while a `v1` collection exists, `search` transparently routes dense-only queries to the legacy collection. Sparse search against legacy collections raises `LegacyCollectionNoSparse` — callers must fall back to dense-only. The shim **never writes** to legacy collections; all new upserts go to `v2`.
3. **Backfill** (out of band, script, not on the request path): `scripts/qdrant_backfill_v2.py` scrolls every point from `{name}` (legacy), re-embeds sparse (dense is reused verbatim — same model, same dim), and upserts into `{name}__v2`. Tenant-scoped, resumable, idempotent by point id. Runbook will be in 1.4.2 Mig Guide.
4. **Cutover**. Once the backfill script reports 100% of tenants migrated for a given logical collection, the ops engineer flips `legacy_fallback=False` for that collection via config.
5. **Cleanup**. After a **30-day** observation window with `legacy_fallback=False` and zero `qdrant_legacy_v1_read_total` metric increments, the legacy collection is dropped. **Target cleanup date: 2026-08-01.** This is tracked as an explicit backlog item; the `legacy_fallback` flag is a feature flag with a mandatory cleanup ticket.
6. **Fatal failure mode**. If detection finds a legacy collection but `legacy_fallback=False` *and* the `v2` collection does not exist, startup fails with `CollectionSchemaMismatch` — deployment is blocked until the ops engineer explicitly acknowledges the migration state. This is the one fatal error class for this repository.

### 10.4 Deprecation Flow for the legacy layout

| Step | Action | Timeline |
|------|--------|----------|
| 1 | Ship `v2` collections + dual-read shim (this PR) | PR 1.4.3 merge |
| 2 | Run `qdrant_backfill_v2.py` in staging, validate hit-rate parity on Phase 3 eval set (tolerance: ±2% recall@10) | Within 7 days of merge |

## 11. Observability

Internal interface, so standard HTTP metrics don't apply. The repository emits:

**Metrics (Prometheus):**
- `qdrant_request_duration_seconds{op, collection, status}` — histogram per `ensure_collection | upsert | search | delete | count`.
- `qdrant_request_total{op, collection, status, error_code}` — counter; `status ∈ {ok, transient_error, recoverable_error, fatal_error}`.
- `qdrant_upsert_points_total{collection}` — counter of successfully upserted points.
- `qdrant_search_hits{collection, vector_name}` — histogram of returned hit counts.
- `qdrant_legacy_v1_collections` — gauge, set at startup.
- `qdrant_legacy_v1_read_total{collection}` — counter; **must reach zero** before step 6 above.
- `qdrant_backfill_progress{collection, tenant_id}` — gauge 0–1.

**Logs (structured, JSON):**
- Every method call logs at DEBUG with `op`, `collection`, `tenant_id`, `duration_ms`, `status`.
- Every non-transient error logs at ERROR with `error_code`, `detail`, and stack trace.
- Startup logs `qdrant.schema_check{collection, version, status}` exactly once per collection.

**Traces (OpenTelemetry):**
- Each method is an OTel span `qdrant.{op}` with attributes `collection`, `vector_name`, `top_k`, `tenant_id` (hashed).
- `ensure_collection` span is parented to the app startup span.
